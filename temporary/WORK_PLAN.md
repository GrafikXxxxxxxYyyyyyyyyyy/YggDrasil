# План реализации фреймворка Иггдрасиль (рабочий план)

Двухчастный план: **Часть 1** — полная реализация движка и поддержка диффузионных, языковых и агентных систем с полной проверкой и тестированием; **Часть 2** — экспериментальные уровни (Stage, World, Universe) и полировка сериализации.

**Связь с другими документами:** [IMPLEMENTATION_PLAN.md](IMPLEMENTATION_PLAN.md) — общая последовательность фаз; [TODO_PART1.md](TODO_PART1.md) — детальные TODO по Части 1.

---

## Часть 1. Движок и поддержка диффузии / язык / агенты

**Цель:** реализовать весь исполняемый движок фреймворка (от фундамента до воркфлоу), обеспечить поддержку сценариев **диффузии**, **языковых моделей** и **агентных систем**, провести полную проверку, тестирование и покрытие ключевых кейсов.

**Граница:** фазы 0–6 включительно. Результат — стабильное ядро, на котором можно собирать гиперграфы задач и воркфлоу, запускать run, сохранять/загружать конфиг и чекпоинты; архитектура готова к подключению конкретных блоков (диффузия, LLM, агенты) через реестр.

### 1.1 Что входит в Часть 1

| Область | Содержание |
|--------|-------------|
| **Структура** | Репозиторий: пакет `yggdrasill/`, тесты в `tests/`, pyproject.toml, зависимости, запуск тестов (фаза 0). |
| **Фундамент** | Port, AbstractBaseBlock, AbstractGraphNode, BlockRegistry; контракты портов и run; двойное наследование для узлов-задач (фаза 1). |
| **Движок** | Edge, хранилище структуры (узлы, рёбра, exposed), Validator, Planner, EdgeBuffers, Executor; run(structure, inputs, **options) → outputs; DAG и один итеративный цикл (K шагов) (фаза 2). |
| **Гиперграф задачи** | Hypergraph с add_node(block_type, config), from_config, to_config, get_input_spec, get_output_spec, run через движок; циклы в графе (фаза 3). |
| **Узлы-задачи** | Семь ролей (Backbone, Injector, Conjector, Inner Module, Outer Module, Helper, Converter), абстракции и заглушки, регистрация, role_from_block_type; опционально автосвязывание (фаза 4). |
| **Сериализация ядра** | Конфиг + чекпоинт для блока и для гиперграфа задачи; save/load, save_config, save_checkpoint, load_from_checkpoint; дедупликация по block_id (фаза 5). |
| **Воркфлоу** | Workflow как гиперграф гиперграфов; тот же движок; run(workflow, inputs); циклы между гиперграфами; сериализация воркфлоу; опционально infer_exposed_ports, auto_connect (фаза 6). |
| **Поддержка доменов** | Архитектура и контракты готовы к подключению блоков: диффузия (Backbone/Injector/Conjector и т.д.), языковые модели (Backbone, Converter, Helper), агентные цепочки (узлы с tool_calls, при необходимости agent_loop). Конкретные реализации блоков могут быть в отдельных пакетах; в Части 1 достаточно заглушек и 1–2 минимальных примера под каждый домен. |
| **Проверка и тесты** | Unit-тесты по компонентам; интеграционные тесты: цепочка узлов, цикл в гиперграфе, воркфлоу из 2–3 гиперграфов, roundtrip config/checkpoint; покрытие кейсов: DAG, цикл K шагов, опциональные порты, агрегация на порту, сериализация и воспроизводимость. |

### 1.2 Критерии завершения Части 1

- [ ] Фазы 0–6 реализованы по планам PHASE_0 … PHASE_6.
- [ ] Все тесты из планов фаз проходят (`pytest tests/`).
- [ ] Гиперграф задачи: сборка из конфига, run, цикл (K шагов), сохранение/загрузка конфига и чекпоинта.
- [ ] Воркфлоу: сборка из конфига, run, цикл между гиперграфами (при необходимости), save/load.
- [ ] Семь ролей узлов-задач задекларированы и доступны через реестр; заглушки позволяют собирать графы под типовые сценарии (цепочка, цикл Backbone ↔ Inner Module, цепочка с Converter).
- [ ] Документированы или протестированы мини-кейсы: «диффузионный сценарий» (например Backbone + Conjector), «языковой сценарий» (Backbone + Converter/Helper), «агентный сценарий» (при наличии agent_loop — вызов узла с tool_calls и инструментами).
- [ ] Сериализация: воспроизводимость run при одинаковых конфиге, чекпоинте и входах (тесты).

### 1.3 Порядок работ (Часть 1)

Строго по зависимостям: **0 → 1 → 2 → 3 → 4 → 5 → 6**. Внутри каждой фазы — по шагам из PHASE_* документов. Тесты писать параллельно реализации (TDD или сразу после компонента). Итоговая проверка: прогон всех тестов и сценариев из TODO_PART1.

## Часть 2. Экспериментальные уровни и полировка

**Цель:** после стабилизации движка (Часть 1) — реализовать или доработать экспериментальные уровни Stage, World, Universe и единый контракт сериализации; углубить логику State, директории мира, циклов автор→среда→творец.

**Граница:** фазы 7–10. Все они помечены как **экспериментальные** и требуют глубокой переработки и осмысления после завершения Части 1.

### 2.1 Что входит в Часть 2

| Область | Содержание |
|--------|-------------|
| **Стадия (7)** | Stage как гиперграф воркфлоу; State (объект с атрибутами); state_input_map, state_output_map; can_run; три типа стадий (Автор, Творец, Среда) и их воркфлоу; взаимодействие с директорией мира. |
| **Мир (8)** | World как гиперграф стадий; цикл автор → среда → творец; state_schema; контент и исполняемая часть; End World; run(world, state, action, context). |
| **Вселенная (9)** | Universe как гиперграф миров; эфир; payload_spec; переход сущностей; run(universe, world_inputs). |
| **Сериализация (10)** | Единый контракт get_config, state_dict, save/load на всех уровнях; версионирование; тесты на воспроизводимость; опционально хранение в БД. |

### 2.2 Когда приступать к Части 2

После полного завершения Части 1: движок стабилен, тесты зелёные, гиперграф и воркфлоу работают в целевых сценариях. Документы PHASE_7–PHASE_10 будут пересмотрены с учётом опыта реализации Части 1 и реальных сценариев использования.
