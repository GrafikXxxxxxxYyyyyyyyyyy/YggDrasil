**Результат:** явная структура каталогов; возможность импортировать пакет и запускать тесты.

## Фаза 1. Фундамент: Block, Node, Port, реестр

**Цель:** реализовать минимальные сущности фундаментального уровня — блок, узел, порты, реестр типов блоков.

**Канон:** [documentation/docs/01_FOUNDATION.md](../documentation/docs/01_FOUNDATION.md).

**Детальный технический план:** [PHASE_1_FOUNDATION.md](PHASE_1_FOUNDATION.md) — контракты Port, Block, Node, Registry; API и сигнатуры; тесты по компонентам; порядок реализации; приёмочные критерии и граничные случаи.

**Шаги:**

1. **Port (порт)**  
   - Имя, тип (или схема), опциональность (обязательный/опциональный), направление (in/out).  
   - Опционально: политика агрегации для входа с несколькими рёбрами (concat, sum, first).

2. **Abstract Base Block**  
   - Атрибуты: `block_id`, `block_type`, конфиг.  
   - Объявление портов: метод или свойство `input_ports`, `output_ports` (список/словарь портов).  
   - Выполнение: `run(inputs: dict) -> dict` (словари по именам портов).  
   - Сериализация: `state_dict()`, `load_state_dict(state_dict)` (для блока без весов — пустой или идентичность).  
   - Референс: 01 §2.

3. **Abstract Graph Node**  
   - Содержит ровно один блок; идентификатор узла `node_id`.  
   - Не хранит данные — только ссылка на блок и привязка к графу (входящие/исходящие рёбра задаются на уровне графа).  
   - Интерфейс для движка: `run(inputs) -> outputs` делегирует в `block.run(inputs)`.  
   - Референс: 01 §3.

4. **Registry (реестр типов блоков)**  
   - Отображение `block_type` (строка) → фабрика (класс или функция), создающая блок по конфигу.  
   - `register(block_type, factory)`, `create(block_type, config) -> Block`.  
   - Референс: 01 §7 (2.8).

**Результат:** можно инстанцировать блоки по типу из реестра, объявлять порты, вызывать `block.run(inputs)`. Определены два равноправных начала: Block (данные, вычисления, материальная сущность) и Node (положение, связи в графе); не смешивать и не подменять. Узлы-задачи (наследующие оба, фаза 4) — единственные сущности, встраиваемые в гиперграф; гиперграф собирается только из них и является цельной единицей системы для полной задачи.

## Фаза 2. Гиперграфовый движок

**Детальный технический план:** [PHASE_2_ENGINE.md](PHASE_2_ENGINE.md).

**Цель:** реализовать пять компонентов движка (хранилище структуры, валидатор, планировщик, буферы, исполнитель), чтобы по графу узлов и гиперрёбер выполнять обход и вызов run.

**Канон:** [documentation/docs/HYPERGRAPH_ENGINE.md](../documentation/docs/HYPERGRAPH_ENGINE.md), 01 §5–6, 03 §6–7.

**Шаги:**

1. **Structure Storage (хранилище структуры)**  
   - Узлы: `node_id -> Node`.  
   - Гиперрёбра: список/множество рёбер; каждое ребро — концы (источники и приёмники) в виде пар `(node_id, port_name)`.  
   - Внешние входы/выходы: списки `(node_id, port_name, name?)`.  
   - Опционально: `execution_version` для инвалидации кэша плана при изменении графа.

2. **Validator (валидатор)**  
   - Проверка: все node_id и port_name в рёбрах существуют; типы портов на концах совместимы; обязательные входы покрыты.  
   - Для графов с циклами: проверка, что задана семантика (например, число итераций или graph_kind).  
   - Возврат: список ошибок/предупреждений; при строгом режиме — исключение или флаг «не валиден».

3. **Execution Planner (планировщик)**  
   - Вход: структура графа (узлы, гиперрёбра).  
   - Выход: **план** — порядок вызовов. Для DAG: топологическая сортировка. Для графа с циклом: разбиение на фазы (начальная, циклическая «K раз [узлы]», конечная).  
   - Кэш: план кэшируется, привязан к execution_version; при изменении графа кэш инвалидируется.

4. **Edge Buffers (буферы)**  
   - В рамках одного run: для каждого приёмника `(node_id, port_name)` — буфер значения.  
   - Инициализация: внешние входы графа записываются в буферы соответствующих портов.  
   - После run узла: выходы узла записываются в буферы всех приёмников по исходящим рёбрам; при нескольких рёбрах на один порт — агрегация по политике.  
   - Внешние выходы графа заполняются из буферов по списку exposed_outputs.

5. **Executor (исполнитель)**  
   - Вход: граф (или ссылка на хранилище + планировщик + буферы), входные данные (словарь по внешним входам), опции (num_loop_steps, max_steps, seed и т.д.).  
   - Алгоритм: валидация (опционально) → получение/построение плана → инициализация буферов из inputs → обход по плану (для каждого шага: собрать входы узла из буферов, вызвать `node.run(inputs)`, записать выходы в буферы) → сбор выходов по exposed_outputs в словарь outputs.  
   - Возврат: `outputs` (словарь по внешним выходам).

**Результат:** единый движок, который по графу узлов и гиперрёбер выполняет run без доменной логики; поддерживаются DAG и один итеративный цикл (K шагов). Agent_loop и несколько циклов можно добавить на следующей фазе.

## Фаза 3. Гиперграф задачи

**Детальный технический план:** [PHASE_3_TASK_HYPERGRAPH.md](PHASE_3_TASK_HYPERGRAPH.md).

**Цель:** класс «гиперграф задачи» (узлы = экземпляры узлов-задач), построение (add_node по block_type/config, add_edge, expose_input/output), from_config, to_config, вызов run через движок.

**Канон:** [documentation/docs/03_TASK_HYPERGRAPH.md](../documentation/docs/03_TASK_HYPERGRAPH.md).

**Шаги:**

1. **Hypergraph (или аналог)**  
   - Хранит узлы (node_id → Node с блоком), гиперрёбра, exposed_inputs, exposed_outputs.  
   - Методы: `add_node(node_id, block_type, config, …)`, `add_edge(source_node, source_port, target_node, target_port)`, `add_hyperedge(ends)` при арности > 2, `expose_input(...)`, `expose_output(...)`, `remove_node`, `remove_edge` при необходимости.  
   - При любом изменении структуры — инкремент execution_version (или инвалидация кэша плана движка).

2. **Интеграция с движком**  
   - Hypergraph передаёт в Executor свою структуру (узлы, рёбра, exposed_inputs/outputs).  
   - Контракт: `run(hypergraph, inputs, **options) -> outputs`.  
   - Опции: `num_loop_steps` (для циклической фазы), `validate=True`, и т.д.

3. **Циклы**  
   - Планировщик должен уметь распознавать цикл в графе (например, Backbone ↔ Inner Module) и строить итеративный план с K итерациями.  
   - K задаётся из options или из конфига гиперграфа (graph_kind, num_loop_steps).

4. **Agent_loop (опционально на этой фазе)**  
   - Если у узла в выходе есть поле `tool_calls`, движок входит в подцикл: вызов узла → выполнение инструментов по tool_id → сбор tool_results → повторный вызов узла; до отсутствия tool_calls или max_steps.  
   - Можно отложить на фазу 4 и сначала обеспечить простой DAG и один итеративный цикл.

**Результат:** можно собрать гиперграф задачи из узлов и рёбер, объявить внешние входы/выходы и вызвать `run(hypergraph, inputs, **options)`; движок выполняет обход и возвращает outputs.

## Фаза 4. Абстрактные узлы-задачи (семь ролей)

**Детальный технический план:** [PHASE_4_ABSTRACT_TASK_NODES.md](PHASE_4_ABSTRACT_TASK_NODES.md).

**Цель:** ввести семь ролей узлов-задач (Backbone, Injector, Conjector, Inner Module, Outer Module, Helper, Converter) как классы/интерфейсы и зарегистрировать их в реестре; минимальные заглушки для проверки графа.

**Канон:** [documentation/docs/02_ABSTRACT_TASK_NODES.md](../documentation/docs/02_ABSTRACT_TASK_NODES.md).

**Шаги:**

1. **Базовый класс или интерфейс для каждой роли**  
   - Каждая роль — наследник Block (и при необходимости Node); объявление портов по канону (входы/выходы по ролям).  
   - Backbone: ядро задачи (вход/выход по задаче).  
   - Injector, Conjector, Inner Module, Outer Module, Helper, Converter — свои порты и семантика по документу 02.

2. **Регистрация в реестре**  
   - Зарегистрировать типы вида `backbone/...`, `conjector/...`, `inner_module/...` и т.д.  
   - Создание экземпляра по `block_type` и config через реестр.

3. **Заглушки (stubs)**  
   - Минимальные реализации (например, передача входа на выход, или фиксированный вывод), чтобы собрать тестовый гиперграф (например, цепочка Converter → Backbone → Converter) и прогнать run.

4. **Agent_loop (если не сделано в фазе 3)**  
   - Узел-агент: выход может содержать `tool_calls`; таблица tool_id → node_id в конфиге гиперграфа; движок реализует подцикл вызова агента и инструментов.

**Результат:** семь ролей задекларированы и доступны через реестр; можно собирать графы из заглушек и реальных блоков (если уже есть), запускать run и тесты.

## Фаза 5. Сериализация блока и гиперграфа задачи

**Детальный технический план:** [PHASE_5_SERIALIZATION.md](PHASE_5_SERIALIZATION.md).

**Цель:** конфиг + чекпоинт для блока и для гиперграфа задачи; сохранение и загрузка из чекпоинта.

**Канон:** [documentation/docs/SERIALIZATION.md](../documentation/docs/SERIALIZATION.md), 01 §2.7, 03 §9.

**Шаги:**

1. **Блок**  
   - `get_config()` / конфиг при создании: block_type, block_id, параметры.  
   - `state_dict()`, `load_state_dict(state_dict)`.  
   - Сохранение: конфиг (YAML/JSON) + чекпоинт (например, pickle или torch.save для весов).

2. **Гиперграф задачи**

## Фаза 6. Воркфлоу

**Детальный технический план:** [PHASE_6_WORKFLOW.md](PHASE_6_WORKFLOW.md).

**Цель:** воркфлоу как гиперграф, узлы которого — целые гиперграфы задач; тот же движок; run(workflow, inputs, **options) -> outputs.

**Канон:** [documentation/docs/04_WORKFLOW.md](../documentation/docs/04_WORKFLOW.md).

**Шаги:**

1. **Workflow**  
   - Узлы: graph_id → Hypergraph (или ссылка на конфиг + ленивая загрузка).  
   - Гиперрёбра: связь внешних выходов одного гиперграфа с внешними входами другого (пары (graph_id, port_name) или аналог).  
   - Exposed inputs/outputs на уровне воркфлоу.

2. **Движок для воркфлоу**  
   - Тот же планировщик, буферы, исполнитель: узел = гиперграф, вызов узла = run(hypergraph, inputs).  
   - Поддержка циклов между гиперграфами: итеративная схема (K раз [G1, G2, …]).

3. **Сериализация воркфлоу**  
   - Конфиг: graphs (graph_id → конфиг гиперграфа или ref), edges, exposed_inputs, exposed_outputs, schema_version, workflow_id.  
   - Чекпоинт: graph_id → чекпоинт гиперграфа; загрузка только чекпоинтов в уже собранный воркфлоу.

**Результат:** можно собрать воркфлоу из нескольких гиперграфов задач, вызвать run(workflow, inputs) и получить outputs; сохранить/загрузить воркфлоу.

## Фаза 7. Стадия (экспериментальная, требует глубокой переработки)

**Статус:** экспериментальная фаза, схема уровня стадии ещё формируется и требует **глубокой детальной переработки и осмысления всей логики** (особенно во взаимодействии с миром, файловой директорией мира, авторскими/творческими стадиями и агентами). Текущий план служит ориентиром, а не окончательной спецификацией.

**Цель (общая):** стадия как уровень, который связывает воркфлоу и `State` (объект с атрибутами для автора/творца/среды и файловой директорией мира): state_input_map, state_output_map, condition can_run; базовые шаблоны стадий «Автор», «Творец», «Среда» с фиксированным числом внутренних воркфлоу и работой с файлами мира.

**Канон:** [documentation/docs/05_STAGE.md](../documentation/docs/05_STAGE.md) + обновлённый контекст канона мира (уровень World).

**Заметка:** подробный технический план для фазы 7 будет переосмыслен и записан заново в отдельном документе (например, `PHASE_7_STAGE.md`) после уточнения общей логики мира и агентов. Пока реализация стадии не рекомендуется как «стабильная» часть фреймворка; её следует рассматривать как экспериментальный слой поверх уже зафиксированных фаз 0–6.

## Фаза 8. Мир (экспериментальная, требует глубокой переработки)

**Статус:** экспериментальная фаза. Схема уровня мира, связь `State` с файловой директорией мира, взаимодействие стадий Автора/Творца/Среды и поддержка End World требуют **глубокого детального осмысления, логической структуризации и дальнейшей переработки**. Текущий план носит ориентировочный характер.

**Цель (общая):** мир как гиперграф стадий; фиксированный цикл автор → среда → творец; state_schema; первая итерация, action; контент мира; run(world, state?, action?, context?, **options) -> state; End World.

**Канон:** [documentation/docs/06_WORLD.md](../documentation/docs/06_WORLD.md).

**Шаги:**

1. **World**  
   - Узлы: стадии (автор, среда, творец); порядок обхода — цикл по трём стадиям.  
   - state_schema: описание слотов state и их семантики.  
   - Условия can_run для каждой стадии; первая итерация (мир может стартовать без предзаданного state, например с шага Development в среде).  
   - Action: опциональный ввод пользователя для стадии среды (Development).

2. **Контент мира**  
   - История, глоссарий, персонажи, сюжет — отдельная структура; обновляется при World update (в среде); сериализация мира — два поля: исполняемая часть (структура + чекпоинты) и контент.

3. **End World**  
   - Мир только с контентом, без исполняемой части; загрузка = загрузка контента; в вселенной выступает узлом без исходящих рёбер с записью.

4. **Сериализация мира**  
   - Два поля: (1) конфиг исполняемой части + чекпоинт мира; (2) контент мира.  
   - Загрузка: восстановление структуры + контент + чекпоинты стадий.

**Результат:** run(world, state?, action?, context?, **options) -> state; мир сохраняем/загружаем в двух полях; End World поддерживается.

## Фаза 9. Вселенная (экспериментальная, требует глубокой переработки)

**Статус:** экспериментальная фаза. Логика взаимодействия миров, эфир, payload_spec, перенос сущностей и границы онтологии «вселенная» требуют **дальнейшего уточнения и глубокого анализа**; текущий план — набросок целевого устройства.

**Цель (общая):** вселенная как гиперграф миров; узлы — миры (развивающиеся и End World); гиперрёбра — обмен по payload_spec; эфир; run(universe, world_inputs?, **options) -> results.

**Канон:** [documentation/docs/07_UNIVERSE.md](../documentation/docs/07_UNIVERSE.md).

**Шаги:**

1. **Universe**  
   - Узлы: world_id → World (развивающийся или End World).  
   - Гиперрёбра: payload_spec (что передаётся — state snapshot, артефакты, сущности и т.д.).  
   - Ether: общий контекст (глоссарий, правила, поток событий); доступен мирам при run.

2. **Исполнитель вселенной**  
   - План: порядок обхода миров (топология или фиксированный список).  
   - Для каждого мира: run(world, …) с входами из world_inputs и из буферов по входящим рёбрам; запись выходов в буферы и в results.

3. **Переход сущностей, кросс-мировое влияние**  
   - По payload_spec могут передаваться сущности с origin_world_id; влияют на историю и контент мира-источника и мира-приёмника. Детали — по документу 07.

4. **Сериализация вселенной**  
   - Конфиг: worlds (world_id → конфиг или ref), edges, ether, order, universe_id, schema_version.  
   - Чекпоинт: агрегат чекпоинтов только развивающихся миров; End World — только контент.

**Результат:** run(universe, world_inputs?, **options) -> results; вселенная сохраняема/загружаема; граница онтологии — уровень вселенной, метавселенная не вводится.

## Фаза 10. Сериализация всех уровней и полировка (экспериментальная, требует глубокой переработки)

**Статус:** экспериментальная фаза. Единый контракт сериализации для мира/вселенной, версия форматов и интеграция с реальными сценариями использования требуют **глубокого детального пересмотра и согласования с финальной канонической схемой уровней 7–9**.

**Цель (общая):** единообразные интерфейсы save/load на всех уровнях; загрузка только чекпоинтов где предусмотрено; версионирование schema_version; тесты на воспроизводимость.

**Канон:** [documentation/docs/SERIALIZATION.md](../documentation/docs/SERIALIZATION.md).

**Шаги:**

1. Для каждого уровня (блок, гиперграф задачи, воркфлоу, стадия, мир, вселенная) — явный контракт: `get_config()`, `state_dict()`, `load_state_dict(...)`, `from_config(config)`, `load_from_checkpoint(path)` где применимо.
2. Мир: два поля (исполняемая часть и контент); вселенная: чекпоинт только по развивающимся мирам.
3. Дедупликация общих блоков при сохранении/загрузке (block_id → один state_dict).
4. Опционально: хранение конфига и чекпоинта в БД (по принципам философии — без смены онтологии, только способ персистенции).

**Результат:** на любом уровне можно сохранить и загрузить сущность; воспроизводимость при одинаковых конфиге, чекпоинте и входах.
