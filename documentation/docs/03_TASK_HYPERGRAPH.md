# 03. Уровень гиперграфа задачи

**Назначение:** полная **техническая** документация **первого исполняемого уровня** системы Иггдрасиль — гиперграфа задачи. Документ задаёт структуру гиперграфа (узлы = экземпляры узлов-задач, гиперрёбра = связи портов), внешние входы и выходы, построение и валидацию, **гиперграфовый движок** (планирование выполнения, буферы, исполнитель), обработку циклов и агентного цикла, контракт run, сериализацию, типичные сценарии и граничные случаи. Канонический источник технических решений уровня гиперграфа задачи.

**Предыдущие уровни:** [01_FOUNDATION.md](01_FOUNDATION.md) — Block, Node, гиперграф, исполнитель, реестр; [02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md) — семь абстрактных узлов-задач и их контракты.

**Философский якорь:** [../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §3 — уровень гиперграфа как первая единая исполняемая конструкция; один гиперграф = одна задача.

**Референс:** [../../reference/Universe_Generator_3.0/docs/02_HYPERGRAPH_LEVEL.md](../../reference/Universe_Generator_3.0/docs/02_HYPERGRAPH_LEVEL.md), [../../reference/WorldGenerator_2.0/HYPERGRAPH_AND_HYPERLINKS.md](../../reference/WorldGenerator_2.0/HYPERGRAPH_AND_HYPERLINKS.md).

**Язык:** русский.

## 1. Место уровня в иерархии

| Уровень | Сущность | Описание |
|---------|----------|----------|
| **Фундамент** | Block, Node, гиперграф, исполнитель, реестр | Структура и контракт; один узел с одним блоком уже исполняем. [01_FOUNDATION.md](01_FOUNDATION.md). |
| **Абстрактные узлы-задачи** | Backbone, Injector, Conjector, Inner Module, Outer Module, Helper, Converter | Спецификации ролей, контракты портов, типичные связи. [02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md). |
| **Гиперграф задачи** | Гиперграф, узлы которого — **экземпляры** узлов-задач | **Первый полноценно исполняемый уровень:** один гиперграф = **одна цельная задача**; по гиперрёбрам передаются **данные** между портами. |
| **Воркфлоу** | Гиперграф гиперграфов (узлы = целые гиперграфы задач) | Комбинированные сценарии (несколько задач в цепочке или ветвлении). |

На **уровне гиперграфа задачи** узлы — не абстракции, а **конкретные реализации** (UNet2D, inner_module/ddim, VAE, LoRA, CLIP-энкодер, агент и т.д.), собранные в один гиперграф. По нему выполняется **run**: данные поступают на внешние входы, проходят по узлам согласно топологии (с учётом циклов) и снимаются с внешних выходов. Комбинирование нескольких задач (текст → картинка → апскейл → видео) делается на уровне **воркфлоу**, а не путём раздувания одного гиперграфа.

## 2. Гиперграф задачи — одна цельная задача

### 2.1 Определение

**Гиперграф задачи** на данном уровне задаёт **одну законченную цель** (одну задачу). Внутри одного гиперграфа собирается **полная реализация** задачи: от внешних входов до внешних выходов. Все узлы — экземпляры узлов-задач (роли Backbone, Injector, Conjector, Inner Module, Outer Module, Converter, Helper в нужной комбинации); гиперрёбра соединяют порты узлов; данные передаются по гиперрёбрам при выполнении run.

### 2.2 Примеры задач, реализуемых одним гиперграфом

| Задача | Краткое описание |
|--------|------------------|
| **Text-to-image** | Текст на входе → изображение на выходе. В одном гиперграфе: Converter (токенизатор), Conjector (CLIP-энкодер, условие), Outer Module (начальный шум, расписание), Backbone (UNet/DiT), Inner Module (солвер DDIM, опционально ControlNet), Converter (VAE decode). |
| **Image-to-video** | Изображение → видео. Полная цепочка внутри одного гиперграфа. |
| **Upscale** | Изображение низкого разрешения → изображение высокого разрешения. |
| **Image-to-image** | Стилизация, перевод домена, inpainting и т.д. |
| **Распознавание** | Классификация, детекция (не обязательно диффузия). |
| **Агентный сценарий** | Агент + инструменты в одном гиперграфе; вход — prompt/наблюдение, выход — финальный response после агентного цикла. |
| **RAG-генерация** | Поиск по корпусу (Helper) → контекст в Conjector → Backbone (LLM) → ответ. |

Гиперграф **не обязан** содержать все семь типов узлов; достаточно тех, что нужны для данной задачи ([02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md) §3.3, §12.1).

### 2.3 Граница с воркфлоу

**Воркфлоу** соединяет **несколько задач** (несколько гиперграфов): выход одного гиперграфа становится входом другого. Один гиперграф задачи остаётся **атомарной единицей** одной задачи; разбивать одну задачу на несколько гиперграфов только ради «удобства» не требуется — внутри одного гиперграфа умещается и цикл деноизинга, и кондиционирование, и декодирование.

## 3. Состав гиперграфа: узлы и гиперрёбра

### 3.1 Узлы

- **Узлы** гиперграфа задачи — **экземпляры** узлов-задач (Backbone, Injector, Conjector, Inner Module, Outer Module, Converter, Helper), созданные по **block_type** и **config** через реестр типов блоков ([01_FOUNDATION.md](01_FOUNDATION.md) §7, [02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md) §11).
- Каждый узел имеет:
  - **node_id** — уникальный в рамках гиперграфа идентификатор (строка или стабильный ID);
  - **блок** — экземпляр с портами и методом run (реализация одной из семи абстракций).
- Данные хранит **блок**; **узел** задаёт положение в графе и связи (входящие и исходящие гиперрёбра). Правило «блок → узел» ([01_FOUNDATION.md](01_FOUNDATION.md) §4) сохраняется: блок помещён в узел и участвует в графе через порты узла.

### 3.2 Гиперрёбра

- **Гиперребро** соединяет порты узлов. Частный случай — пара источник–приёмник: `(source_node_id, source_port_name) → (target_node_id, target_port_name)`. Движок **нативно** поддерживает гиперрёбра **произвольной арности** ([01_FOUNDATION.md](01_FOUNDATION.md) §5.1): одно гиперребро может связывать более двух концов.
- **Направление гиперребра:** у каждого гиперребра задаётся, какие концы — **источники** (выходные порты), какие — **приёмники** (входные порты). Типичные случаи: (1) один источник → несколько приёмников (раздача одного значения по нескольким портам); (2) несколько источников → один приёмник (агрегация по политике). При арности 2 — один источник, один приёмник. Конкретное кодирование в `add_hyperedge(ends)` задаётся реализацией (например, список пар (node_id, port_name) с разметкой in/out или порядок: сначала источники, затем приёмники).
- По гиперрёбрам передаются **данные** (тензоры, словари, сообщения и т.д.) — то, что течёт между портами блоков-задач. Тип данных на гиперребре должен быть **согласован** с типами портов на концах; уровень гиперграфа не предписывает конкретную систему типов (тензор, схема, семантическое имя), только требование совместимости при валидации.
- **Один выходной порт** может быть источником **нескольких** гиперрёбер (одно значение раздаётся нескольким приёмникам). **Один входной порт** может быть приёмником одного или нескольких гиперрёбер; при нескольких входящих рёбрах на один порт требуется **политика агрегации** (concat, sum, first и т.д.), задаваемая контрактом блока или конфигом ребра/узла.

### 3.3 Внешние входы и выходы гиперграфа

- **Внешние входы (exposed inputs):** список записей `(node_id, port_name, name?)`. Порт считается входом гиперграфа. При вызове `run(hypergraph, inputs)` ключи словаря **inputs** — либо **name** (если задан при expose_input), либо канонический идентификатор `(node_id, port_name)` или строка `"node_id:port_name"` по соглашению реализации. Значения подаются на соответствующие порты.
- **Внешние выходы (exposed outputs):** список записей `(node_id, port_name, name?)`. Ключи словаря **outputs**, возвращаемого run, — либо **name** (если задан при expose_output), либо канонический идентификатор порта. Значения снимаются с этих портов после выполнения.
- Гиперграф вызывается как **чёрный ящик**: внешний наблюдатель (воркфлоу, этап, тест) передаёт данные на внешние входы и получает данные с внешних выходов; внутренняя топология и типы узлов скрыты за контрактом «входы/выходы гиперграфа».

## 4. Построение гиперграфа

### 4.1 Добавление и удаление узлов

- **add_node(node_id, block_type, config, …)** — создать узёл-задачу через реестр по `block_type` и `config` (реестр возвращает объект Block+Node с заданным node_id), добавить в гиперграф. Опционально: загрузка весов (pretrained), флаг trainable, block_id.
- **add_node(Node)** — добавить готовый узел (например, при загрузке из конфига).
- **remove_node(node_id)** — удалить узел и связанные с ним гиперрёбра; инвалидирует кэш плана (execution_version). При загрузке из конфига удаление обычно не используется; при программной сборке графа — допустимо.
- При добавлении узла может использоваться **автосвязывание (auto_connect)**: по типу узла-задачи (backbone, inner_module, conjector, converter и т.д.) система выводит типичные связи с уже существующими узлами и предлагает или создаёт гиперрёбра автоматически ([02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md) §§ 4.5–10.5, §12.2). Это позволяет собирать гиперграф задачи (например, text-to-image) **в несколько шагов** — добавление нужных узлов и при необходимости явное связывание нестандартных портов.

### 4.2 Добавление и удаление гиперрёбер

- **add_edge(source_node_id, source_port, target_node_id, target_port)** — добавить гиперребро (в частном случае — ребро размера 2) между портами. Все четыре идентификатора должны существовать (узлы и порты объявлены в блоках).
- **add_hyperedge(ends)** — добавить гиперребро произвольной арности: `ends` — список пар `(node_id, port_name)` (источники и приёмники по соглашению реализации). Типы портов на всех концах должны быть согласованы.
- **remove_edge(source_node_id, source_port, target_node_id, target_port)** — удалить ребро размера 2. Для гиперребра арности > 2 реализация может предоставлять **remove_hyperedge(ends)** — удаление гиперребра по списку концов; инвалидирует кэш плана.
- Валидация при добавлении (или при явном вызове `validate`): типы портов на концах совместимы; дубликаты рёбер по смыслу не создаются (один и тот же `(source, port) → (target, port)` не дублируется).

### 4.3 Объявление внешних входов и выходов

- **expose_input(node_id, port_name, name?)** — пометить порт узла как внешний вход гиперграфа. Опционально задаётся **name** для ключа в словаре `inputs` при вызове run.
- **expose_output(node_id, port_name, name?)** — пометить порт узла как внешний выход гиперграфа.
- Без объявленных внешних входов/выходов гиперграф не может быть использован как чёрный ящик извне (воркфлоу, этап); для внутренних тестов или вложенного вызова можно вызывать run с подачей данных напрямую на порты узлов по топологии.

### 4.4 Построение из конфига

- **Конфиг гиперграфа** задаёт: **nodes** (список записей `{node_id, block_type, block_id?, config?, trainable?}`), **edges** (список `{source_node, source_port, target_node, target_port}`; при поддержке гиперрёбер большей арности — список концов с разметкой источника/приёмника), **exposed_inputs**, **exposed_outputs** (списки `{node_id, port_name, name?}`). Опционально: graph_id, **graph_kind**, metadata, schema_version.
- **graph_kind** — опциональный тег конфига (например, `"diffusion"`, `"agent"`, `"rag"`). Подсказка планировщику и валидатору для распознавания шаблона графа; не заменяет анализ структуры (топология, циклы), но может ускорить построение плана и проверки. Реализация может использовать graph_kind для выбора стратегии планирования или шаблона валидации.
- **from_config(config, registry?, validate?)** — создать гиперграф из конфига: для каждого узла вызвать `registry.build(config узла)`, добавить узлы, добавить гиперрёбра, восстановить exposed_inputs/outputs. При `validate=True` выполнить валидацию (§5). При загрузке конфиг и чекпоинт задают воспроизводимое поведение ([01_FOUNDATION.md](01_FOUNDATION.md) §9).

### 4.5 Версионирование структуры

- При любом изменении структуры (add_node, remove_node, add_edge, remove_edge, изменение exposed_inputs/exposed_outputs) рекомендуется инкрементировать **execution_version** (или эквивалент) гиперграфа. Это используется движком для **инвалидации кэша плана выполнения**: при следующем run планировщик перестраивает план, если версия изменилась (§6).

## 5. Валидация гиперграфа

Перед выполнением или сериализацией выполняется **валидация** структуры. Валидатор проверяет, что идеальная структура **реализуема** ([01_FOUNDATION.md](01_FOUNDATION.md) §8).

### 5.1 Обязательные проверки

| Проверка | Описание |
|----------|----------|
| **Существование узлов и портов** | Все node_id в гиперрёбрах и в exposed_inputs/outputs существуют в множестве узлов. Все port_name соответствуют объявленным портам блоков (входы/выходы по контракту узла-задачи). |
| **Совместимость типов** | Тип данных на выходном порте источника и тип входного порта приёмника совместимы (по правилам системы или по явной схеме типов). |
| **Обязательные порты** | Обязательные входные порты узлов имеют хотя бы одно входящее гиперребро (или значение с внешнего входа), если иное не задано политикой (опциональный порт, значение по умолчанию). |
| **Внешние входы/выходы** | Списки exposed_inputs и exposed_outputs ссылаются на существующие (node_id, port_name). |

### 5.2 Дополнительные проверки

| Проверка | Описание |
|----------|----------|
| **Связность (опционально)** | Узлы, участвующие во внешних входах, достижимы от входов; от них достижимы узлы внешних выходов (если граф не допускает «висячих» узлов). |
| **Циклы** | При наличии циклов (например, Backbone ↔ Inner Module в диффузии) исполнитель должен поддерживать итеративную схему (§6, §7). Валидатор может предупреждать о циклах и проверять, что цикл помечен или распознаваем (num_loop_steps, graph_kind). |
| **Соответствие шаблону задачи (опционально)** | Валидатор может проверять соответствие графа заявленному шаблону (graph_kind или явная схема): например, «диффузия изображений» — наличие Backbone, Inner Module (солвер), опционально Conjector, Outer Module, Converter. |

### 5.3 Результат валидации

- Валидация возвращает список **ошибок** (блокирующие выполнение) и **предупреждений**. В строгом режиме run не выполняется при наличии ошибок; при загрузке из конфига с `validate=True` невалидный гиперграф не создаётся или выдаёт явное исключение.

## 6. Гиперграфовый движок

**Гиперграфовый движок** — центральный механизм выполнения: он обеспечивает единую модель обхода узлов, передачи данных по гиперрёбрам и вызова run узлов. На уровне гиперграфа задачи движок работает с узлами = экземплярами узлов-задач; на уровне воркфлоу — с узлами = гиперграфами (каждый из которых внутри себя выполняется тем же движком или делегирует ему). Движок **не содержит доменной логики** конкретных задач (диффузия, LLM, агент): только структура, порядок обхода, буферы и вызов run.

### 6.1 Компоненты движка

| Компонент | Назначение |
|-----------|------------|
| **Хранилище структуры** | Множество узлов-задач (node_id → объект Block+Node), множество гиперрёбер, списки exposed_inputs и exposed_outputs. Версия структуры (execution_version) для инвалидации кэша плана. |
| **Валидатор** | Проверка согласованности структуры (§5). Вызывается до первого run или при from_config(validate=True). |
| **Планировщик (execution planner)** | Строит **план выполнения**: порядок обхода узлов, разбиение на фазы при наличии циклов (например, «повторить N раз: [Backbone, Inner Module]»), распознавание узла-агента и вставка подплана agent_loop. План кэшируется и привязан к execution_version. |
| **Буферы (edge buffers)** | Во время run каждому приёмнику (target_node_id, target_port) сопоставляется **буфер значения**: результат с (source_node_id, source_port). Для внешних входов буферы инициализируются из словаря inputs; для внешних выходов значения из буферов считываются в словарь outputs по окончании run. При нескольких рёбрах на один порт применяется политика агрегации. |
| **Исполнитель (executor)** | По плану выполнения для каждого шага: (1) разрешить входы узла (собрать из буферов входящих гиперрёбер и внешних входов), (2) вызвать `node.block.run(inputs)`, (3) записать выходы в буферы исходящих гиперрёбер и при необходимости во внешние выходы. При обнаружении tool_calls (узел-агент) — выполнить подцикл вызова инструментов и повторного вызова узла до завершения. |

### 6.2 Планирование выполнения

- **Цель плана:** определить последовательность вызовов узлов так, чтобы к моменту вызова узла все его входные порты были заполнены (входящими гиперрёбрами или внешними входами).
- **DAG (без циклов):** достаточно топологической сортировки по направлению гиперрёбер (Kahn или DFS). План = этот порядок; выполнение — один проход.
- **Гиперграф с циклом:** цикл (например, Backbone → Inner Module → Backbone) не допускает единственной топологической сортировки. План задаётся как **итеративная схема**: (1) **начальная фаза** — узлы, не входящие в цикл (например, Conjector, Converter encode, Outer Module — начальный шум); (2) **циклическая фаза** — повторять K раз пару [Backbone, Inner Module] (или заданный подграф); (3) **конечная фаза** — узлы после цикла (например, Converter decode). Число K задаётся конфигом (num_inference_steps) или опцией run. Планировщик распознаёт циклы (анализ сильно связных компонент или явная разметка graph_kind) и формирует такой план.
- **Узел-агент (agent_loop):** план для графа с агентом содержит метку «узел agent_id; режим agent_loop». Исполнитель при выполнении этого узла: вызывает agent.run(inputs); если в выходе есть tool_calls — выполняет соответствующие узлы-инструменты, собирает tool_results, снова вызывает agent.run(inputs + tool_results); повторяет до отсутствия tool_calls или до max_steps (§7.3).

### 6.3 Кэширование и инвалидация плана

- После первого run (или явного вызова build_plan()) планировщик сохраняет построенный план и значение execution_version.
- При изменении структуры гиперграфа execution_version увеличивается; при следующем run при несовпадении версии план перестраивается. Изменение только **данных** (inputs) не меняет структуру — план остаётся валидным.

### 6.4 Буферы и разрешение портов

- **Инициализация:** для каждого внешнего входа (node_id, port_name) записать в буфер значение из inputs.
- **Разрешение входов узла:** для узла N и его входного порта P собрать все гиперрёбра, у которых (N, P) — приёмник; взять значения из буферов (и если (N, P) в exposed_inputs — добавить из inputs). Сформировать словарь inputs для `node.block.run(inputs)`.
- **Запись выходов:** после run для каждого выходного порта P и каждого исходящего гиперребра записать outputs[P] в буферы приёмников. Если (N, P) в exposed_outputs — записать в слот внешнего выхода.
- **Политика агрегации:** если на один входной порт приходит несколько гиперрёбер, значение для порта вычисляется по правилу (concat, sum, first и т.д.), заданному контрактом блока или конфигом.
- Буферы создаются в начале run и уничтожаются в конце; между вызовами run не сохраняются (в отличие от состояния внутри блоков).

### 6.5 Жизненный цикл одного run

1. **Вход:** hypergraph, inputs, options (training, device, num_loop_steps, max_steps, seed и т.д.).
2. **Валидация (опционально):** при первом run или по опции выполнить валидацию структуры (§5), если она ещё не выполнялась после последнего изменения графа; при наличии ошибок — не выполнять run (или выбросить исключение по политике реализации).
3. **Проверка плана:** если закэшированный план устарел (execution_version изменился), вызвать планировщик, обновить кэш плана.
4. **Инициализация буферов:** для каждого внешнего входа (node_id, port_name) записать в буфер значение из inputs (по ключу name или по (node_id, port_name)).
5. **Обход по плану:** для каждого шага плана (узел или agent_loop): разрешить входы узла из буферов и внешних входов; вызвать `node.block.run(inputs)`; записать выходы в буферы и при необходимости во внешние выходы. Для agent_loop — внутренний подцикл до завершения агента.
6. **Сбор выходов:** по списку exposed_outputs собрать из буферов значения в словарь outputs (ключи — name или канонический идентификатор порта).
7. **Выход:** вернуть outputs.

Доменная логика задач (диффузия, LLM, агент) не входит в этот цикл — только структура, план, буферы и вызов run узлов.

### 6.6 Единая модель — любой масштаб

- С точки зрения движка узел — **чёрный ящик** с объявленными входами/выходами и операцией run(inputs) → outputs. Движок не «знает», что внутри узла: блок-задача, вложенный гиперграф или иная реализация. Такая унификация позволяет на уровне воркфлоу подставлять на место узла целый гиперграф задачи (который внутри выполняется тем же движком). **Расширяемость без изменений движка:** новые типы узлов добавляются через реестр и контракт; движок не меняется.

## 7. Порядок выполнения и циклы

### 7.1 Топологический порядок (DAG)

- При отсутствии циклов порядок обхода узлов определяется **топологической сортировкой** по направлению гиперрёбер (от внешних входов к внешним выходам). Исполнитель вызывает узлы в этом порядке; данные передаются по буферам.

### 7.2 Цикл Backbone ↔ Inner Module (диффузия и аналоги)

- Типичный цикл на уровне гиперграфа задачи: **Backbone** (шаг предсказания) → **Inner Module** (солвер: latent, pred, timestep → next_latent, next_timestep) → снова **Backbone** (на следующей итерации). Число итераций K задаётся конфигом (num_inference_steps) или опцией run.
- На каждой итерации: сначала вызывается Backbone (входы: latent, timestep, condition; выход: pred), затем Inner Module (входы: latent, timestep, pred; выходы: next_latent, next_timestep). Буферы для (Backbone, latent) и (Backbone, timestep) обновляются выходом Inner Module и на следующей итерации подаются в Backbone. После K итераций финальные latent снимаются с выхода Inner Module (или с выхода последнего шага) и при необходимости подаются в Converter (decode) или во внешний выход.

### 7.3 Агентный цикл (agent_loop)

- Если в гиперграфе есть **узел-агент** (блок с агентным протоколом: в выходе возможны tool_calls, на вход — tool_results), исполнитель поддерживает режим **agent_loop**:
  1. Вызвать agent.run(inputs) — на первом шаге inputs включают prompt/context с внешних входов гиперграфа.
  2. Если в выходе агента есть **tool_calls** — для каждого вызова выполнить соответствующий узел-инструмент (по таблице tool_id → node_id из конфига), собрать tool_results.
  3. Вызвать агента снова с inputs + tool_results; повторять до отсутствия tool_calls или до max_steps/stop.
  4. Выход гиперграфа — финальный **response** агента (и при необходимости другие внешние выходы).
- Связь агента с инструментами может задаваться **динамически** (по tool_id), а не только статичными гиперрёбрами; конфиг задаёт, какие узлы являются инструментами и как они сопоставляются tool_id.

### 7.4 Несколько циклов и сложная топология

- В одном гиперграфе возможны **несколько циклов** (например, вложенные или параллельные). Планировщик строит план с несколькими циклическими фазами или с явным порядком итераций. Детали (порядок внутри одной итерации, зависимости между циклами) задаются конфигом и анализом графа. Движок не жёстко привязан к одному шаблону «один цикл» — план выполнения выводится из структуры и опций.

## 8. Контракт run гиперграфа

### 8.1 Сигнатура и семантика

- **run(hypergraph, inputs, **options) → outputs**
- **Вход (inputs):** словарь — значения для внешних входов гиперграфа. Ключи — **name** (если задан при expose_input) или канонический идентификатор порта (node_id, port_name или строка по соглашению реализации). Значения — данные в формате, ожидаемом портами (тензоры, словари, строки и т.д.).
- **Выход (outputs):** словарь — значения с внешних выходов гиперграфа. Ключи — name или канонический идентификатор порта; значения — результаты с соответствующих портов после выполнения всего гиперграфа (и при наличии агента — после завершения agent_loop).

### 8.2 Опции run

| Опция | Назначение |
|-------|------------|
| **training** | bool; режим train/eval для блоков, поддерживающих его. |
| **device** | Устройство (CPU/GPU) для переноса данных и блоков при необходимости. |
| **num_loop_steps** | Число итераций цикла (например, num_inference_steps для диффузии). |
| **max_steps** | Максимум шагов agent_loop (для узла-агента). |
| **seed** | Опционально: начальное значение для стохастичности в блоках; обеспечивает воспроизводимость при наличии случайности внутри узлов. |
| **callbacks** | Опциональные колбэки (прогресс, логирование). |
| **dry_run** | Проверить план без фактического вызова run блоков. |

Конкретный набор опций задаётся реализацией; контракт гиперграфа уровня задачи не меняется от опций — опции лишь параметризуют выполнение (число шагов цикла, лимиты агента).

### 8.3 Вызов извне

- Гиперграф задачи вызывается из **воркфлоу**, **этапа** или теста именно так: передаются данные на внешние входы, снимаются данные с внешних выходов. Внутренняя топология для вызывающего не видна; гиперграф выступает как узел верхнего уровня с контрактом run(inputs) → outputs.

### 8.4 Детерминированность и воспроизводимость

- При одних и тех же **конфиге**, **чекпоинте** и **входах** порядок исполнения задан структурой графа и планом; результат run детерминирован в пределах контракта блоков (явная стохастичность — в блоках, с сериализуемым seed при необходимости). **Конфиг + чекпоинт + входы** задают воспроизводимый эксперимент ([01_FOUNDATION.md](01_FOUNDATION.md) §5.2, философия проекта).

## 9. Сериализация уровня гиперграфа

### 9.1 Конфиг структуры

- Как на фундаменте ([01_FOUNDATION.md](01_FOUNDATION.md) §9.1): **nodes** (node_id, block_type, block_id?, config?, trainable?), **edges** (source_node, source_port, target_node, target_port; при гиперрёбрах большей арности — список концов), **exposed_inputs**, **exposed_outputs** (node_id, port_name, name?). Опционально: schema_version, graph_id, graph_kind, metadata. Узлы — блоки-задачи, поэтому в config узла входят параметры конкретной реализации (архитектура, размерности и т.д.).

### 9.2 Чекпоинт (веса и состояние)

- **Чекпоинт** сохраняет state_dict каждого блока. По умолчанию ключ в чекпоинте — **node_id** (словарь node_id → state_dict). Если несколько узлов разделяют один блок (один block_id / один экземпляр в памяти, §12.6), в чекпоинте достаточно **одной** записи по block_id (или каноническому идентификатору общего блока), чтобы не дублировать веса; при загрузке один и тот же state_dict передаётся во все узлы, ссылающиеся на этот блок. Формат — один файл (словарь идентификаторов → state_dict) или директория с файлами. Загрузка — передача state_dict в каждый блок (load_state_dict); для общего блока — один load_state_dict в общий экземпляр.

### 9.3 Полное сохранение и загрузка

- **Сохранить:** записать конфиг структуры + чекпоинт (например, config.json + checkpoint.json или директория с чекпоинтами по узлам).
- **Загрузить:** прочитать конфиг, собрать гиперграф через реестр (from_config), загрузить чекпоинт в блоки. Конфиг + чекпоинт + входы задают воспроизводимое поведение.

Сериализация уровня гиперграфа **совпадает по формату** с сериализацией фундамента; отличие в том, что узлы — экземпляры узлов-задач (создаются по block_type из реестра типов задач, [02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md) §11, §13).

## 10. Типичные сценарии и примеры

### 10.1 Text-to-image (диффузия)

| Элемент | Реализация |
|---------|------------|
| **Узлы** | Converter (токенизатор текста), Conjector (CLIP-энкодер: текст → эмбеддинг), Outer Module (генератор начального шума, расписание таймстепов), Backbone (UNet/DiT), Inner Module (солвер DDIM/Euler; опционально ControlNet), Converter (VAE decode: latent → image). Опционально: Injector (LoRA), Helper (RAG для промпта). |
| **Поток** | Внешний вход (prompt) → Converter (tokenize) → Conjector (condition) → Backbone (condition). Outer Module → начальный latent и расписание → Backbone, Inner Module. Цикл: Backbone (latent, timestep, condition → pred) → Inner Module (latent, timestep, pred → next_latent, next_timestep) → Backbone (след. итерация). После K шагов: выход Inner Module (финальный latent) → Converter (decode) → внешний выход (image). |
| **Внешние входы** | prompt (и опционально num_steps, seed). |
| **Внешние выходы** | image. |

### 10.2 Простая цепочка (DAG)

- Три узла A → B → C: гиперрёбра A.out → B.in, B.out → C.in. Внешние входы — порты A; внешние выходы — порты C. План — топологический порядок [A, B, C]; один проход.

### 10.3 Агент + инструменты

- Узлы: один Backbone (агент), несколько узлов-инструментов (Helper или отдельные блоки). Внешние входы: prompt/context. Внешний выход: response. Исполнитель в режиме agent_loop вызывает агента; при tool_calls выполняет узлы-инструменты по таблице tool_id → node_id и передаёт tool_results агенту до завершения.

### 10.4 Минимальный воспроизводимый пример

- Гиперграф из 2–3 узлов-задач с потоком данных (например, Converter → Conjector → Backbone без цикла, или A → B → C). Вызов run(hypergraph, inputs), проверка outputs. Тест фиксирует конфиг и входы и проверяет совпадение выходов (или их структуры) для воспроизводимости.

## 11. Гиперграф задачи как единица развёртывания

- По канону ([CANON.md](../CANON.md)) **гиперграф задачи — единица развёртывания**: возможно **распределённое и децентрализованное** выполнение на сети эндпоинтов. Отдельные узлы (блоки-задачи) могут выполняться на разных узлах сети; гиперрёбра передают данные между эндпоинтами по согласованному протоколу. Контракт run(inputs) → outputs сохраняется; способ доставки данных (локальная память или сеть) не меняет семантики гиперграфа.
- **API-блоки:** любой блок, доступный по API (LLM, эмбеддинги, генерация и т.д.), подключается через тот же контракт портов и run; для остальной системы он остаётся узлом-задачей. Гиперграф с такими узлами может выполняться локально, на одном удалённом эндпойнте или распределённо ([philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md) §1.25, §3.9).

## 12. Граничные случаи и тонкости

### 12.1 Один узел

- Гиперграф может состоять из **одного узла**. Внешние входы и выходы — порты этого узла. Run гиперграфа сводится к вызову run блока с подачей inputs на его порты и возврату outputs. Валидация и планировщик при этом остаются применимы (план = один шаг).

### 12.2 Пустые внешние входы/выходы

- Теоретически гиперграф может не иметь объявленных внешних входов (данные подаются только по внутренним рёбрам от «источников» — узлов без входящих рёбер, например Outer Module как генератор шума). Или не иметь внешних выходов (результат только внутри графа). Для использования как чёрного ящика извне (воркфлоу) нужны хотя бы один внешний вход и один внешний выход; для вложенного или тестового вызова — не обязаны.

### 12.3 Гиперрёбра арности > 2

- Один гиперребро может связывать несколько концов (например, один выход Conjector (condition) раздаётся в несколько узлов — Backbone и Inner Module). Движок нативно поддерживает такую семантику; планировщик и буферы учитывают множественные концы при разрешении входов и записи выходов.

### 12.4 Injector без отдельного узла в графе

- Injector встроен в Backbone по конфигурации; его вызов может быть частью вызова Backbone ([02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md) §5.5). В графе гиперграфа задачи узел Injector может отсутствовать как отдельная вершина — тогда гиперрёбер от/к Injector нет; условие для инжекции может приходить на порты Backbone или обрабатываться внутри Backbone с учётом привязанного Injector.

### 12.5 Обучение на уровне гиперграфа

- Отдельные узлы (блоки) могут быть помечены как trainable; при run в режиме training=True градиенты и обновление весов обрабатываются согласно контракту блоков и выбранному оптимизатору. Детали (обратное распространение по графу, накопление градиентов по гиперрёбрам) задаются реализацией; уровень гиперграфа задачи задаёт лишь то, что структура и run едины, а обучение — вариант выполнения с сохранением контракта.

### 12.6 Повторное использование одного чекпоинта (один экземпляр в памяти)

- Если один и тот же чекпоинт (или block_id, ссылающийся на общий ресурс) используется в **нескольких узлах** гиперграфа, в памяти должен существовать **один экземпляр** блока, а все узлы ссылаются на него. Система распознаёт идентичность по идентификатору и не создаёт копии (принцип канона: [philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md)). При построении из конфига или при add_node реализация может проверять, зарегистрирован ли уже экземпляр с данным block_id/checkpoint_ref, и подставлять ссылку вместо создания нового блока.

### 12.7 Обработка ошибок и сбоев

- Философия проекта не предписывает единую политику обработки ошибок (сбой блока, таймаут, неполные данные). Как реагировать на сбой узла, ретраи, откат — задаётся **контрактом блоков** и политикой вызывающего (воркфлоу, этап, тест). Движок уровня гиперграфа не добавляет обязательной «магии» поверх этого: порты могут объявлять статус или fallback по соглашению; при необходимости обработка ошибок реализуется в блоках и в обвязке run. Уровень гиперграфа задачи фиксирует лишь то, что при run выполняются шаги плана и передача данных по буферам; детали устойчивости — в рамках контракта и реализации.

## 14. Сводные таблицы

### 14.1 Уровень гиперграфа задачи (кратко)

| Понятие | Определение |
|---------|-------------|
| **Гиперграф задачи** | Одна цельная задача (text-to-image, upscale, агент с инструментами и т.д.). Узлы — экземпляры узлов-задач (семь ролей); по гиперрёбрам — данные между портами. |
| **Первый исполняемый уровень** | По гиперграфу передаются данные и выполняется run; вход/выход = exposed_inputs, exposed_outputs. |
| **Построение** | add_node/add_hyperedge, remove_node/remove_edge, expose_input/expose_output; или from_config(config). |
| **Валидация** | Существование узлов/портов, совместимость типов, обязательные порты, опционально связность и циклы. |
| **Движок** | Структура, планировщик, буферы, исполнитель; жизненный цикл run (план → буферы → обход → выход); единая модель без доменной логики в ядре. |
| **Воспроизводимость** | Конфиг + чекпоинт + входы задают детерминированный (в пределах контракта блоков) результат run. |
| **Выше** | Воркфлоу (гиперграф гиперграфов); данные между задачами. |

### 14.2 Роли узлов в типичном гиперграфе (напоминание)

| Роль | Место в графе | Типичная связь в гиперграфе задачи |
|------|----------------|-------------------------------------|
| Backbone | Ядро | latent, timestep от Inner/Outer; condition от Conjector; pred в Inner или выход. |
| Injector | Внутри Backbone | Может не быть отдельным узлом; condition от Converter/Conjector. |
| Conjector | Рядом с ядром | input от Converter/внешний вход; condition в Backbone. |
| Inner Module | Внутри цикла | latent, timestep, pred от Backbone/пред. шаг; next_latent, next_timestep в Backbone/выход. |
| Outer Module | Рядом с циклом | output в цикл (начальное состояние, расписание); input от цикла (результат). |
| Converter | Преобразование данных | Вход/выход — с внешними входами/выходами или с другими узлами. |
| Helper | Вспомогательное | Вход/выход — с Backbone, Conjector, внешним входом и т.д. по необходимости. |

### 14.3 Ключевые инварианты уровня (для реализации и тестов)

Уровень гиперграфа задачи задаёт следующие **инварианты** — утверждения, которые должны выполняться для любого корректного гиперграфа задачи и его выполнения:

| Инвариант | Формулировка |
|-----------|--------------|
| **Один гиперграф — одна задача** | Гиперграф задачи реализует одну законченную цель; комбинирование нескольких задач — уровень воркфлоу. |
| **Узлы = экземпляры узлов-задач** | Каждый узел содержит ровно один блок; блок реализует одну из семи ролей (Backbone, Injector, Conjector, Inner Module, Outer Module, Helper, Converter). |
| **Данные только по гиперрёбрам** | Передача данных между узлами происходит только по объявленным гиперрёбрам и внешним входам/выходам; буферы заполняются из inputs или выходов узлов. |
| **План по структуре** | Порядок выполнения определяется структурой графа и опциями (num_loop_steps, max_steps), а не значениями данных на рёбрах. |
| **Контракт run** | run(hypergraph, inputs, **options) → outputs; ключи inputs/outputs заданы exposed_inputs/exposed_outputs (name или канонический идентификатор). |
| **Воспроизводимость** | При фиксированных конфиге, чекпоинте, входах и опциях (включая seed) результат run детерминирован в пределах контракта блоков. |
| **Движок без доменной логики** | Планировщик и исполнитель не содержат ветвлений по типу задачи (диффузия, агент и т.д.) в ядре; специфика — в блоках и в плане (agent_loop, итеративная фаза). |

Соблюдение этих инвариантов обеспечивает согласованность реализации с каноном и предсказуемость поведения на уровне гиперграфа задачи.

---

**Итог.** Документ задаёт **полную техническую спецификацию уровня гиперграфа задачи**: структура (узлы = экземпляры узлов-задач, гиперрёбра, внешние входы/выходы), построение (включая удаление узлов/рёбер) и валидация, гиперграфовый движок (планирование, буферы, исполнитель, жизненный цикл run, циклы и agent_loop), контракт run и воспроизводимость, сериализация, типичные сценарии, единица развёртывания и граничные случаи (один узел, пустые входы/выходы, гиперрёбра арности > 2, Injector без отдельного узла, обучение, повторное использование одного чекпоинта, обработка ошибок). Реализация и уровень воркфлоу опираются на эту спецификацию.
