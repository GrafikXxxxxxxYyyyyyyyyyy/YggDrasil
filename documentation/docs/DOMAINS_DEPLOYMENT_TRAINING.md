# Поддержка доменов, развёртывание и обучение

**Назначение:** единый документ по **поддержке ключевых доменов** (агентные системы, диффузионные модели, языковые модели), **развёртыванию на нескольких эндпоинтах** (Multi Endpoint Deployment), **обучению на каждом уровне** и **сценариям использования API**. Задаёт, как фреймворк обеспечивает равноправную поддержку этих сценариев в рамках одного канона и контрактов.

**Связь с уровнями:** [01_FOUNDATION.md](01_FOUNDATION.md), [02_ABSTRACT_TASK_NODES.md](02_ABSTRACT_TASK_NODES.md), [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md), [04_WORKFLOW.md](04_WORKFLOW.md), [05_STAGE.md](05_STAGE.md), [06_WORLD.md](06_WORLD.md), [07_UNIVERSE.md](07_UNIVERSE.md), [HYPERGRAPH_ENGINE.md](HYPERGRAPH_ENGINE.md).

**Язык:** русский.

## 1. Поддержка доменов: общий принцип

Фреймворк Иггдрасиль обеспечивает **равноправную и глубоко встроенную поддержку** нескольких доменов в рамках **одной архитектуры** и **одних контрактов**. Пользователь может использовать систему:

- **только** для агентных систем (агент + инструменты, agent_loop);
- **только** для диффузионных моделей (text-to-image, upscale, image-to-video и т.д.);
- **только** для языковых моделей (инференс, генерация, RAG);
- **только** для мира (цикл стадий, state, контент);
- или в **любой комбинации** (языковая модель + диффузия в воркфлоу, агентные стадии в мире, миры с диффузией в стадии творца и т.д.).

Поддержка не является надстройкой: контракты блока, гиперграфа, воркфлоу, стадии, мира и вселенной **допускают** эти домены как равноправные. Движок гиперграфа не различает «тип задачи» в ядре — только структуру, порты и run ([HYPERGRAPH_ENGINE.md](HYPERGRAPH_ENGINE.md)).

## 2. Поддержка агентных систем

### 2.1 Что такое агентная система в каноне

**Агент** — блок (узел-задача), который: (1) хранит **внутреннее состояние** между вызовами (история диалога, память, план); (2) при вызове run может возвращать не только ответ, но и **вызовы инструментов** (tool_calls): идентификатор инструмента и аргументы; (3) принимает на следующем шаге **результаты инструментов** (tool_results) и продолжает рассуждение; (4) завершает работу, когда выдаёт итоговый выход без tool_calls или по условию (max_steps, stop). **Инструменты (tools)** — блоки или целые гиперграфы, вызываемые агентом по tool_id; контракт: именованные входы (аргументы вызова) и выход (результат).

**Агентный цикл (agent_loop):** движок гиперграфа при выполнении узла-агента входит в подцикл: вызов агента → при наличии tool_calls выполнение соответствующих узлов-инструментов (по таблице tool_id → node_id из конфига) → сбор tool_results → повторный вызов агента с inputs + tool_results; повтор до отсутствия tool_calls или до max_steps. Подробно: [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §7.3, [HYPERGRAPH_ENGINE.md](HYPERGRAPH_ENGINE.md) §2, §6.

**Подробно:** [AGENT_SYSTEMS.md](AGENT_SYSTEMS.md) — полное описание агентных систем в рамках фреймворка (паритет с n8n, LangChain, LangGraph, DeepAgents; маппинг на роли, типичные графы, форматы данных, цикл выполнения, конфигурация, сериализация).

### 2.2 На каких уровнях поддерживаются агенты

| Уровень | Поддержка |
|---------|-----------|
| **Гиперграф задачи** | Узел-агент (Backbone или выделенная роль) + узлы-инструменты; выполнение = agent_loop. Вход гиперграфа — prompt/context; выход — финальный response агента. |
| **Воркфлоу** | Узел воркфлоу может быть **гиперграфом-агентом** (внутри — агент и инструменты); вызов такого узла = run гиперграфа с agent_loop. Инструментом может быть отдельный гиперграф-узел воркфлоу при соответствующей архитектуре (динамический вызов по tool_id). |
| **Стадия** | Стадия может состоять из воркфлоу, в котором центральный узел — агент (например, стадия автора или стадия среды). state_input_map подаёт слоты state на входы воркфлоу (в т.ч. контекст для агента); state_output_map записывает выход воркфлоу в state. |
| **Мир** | Стадии мира (автор, среда, творец) могут быть реализованы как агентные воркфлоу. В частности, **развитие мира (Development)** в стадии среды может быть агентом с инструментами (чтение мира, запрос к хранилищу, вызов LLM). **Action** пользователя передаётся агенту как часть контекста. |
| **Вселенная** | **Сущности (entities)** с состоянием (персонажи, агенты) передаются между мирами по payload_spec; переход сущности из мира A в мир B и обратное влияние на мир A реализуют кросс-мировое взаимодействие, в т.ч. агентное. [07_UNIVERSE.md](07_UNIVERSE.md) §4.2, §10.2. |

Агентные системы применимы на **любом разумном этапе**: от одного гиперграфа с агентом и инструментами до мира с агентными стадиями и вселенной с сущностями-агентами между мирами. Сериализация состояния агента входит в state_dict блока и восстанавливается при загрузке чекпоинта ([SERIALIZATION.md](SERIALIZATION.md)).

## 3. Поддержка диффузионных моделей

**Подробно:** [DIFFUSION_MODELS.md](DIFFUSION_MODELS.md) — полное описание использования фреймворка для диффузионных моделей (маппинг на роли, типичные графы, форматы данных, цикл деноизинга, конфигурация, обучение, сериализация, воркфлоу).

### 3.1 Типичная схема в гиперграфе задачи

Диффузионный пайплайн (text-to-image, image-to-image, upscale и т.д.) реализуется **одним гиперграфом задачи** ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §2.2, §10.1). Узлы: Converter (токенизатор текста), Conjector (CLIP-энкодер: текст → эмбеддинг), Outer Module (генератор шума, расписание таймстепов), Backbone (UNet/DiT), Inner Module (солвер DDIM/Euler; опционально ControlNet), Converter (VAE decode). Поток: внешний вход (prompt) → условие → Backbone + Inner Module в **цикле** (K шагов деноизинга) → decode → внешний выход (image). Движок строит итеративный план для цикла Backbone ↔ Inner Module; число шагов K задаётся конфигом или опцией run (num_loop_steps).

### 3.2 На каких уровнях используются диффузионные модели

| Уровень | Использование |
|---------|----------------|
| **Гиперграф задачи** | Один гиперграф = одна диффузионная задача (text-to-image, upscale, inpainting и т.д.). Полная цепочка encode → цикл деноизинга → decode внутри одного графа. |
| **Воркфлоу** | Несколько диффузионных (и не только) задач в цепочке или ветвлении: например, text-to-image → upscale → image-to-video. Узлы воркфлоу = гиперграфы; по рёбрам передаются изображения, латенты и т.д. |
| **Стадия** | Стадия творца в мире часто реализуется воркфлоу с диффузионным гиперграфом: по слотам state (конфигурация генерации) вызывается генерация артефактов (изображения, видео). Стадия автора может использовать распознавание/описание (не обязательно диффузию). |
| **Мир** | Цикл автор → среда → творец: стадия творца генерирует артефакты (диффузия, кодек); стадия автора описывает артефакты; среда обновляет мир и порождает новый state. Диффузия — типичная реализация «идея → материя» на стадии творца. |

Диффузионные модели поддерживаются **без отдельной онтологии**: те же блоки (Backbone, Inner Module, Conjector, Converter), тот же движок, те же контракты run и сериализации. Обучение (LoRA, full fine-tune, ControlNet) — по правилам обучения на уровнях (§6).

## 4. Поддержка языковых моделей (LLM)

**Подробно:** [LANGUAGE_MODELS.md](LANGUAGE_MODELS.md) — полное описание использования фреймворка для языковых моделей (маппинг на роли, типичные графы, форматы данных, цикл генерации, конфигурация, обучение, сериализация, воркфлоу).

### 4.1 Роли LLM в архитектуре

Языковые модели в каноне выступают как **блоки-задачи** (чаще всего Backbone или узлы с выделенной ролью): вход — prompt/context (и опционально tool_results для агента), выход — текст (и опционально tool_calls). LLM может быть узлом в гиперграфе задачи (одиночная генерация, RAG-цепочка), ядром агента (агент = LLM + логика tool calling), частью стадии (автор, среда, развитие мира). Контракт портов единый независимо от того, выполняется ли блок **локально** (свои веса) или **через API** (§5).

### 4.2 Сценарии использования LLM

| Сценарий | Описание |
|----------|----------|
| **Один LLM в гиперграфе** | Гиперграф из одного или нескольких узлов: например, Converter (токенизатор) → Backbone (LLM) → вывод. Внешние входы — prompt; выход — сгенерированный текст. |
| **RAG** | Helper (поиск по корпусу) → контекст в Conjector или на вход Backbone → Backbone (LLM) → ответ. [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §2.2. |
| **Агент на базе LLM** | Узел-агент с LLM внутри (локально или по API); инструменты — другие узлы гиперграфа. Agent_loop выполняется движком. |
| **Стадии мира** | Стадия автора, стадия среды (Development), опционально другие стадии — могут быть реализованы воркфлоу с LLM (описание артефактов, генерация следующего шага мира, диалог с пользователем). |

LLM поддерживаются на **всех уровнях**, где уместен блок с контрактом «вход текст/контекст → выход текст»: гиперграф задачи, воркфлоу, стадия (через воркфлоу), мир (через стадии). Выбор «локальная модель vs API» задаётся конфигом блока и не меняет контракта графа/воркфлоу/стадии (§5).

## 5. Использование API и сценарии с API

### 5.1 Принцип: единый контракт при локальном и API-режиме

**Любой блок**, для которого существует внешний API (LLM, VLM, эмбеддинги, генерация изображений по API и т.д.), **должен иметь возможность работать через API провайдера** без загрузки весов в память. Конфиг блока задаёт режим: локальный (checkpoint_ref, веса в памяти) или API (backend: "api", provider, model_id, endpoint_url при необходимости). **Контракт портов и run** для вызывающего (гиперграф, воркфлоу, стадия) **одинаков**: входы и выходы по именам и типам; способ выполнения (локальный вызов или HTTP-запрос к API) скрыт внутри блока.

### 5.2 Требования к блоку в режиме API

- **Инференс без локальных весов:** при run блок выполняет запрос к API провайдера и возвращает ответ; веса не загружаются. Учётные данные (api_key и т.д.) задаются через переменные окружения или защищённое хранилище, не хранятся в конфиге репозитория.
- **Сериализация:** в конфиг графа/воркфлоу попадают только идентификаторы (provider, model_id, endpoint_url); секреты при инициализации блока подставляются из окружения или хранилища.
- **Реестр:** блок в режиме API регистрируется под тем же или совместимым типом узла-задачи (например, llm/openai, llm/api), чтобы граф собирался из конфига без ветвлений «если API — иначе локально».

### 5.3 Сценарии с API

| Сценарий | Описание |
|----------|----------|
| **Гиперграф с LLM по API** | Узел Backbone (LLM) сконфигурирован как API; остальные узлы (Converter, Conjector и т.д.) локальные. run(hypergraph, inputs) выполняет инференс; вызов LLM — внутри блока через API. |
| **Агент с LLM по API и локальными инструментами** | Узел-агент использует LLM по API; инструменты — локальные блоки или гиперграфы. Agent_loop без изменений: движок вызывает агента и инструменты по плану. |
| **Воркфлоу: один узел по API** | Один из гиперграфов-узлов воркфлоу развёрнут как сервис по URL (или внутри него блок по API); оркестратор вызывает run(hypergraph, …) локально или по сети; контракт входа/выхода единый. |
| **Мир с API-стадиями** | Стадии мира (автор, среда, творец) могут использовать блоки (LLM, генерация) по API; state и цикл мира не зависят от того, локальные это блоки или API. |

Обучение при использовании API: блоки только по API обычно не обучают локально (нет весов); обучаемые части мира/воркфлоу — те узлы, у которых есть локальные веса (LoRA, адаптеры, свои модели). Сценарии обучения с повторным использованием и API (один блок в нескольких узлах, LoRA при API) задаются реализацией и конфигом.

## 6. Развёртывание на нескольких эндпоинтах (Multi Endpoint Deployment)

### 6.1 Цель

Система должна позволять **запуск на нескольких эндпоинтах** с **сохранением корректной работы**: порядок выполнения, контракты, поток state и семантика цикла не меняются. Идеально — **каждый гиперграф (задачи, воркфлоу) может быть развёрнут на своём эндпоинте**; оркестрация вызовов между эндпоинтами заменяет локальные вызовы без изменения поведения.

### 6.2 Принцип: гиперграф как единица развёртывания

- **Гиперграф задачи** — естественная единица развёртывания: сериализуем, имеет явные внешние входы/выходы, один вызов = один run. При развёртывании на эндпоинте гиперграф экспонирует API (например, POST с входными данными → ответ с выходными); контракт совпадает с контрактом run(hypergraph, inputs) → outputs.
- **Воркфлоу:** узлы воркфлоу = гиперграфы; каждый такой гиперграф может быть **сервисом на своём URL**. Оркестратор воркфлоу для каждого узла знает: локальный граф или endpoint_url; при URL выполняет запрос к эндпоинту, передаёт выход предыдущего узла на вход следующего. Сериализация данных между эндпоинтами (JSON, бинарные артефакты по ссылкам или base64) по соглашению реализации.
- **Стадия и мир:** стадия — гиперграф воркфлоу; воркфлоу-узлы могут быть удалёнными. Между стадиями передаётся state (сериализуемый объект), поэтому state можно передавать между эндпоинтами. Мир при распределённом запуске сохраняет цикл автор → среда → творец и правила выполнения стадий (can_run, World update при полном state, Development).

### 6.3 Следствия для реализации

- **Реестр эндпоинтов:** в конфиге воркфлоу/стадии/мира для каждого узла-гиперграфа задаётся либо локальная реализация, либо endpoint_url. Загрузка и запуск сервиса на этом URL — задача развёртывания (контейнер, serverless и т.д.).
- **Эквивалентность:** поведение при распределённом развёртывании (каждый граф на своём эндпоинте) **эквивалентно** поведению при запуске всего в одном процессе — без изменения семантики и контрактов. Движок и оркестратор не содержат доменной логики; только вызов run локально или по сети.

## 7. Обучение на каждом уровне

### 7.1 Принцип

На **каждом уровне** всё, что **может быть обучено** (блоки с весами, адаптеры, LoRA и т.д.), **должно поддерживать обучение**. Любой блок с обучаемыми параметрами доступен для тренировки на своём уровне (отдельно или в составе графа/воркфлоу/стадии/мира). Конфиг задаёт, какие узлы или графы помечены как trainable; остальные могут быть заморожены.

### 7.2 Обучение по уровням

| Уровень | Что обучается | Как |
|---------|----------------|-----|
| **Блок** | Параметры блока (state_dict) | trainable_parameters(), loss по входам/выходам портов, оптимизатор, сохранение чекпоинта блока. |
| **Узел** | Блок внутри узла | То же, что блок; узел идентифицирует блок для оптимизатора. |
| **Гиперграф задачи** | Подмножество узлов (один, несколько или все) | Forward по графу, loss на выходах графа или выбранных узлов, backward по trainable узлам; заморозка остальных. LoRA, ControlNet, Backbone — узлы графа; каждый может быть trainable/frozen по конфигу. [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §12.5. |
| **Воркфлоу** | Подмножество гиперграфов-узлов (или узлов внутри них) | trainable_parameters() / set_trainable() на уровне воркфлоу; опция training пробрасывается в run гиперграфов. Чекпоинт воркфлоу — агрегат чекпоинтов trainable гиперграфов. [04_WORKFLOW.md](04_WORKFLOW.md) §10, §11.8. |
| **Стадия** | Подмножество воркфлоу-узлов | Аналогично: trainable воркфлоу в составе стадии; при run(stage, state, training=True) опция пробрасывается в воркфлоу. |
| **Мир** | Подмножество стадий или блоков внутри стадий | При run(world, …, training=True) опция пробрасывается в стадии и вложенные воркфлоу/гиперграфы. Чекпоинт мира включает веса всех trainable сущностей. [06_WORLD.md](06_WORLD.md). |
| **Вселенная** | Развивающиеся миры | Отдельные миры (и вложенные стадии/воркфлоу) могут быть помечены trainable; при run(universe, …, training=True) опция пробрасывается в соответствующие миры. Чекпоинт вселенной — агрегат чекпоинтов развивающихся миров. [07_UNIVERSE.md](07_UNIVERSE.md) §14.1. |

На каждом уровне: **выбор trainable сущностей**, **конфиг обучения** (lr, schedule, заморозка), **loss** (по портам/выходам), **чекпоинты** (сериализуемое состояние). Дедупликация: если один и тот же блок используется в нескольких узлах (общий checkpoint_ref), параметры в оптимизатор передаются **один раз**; градиенты по всем использованиям суммируются (или по соглашению — обучение только по одному месту). Подробнее — в конфиге обучения и соглашениях реализации.

### 7.3 Примеры: LoRA, ControlNet, полное дообучение

- **LoRA (и другие адаптеры):** блок-адаптер в графе; обучаются только параметры адаптера при замороженном Backbone (или оба по конфигу). Уровень: блок как узел графа; граф остаётся исполняемым.
- **ControlNet:** узел графа с обучаемыми весами; обучение отдельно или совместно с Backbone; уровень — гиперграф (подмножество узлов trainable).
- **Backbone:** полное или частичное дообучение Backbone как узла графа; уровень — блок/узел/граф.

Все случаи применимы на **любом уровне**: один блок, один гиперграф, гиперграф внутри воркфлоу/стадии/мира. Система не запрещает обучение ни одного типа блока; конфиг задаёт, что trainable, что frozen.

## 8. Поддерживаемые парадигмы и расширяемость

Фреймворк не ограничивается только агентами, диффузией и LLM. **Любая парадигма**, реализуемая в виде блоков с контрактом портов и run, допускается. Ниже — примеры парадигм, которые вписываются в ту же онтологию.

### 8.1 NeRF и 3D/сценовые модели

Модели нейрального рендеринга (NeRF и аналоги), 3D-реконструкция, генерация сцен — реализуются как **блоки** с входами (изображения, камеры, параметры) и выходами (рендер, меш, эмбеддинги сцены). Такой блок может быть узлом гиперграфа задачи (например, Conjector или Backbone в цепочке «изображение → NeRF → вид с новой камеры»), узлом воркфлоу (целый пайплайн 3D) или частью стадии (например, стадия творца, генерирующая 3D-артефакты). Контракт единый; мультимодальность (изображение, 3D, камера) задаётся типами портов ([01_FOUNDATION.md](01_FOUNDATION.md) §2.4; философия Part3 §1.24).

### 8.2 Графовые нейросети (GNN) и графы знаний

Блоки, работающие с графами (GNN, графы знаний, reasoning over knowledge graph), входят в систему как узлы-задачи: вход — граф или эмбеддинги узлов/рёбер, выход — обновлённые эмбеддинги, предсказания, подграфы. Могут выступать в ролях Helper (поиск по графу), Backbone (модель рассуждений) или Converter (преобразование формата). Гиперграф задачи может объединять GNN с LLM (RAG по графу + генерация текста) или с диффузией (условие по графу → изображение). Онтология не меняется: граф — один из типов данных на портах.

### 8.3 Цепочки типа LangChain / композитные агенты

Сценарии в духе LangChain (цепочки промптов, агенты с инструментами, ветвление по условию) реализуются **внутри фреймворка** как комбинация блоков и гиперграфа: узлы — LLM, инструменты, конвертеры; рёбра — передача контекста и ответов; при необходимости — узел-агент с agent_loop. Фреймворк не предписывает «свой» формат цепочек — любой граф, собранный из зарегистрированных блоков с совместимыми портами, является законным. Импорт конфигураций из внешних форматов (если реализация поддерживает) не меняет контракта: после импорта граф живёт по правилам гиперграфа задачи или воркфлоу (философия Part3 §1.31, интероперабельность).

### 8.4 Итог по парадигмам

Поддержка доменов **расширяема**: новые парадигмы добавляются как новые типы блоков и узлов-задач в реестре, без изменения движка и онтологии. Агенты, диффузия, LLM, NeRF, GNN, кастомные пайплайны — равноправны в рамках единого контракта и гиперграфового движка.

---

**Итог.** Документ задаёт **поддержку доменов** (агентные системы, диффузионные модели, языковые модели) на всех уровнях в рамках единого канона; **сценарии использования API** (единый контракт при локальном и API-режиме); **развёртывание на нескольких эндпоинтах** (гиперграф как единица развёртывания, эквивалентность поведения); **обучение на каждом уровне** (trainable/frozen по конфигу, дедупликация параметров, чекпоинты). Реализация опирается на контракты уровней и движок гиперграфа без доменной логики в ядре.
