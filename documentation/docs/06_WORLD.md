# 06. Уровень мира

**Назначение:** полная **техническая** документация **четвёртого исполняемого уровня** системы Иггдрасиль — мира. Документ задаёт мир как **гиперграф стадий**: узлы мира — **целые стадии** (автор, среда, творец), по гиперрёбрам мира передаётся **state** (контейнер слотов). Описаны **цикл мира** (порядок: автор → среда → творец), **state** и схема state, условия выполнения стадий (can_run), **первая итерация** без предзаданного state, **action** как опциональный контекст, **контент мира** и его обновление, контракт **run(world, state, action?, context?)**, сериализация в **два поля** (исполняемая часть и контент), **развивающийся** и **завершённый (End World)** мир, граничные случаи. Канонический источник технических решений уровня мира.

**Предыдущий уровень:** [05_STAGE.md](05_STAGE.md) — стадия (гиперграф воркфлоу, state на входах/выходах).

**Философский якорь:** [../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §6 — уровень мира как новая реализация идеи стремления; материя = state, идея = action; мир — непрерывная цепь переходов (гиперграф стадий); порядок фиксирован, граф цикличен; мир может развиваться сам; первая итерация без предзаданного state; развивающийся и завершённый мир. [../../philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md) — сериализация мира: исполняемая часть и контент; End World — только контент.

**Референс:** [../../reference/Universe_Generator_3.0/docs/05_WORLD_LEVEL.md](../../reference/Universe_Generator_3.0/docs/05_WORLD_LEVEL.md), [../../reference/WorldGenerator_2.0/World_Level.md](../../reference/WorldGenerator_2.0/World_Level.md), [../../reference/WorldGenerator_2.0/World_Serialization.md](../../reference/WorldGenerator_2.0/World_Serialization.md), [../../reference/WorldGenerator_2.0/END_WORLDS.md](../../reference/WorldGenerator_2.0/END_WORLDS.md).

## 1. Место уровня в иерархии

| Уровень | Сущность | Что течёт по связям | Описание |
|---------|----------|---------------------|----------|
| **Стадия** | Гиперграф, узлы которого — воркфлоу | Внутри стадии: **данные**; вход/выход стадии: **state** | Одна стадия = одна метаморфоза state. [05_STAGE.md](05_STAGE.md). |
| **Мир** | **Мир** — гиперграф, узлы которого — **стадии** | **State** (контейнер слотов) | **Четвёртый исполняемый уровень.** Непрерывная цепь стадий: автор → среда → творец; по рёбрам передаётся state; граф **цикличен**. |
| **Вселенная** | Гиперграф миров (узлы = миры) | Обмен (payload, state snapshot, сущности) | Узлы = миры; общая среда вселенной. Документ уровня вселенной — по мере заполнения канона. |

Ключевое отличие мира от стадии: **стадия** получает state и возвращает state после **одной** метаморфозы; **мир** — это **цикл стадий**, в котором state последовательно проходит стадию автора, стадию среды, стадию творца и снова автора. По гиперрёбрам мира передаётся **state**; каждая стадия выполняет свою метаморфозу (материя → идея, идея → материя, переход). Мир может **развиваться сам**, итеративно; **action** пользователя опционален ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §6).

## 2. Мир — определение и смысл

### 2.1 Определение

**Мир (World)** — это **гиперграф, узлами которого являются стадии**. В каноне мир — **циклическая цепочка из трёх стадий**: стадия автора, стадия среды, стадия творца. По **гиперрёбрам мира** передаётся **state**: выход одной стадии (обновлённый state) становится входом следующей. Порядок обхода **фиксирован**; после стадии творца state снова попадает к стадии автора (цикл замыкается через стадию среды: обновление мира и порождение нового state).

- **Узлы мира** — **стадии** (каждая реализуется как гиперграф воркфлоу по [05_STAGE.md](05_STAGE.md)). У каждой стадии есть **роль** в цикле: автор, среда, творец. Контракт стадии: run(stage, state, context?) → state; условие can_run(stage, state, context?).
- **Гиперрёбра мира** — связи «выход стадии N → вход стадии N+1»; по ним передаётся **state**. В каноническом цикле из трёх стадий: автор → среда → творец → автор → …
- **Выполнение мира** — последовательный вызов **run(stage, state, context)** для каждой стадии в порядке цикла; после каждой стадии state обновляется и передаётся следующей. Стадия среды может включать **обновление контента мира** (при полном state) и **порождение нового state** (развитие мира) для следующего витка цикла.

Таким образом, **мир = гиперграф стадий** (цепочка стадий с замыканием в цикл); структура мира — порядок стадий и их конфигурация; state течёт по рёбрам, на каждой стадии претерпевая метаморфозу; цикл замыкается через стадию среды.

### 2.2 Граница со стадией

**Стадия** — атомарная единица метаморфозы: получает state, при выполнении условия can_run вызывает свои воркфлоу, возвращает обновлённый state. **Мир** оркестрирует **последовательность стадий**: передаёт state из стадии в стадию, проверяет условия выполнения, при необходимости передаёт **action** в стадию среды, сохраняет state и контент мира по правилам уровня. Мир не изменяет контракт стадии — только порядок вызова и передачу state и context ([05_STAGE.md](05_STAGE.md) §8).

### 2.3 Три стадии в цикле мира (философский контекст)

По философии проекта ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §5, §6) в цикле мира участвуют **три стадии**:

| Стадия | Роль в цикле |
|--------|---------------|
| **Стадия автора** | Воплощение **идеального** начала: материальные артефакты → идея (описание того, что произошло в мире). |
| **Стадия среды** | **Переход** — живая связь материального и идеального: принимает описания от автора, помещает их в историю мира; **обновляет контент мира** при полном state; **порождает новый state** (развитие мира) для следующего витка. |
| **Стадия творца** | Воплощение **материального** начала: идея (намерение, конфигурация) → артефакты (видео, звук, изображения). |

Порядок исполнения в мире фиксирован; гиперграф **цикличен**. Семантика слотов state (намерение, конфигурация, артефакты, описания, нарратив и т.д.) и условия выполнения стадий задаются на уровне мира; каждая стадия предоставляет контракт run(stage, state, context) → state и can_run(stage, state, context).

## 3. State и схема state на уровне мира

### 3.1 State как поток между стадиями

**State** (состояние мира) — **контейнер с именованными слотами**. Он передаётся по гиперрёбрам мира от стадии к стадии и претерпевает метаморфозы: каждая стадия читает одни слоты и дополняет или перезаписывает другие. Определение state и его использование внутри стадии заданы в [05_STAGE.md](05_STAGE.md) §3; на уровне мира фиксируются **схема state** (какие слоты существуют, их семантика в цикле) и **поток state** по стадиям.

- **Представление state:** словарь (slot_name → value), объект с полями или индексируемая структура. Конкретный формат задаётся реализацией.
- **Использование в мире:** при run(world, state, …) мир передаёт state первой стадии в порядке цикла (или точке входа при первой итерации); после каждой стадии получает обновлённый state и передаёт его следующей стадии; после последней стадии в цикле state переходит к первой (следующий виток) или возвращается как результат run в зависимости от контракта.

### 3.2 Схема state (state_schema)

**Схема state** — описание структуры state, задаваемое конфигом мира: перечень слотов (имена или индексы), опционально типы или семантика. Схема используется для:

- создания **пустого state** при старте (все слоты пусты или с значениями по умолчанию);
- **валидации** при сохранении state (World update) и при передаче между стадиями;
- проверки **условий выполнения** стадий (can_run ссылается на заполненность слотов).

Семантика слотов в каноне (намерение, конфигурация, артефакты, описания, нарратив и т.д.) задаётся уровнем мира или соглашением; стадии оперируют state через state_input_map и state_output_map, согласованные с этой схемой.

### 3.3 Условия выполнения стадий в цикле мира

Каждая стадия объявляет условие **can_run(stage, state, context)** ([05_STAGE.md](05_STAGE.md) §3.4). Мир при обходе цикла для каждой стадии проверяет это условие; если оно **не выполнено**, стадия **пропускается** — state передаётся следующей стадии без изменений. Типичные условия — заполненность определённых слотов state (required_slots) или предикат от state и context. Конкретные правила (при каком state какая стадия выполняется или пропускается) задаются конфигом стадий и конфигом мира; уровень мира обеспечивает механизм проверки и пропуска.

**Стадия среды** в каноне может включать два логических подшага: (1) **обновление мира** (World update) — применение state к контенту мира, сохранение state — выполняется только при **полном** state (все нужные слоты заполнены); (2) **развитие мира** (Development) — порождение **нового state** для следующего витка (например, слот намерения) — может выполняться **безусловно** или по своему правилу. Детали реализуются внутри стадии среды как один или несколько воркфлоу; мир передаёт стадии среды state и опционально action и контекст (контент мира, хранилище для сохранения state).

## 4. Состав мира: узлы, гиперрёбра, цикл

### 4.1 Узлы мира = стадии

- Каждый **узел мира** — целая **стадия** ([05_STAGE.md](05_STAGE.md)). У каждого узла есть идентификатор в рамках мира (**stage_id**: например, `author`, `environment`, `creator`).
- Стадия может быть задана вложенным конфигом (по спецификации 05) или ссылкой (ref на конфиг стадии). При загрузке мира каждая стадия-узел строится из своего конфига.
- **Порядок обхода** задаётся **циклом** — упорядоченным списком stage_id. В каноне цикл из трёх стадий: автор → среда → творец; после творца переход снова к автору.

### 4.2 Гиперрёбра мира и цикл

- **Гиперрёбра мира** задают передачу **state** от стадии к стадии. В канонической реализации рёбра не хранятся явно как список пар: **порядок цикла** однозначно определяет, куда передаётся state после каждой стадии (следующая в списке; после последней — первая).
- **Цикл** — список stage_id в порядке выполнения. Один проход по циклу — последовательный вызов run(stage, state, context) для каждой стадии в этом порядке; state после стадии i передаётся стадии i+1 (после последней — первой).

### 4.3 Контракт мира по state и контенту

| Элемент | Описание |
|---------|----------|
| **Цикл (cycle)** | Упорядоченный список stage_id. Задаёт порядок обхода стадий и замыкание цикла. |
| **state_schema** | Описание слотов state для создания пустого state и валидации. |
| **Контент мира** | Наполнение мира (история, глоссарий, персонажи, правила, артефакты и т.д.); читается и обновляется стадиями (в частности, стадией среды при World update). |
| **Хранилище state (storage)** | Конфигурация сохранения state при обновлении мира (путь, URI, политика версионирования); используется стадией среды при World update. |

## 5. Построение мира

### 5.1 Добавление стадий и задание цикла

- **add_stage(stage_id, stage)** — добавить стадию как узел мира с заданным stage_id. Стадия уже собрана и имеет контракт по state (state_input_map, state_output_map, condition).
- **add_stage(stage_id, config, registry?)** — построить стадию из конфига (05.from_config) и добавить как узел. Конфиг — полный конфиг стадии или путь к файлу. Дублирование stage_id при добавлении должно приводить к ошибке.
- **set_cycle(ordered_stage_ids)** — задать порядок обхода: список stage_id в порядке выполнения. Каждый stage_id должен присутствовать в узлах мира; после последнего в списке переход к первому (цикл замыкается).
- **remove_stage(stage_id)** — удалить стадию из мира; инвалидирует цикл (цикл должен быть обновлён так, чтобы не ссылаться на удалённую стадию).

### 5.2 Схема state и хранилище

- **set_state_schema(schema)** — задать описание state (перечень слотов, опционально типы или семантика). Используется для создания пустого state и валидации.
- **set_storage(config)** — задать конфигурацию хранилища для сохранения state при World update (директория, URI, шаблон имён, политика версионирования). Передаётся в контексте стадии среды при run.
- **set_initial_content(content)** или **set_content(content)** — задать начальный контент мира (история, глоссарий, правила и т.д.), доступный стадиям при выполнении (в частности, стадии среды при Development).

### 5.3 Построение из конфига

- **Конфиг мира:** **cycle** — список stage_id в порядке обхода; **stages** — словарь stage_id → конфиг стадии (по 05 §9.1 или ref); **state_schema**; **storage** (опционально); **content** или **initial_content** (опционально); world_id, schema_version, metadata.
- **from_config(config, registry?, validate?)** — для каждого stage_id из cycle построить стадию из конфига (05.from_config), собрать мир; установить цикл, state_schema, storage, контент. При validate — валидация мира (§6).

## 6. Валидация мира

Перед выполнением или сериализацией выполняется **валидация** структуры мира.

### 6.1 Обязательные проверки

| Проверка | Описание |
|----------|----------|
| **Цикл и узлы** | Каждый stage_id из cycle присутствует в множестве узлов мира; в cycle нет дубликатов; длина cycle равна числу узлов (для канонического цикла). |
| **Контракты стадий** | Для каждой стадии ключи в state_input_map и state_output_map ссылаются на слоты из state_schema (если схема задана). |
| **Условия выполнения** | Если condition стадии задан как required_slots, указанные слоты входят в state_schema. |
| **Хранилище** | Если задан storage, конфигурация валидна (путь или URI корректен; при проверке прав — возможность записи). |

### 6.2 Дополнительные проверки

| Проверка | Описание |
|----------|----------|
| **Совместимость потока state** | Выходные слоты стадии i (по state_output_map) совместимы с входными ожиданиями стадии i+1 (по state_input_map) в порядке цикла. |
| **Валидность стадий** | Опционально вызывать валидацию каждой стадии по правилам 05 §6. |

### 6.3 Результат валидации

- Валидация возвращает список **ошибок** и **предупреждений**. В строгом режиме run не выполняется при наличии ошибок; при from_config(validate=True) невалидный мир не создаётся или выдаёт исключение.

## 7. Исполнитель мира и жизненный цикл run

Мир выполняет **обход стадий в порядке цикла**. Для каждой стадии: проверка can_run(stage, state, context); при истине — run(stage, state, context) → state; при лжи — state без изменений к следующей стадии. После последней стадии в цикле state переходит к первой стадии (следующий виток) или возвращается как результат в зависимости от контракта run мира.

### 7.1 Компоненты

| Компонент | Назначение |
|-----------|------------|
| **Хранилище структуры** | Множество узлов (stage_id → стадия), цикл (список stage_id), state_schema, storage, контент мира, world_id, metadata. |
| **Валидатор** | Проверка согласованности мира (§6). |
| **Исполнитель** | Обход стадий по циклу; для каждой стадии — проверка can_run, вызов run(stage, state, context); передача state следующей стадии; при необходимости передача action и контента мира в context для стадии среды. |

### 7.2 Жизненный цикл одного прохода по циклу (один виток)

1. **Вход:** world, state (или пустой/отсутствующий при первой итерации), action? (опционально), context? (опции run, ссылка на мир и т.д.).
2. **Инициализация:** если state не передан или пуст, по правилам **первой итерации** (§8.2) определить начальную точку обхода и при необходимости получить первый state (например, от стадии среды — Development).
3. **Обход:** для каждой стадии в порядке cycle: (a) проверить can_run(stage, state, context); (b) если ложно — оставить state без изменений, перейти к следующей стадии; (c) если истинно — сформировать context (в т.ч. action для стадии среды, контент мира, storage); вызвать **run(stage, state, context)** → state; при необходимости сохранить state в хранилище (если стадия среды выполнила World update); передать state следующей стадии в цикле.
4. **Замыкание цикла:** после последней стадии в cycle state переходит к первой стадии — либо начинается следующий виток (если run мира выполняет несколько витков), либо state возвращается как результат одного витка.

## 8. Контракт run мира

### 8.1 Сигнатура и семантика

- **run(world, state?, action?, context?, **options) → state (или результат витка)**
- **Вход (state):** контейнер слотов. Может быть передан пустым или отсутствовать — тогда применяется логика первой итерации (§8.2).
- **Вход (action):** опционально — действие пользователя (реплика, команда, выбор). Передаётся в **context** при вызове стадии среды; стадия среды может использовать action при порождении нового state (Development). Мир может развиваться **без** action — пользователь не обязательная причина каждого шага ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §6).
- **Вход (context):** опционально — ссылка на мир, контент мира, storage, опции run. Передаётся стадиям при run(stage, state, context).
- **Выход:** state после одного витка цикла (или после N витков, или итератор по state после каждого витка — по соглашению реализации). При одном витке — state после прохода всех стадий в порядке cycle с учётом пропусков.
- **Опции:** num_steps (число витков), training, device, seed, callbacks, dry_run и т.д.; пробрасываются в run стадий по соглашению реализации.

### 8.2 Первая итерация: старт без предзаданного state

Если при первом запуске **state не передан или пуст**:

- Стадии, условие которых требует заполненных слотов (например, автор, творец по данным слотам), **пропускаются** — can_run ложно.
- **Стадия среды** может выполниться: подшаг **Development** (развитие мира) часто задаётся как **безусловный** — читает контент мира и опционально action, порождает **первый state** (например, слот намерения). Этот state передаётся следующей стадии в цикле (стадия творца при порядке автор → среда → творец; или следующая по конфигу).
- Далее цикл продолжается как обычно: state проходит оставшиеся стадии, затем при следующем витке — все стадии по порядку.

Таким образом, мир способен **породить первый state из себя**, без обязательного внешнего state; первая итерация может **начаться со стадии среды** (Development). Внешний ввод (state от пользователя) встраивается в ту же логику: при переданном state цикл стартует с первой стадии в порядке cycle, каждая стадия выполняется или пропускается по can_run.

### 8.3 Передача state и action пользователем

- **Передача state** на вход run(world, state, …) задаёт начальное заполнение слотов. Каждая стадия выполняется или пропускается по своему условию can_run; полный state после обновления мира (World update в среде) может сохраняться в storage по конфигу.
- **Передача action** обогащает контекст стадии среды при Development; при отсутствии action развитие мира идёт по текущему контенту мира. Контракт run мира может принимать action как именованный аргумент или включать его в context.

### 8.4 Детерминированность и воспроизводимость

При одних и тех же **конфиге мира**, **чекпоинтах всех стадий** (и вложенных воркфлоу), **state**, **action** и **context** результат одного витка детерминирован в пределах контрактов стадий. Конфиг мира + чекпоинты стадий + state + action + context задают воспроизводимое поведение.

## 9. Сериализация уровня мира

### 9.1 Два крупных поля сериализации

Сериализация мира выделяет **два крупномасштабных поля** ([../../philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md)):

| Поле | Содержимое |
|------|------------|
| **Исполняемая часть** | Структура цикла (cycle), конфиги всех стадий (и вложенных воркфлоу, гиперграфов задач), state_schema, storage, ссылки на чекпоинты, параметры запуска. Всё, что нужно для **сборки и выполнения** мира. |
| **Контент мира** | История, глоссарий, персонажи, правила, стиль, артефакты, нарратив — всё, что **наполняет** мир и обновляется при обновлении мира (World update в стадии среды). |

**Развивающийся мир** сериализуется **обоими** полями: из такой сериализации мир можно собрать, запустить и восстановить контент. **Завершённый мир (End World)** сериализуется **только контентом** — без исполняемой части (без конфигов стадий, без чекпоинтов, без возможности запуска цикла). End World используется как мир только для чтения (например, книга, зафиксированный сюжет); другие миры могут ссылаться на него и обогащать свой контекст, не изменяя End World ([../../reference/WorldGenerator_2.0/END_WORLDS.md](../../reference/WorldGenerator_2.0/END_WORLDS.md)).

### 9.2 Конфиг мира (исполняемая часть)

- **Конфиг мира** задаёт: **cycle** (список stage_id); **stages** (stage_id → конфиг стадии по 05 §9.1 или ref); **state_schema**; **storage** (опционально); **schema_version**, world_id, metadata.
- При загрузке по конфигу каждая стадия строится из своего конфига (05.from_config); восстанавливаются цикл, state_schema, storage. Контент мира загружается отдельно (из поля контента сериализации или initial_content).

### 9.3 Чекпоинт мира

- Мир **не имеет своего единого state_dict** — состояние распределено по стадиям (и рекурсивно по воркфлоу и гиперграфам). **Чекпоинт мира** — совокупность чекпоинтов всех входящих в него стадий (словарь stage_id → чекпоинт стадии или директория по stage_id). По соглашению реализация предоставляет **state_dict()** (агрегат state_dict стадий) и **load_state_dict(state)**.
- **Полное сохранение:** конфиг мира (исполняемая часть) + чекпоинты всех стадий + опционально текущий state и снапшот контента мира. **Загрузка:** прочитать конфиг, построить мир (стадии из конфигов), загрузить чекпоинты в стадии, восстановить контент из снапшота или initial_content. Возможна загрузка только чекпоинтов в уже собранный мир (load_checkpoint(checkpoint_dir)).

## 10. Контент мира, обновление мира и End World

### 10.1 Контент мира

**Контент мира** — всё, что наполняет мир как субъект цикла: история, глоссарий, персонажи, правила, стиль, артефакты (в сохранённом виде), нарратив. Контент **читается** стадиями (в частности, стадией среды при Development и при обновлении мира) и **обновляется** при **World update** (когда стадия среды применяет полный state к миру — дополнение истории, обновление глоссария и т.д. по правилам реализации). Формат контента задаётся конфигом мира и реализацией (словарь слотов, структурированный документ, хранилище с версионированием и т.д.).

### 10.2 Обновление мира (World update) и развитие (Development)

В стадии среды при полном state может выполняться **World update**: применение state к контенту мира, сохранение state в storage. При **Development** стадия среды порождает **новый state** для следующего витка (например, слот намерения) на основе контента мира и опционально action. Условия (World update только при полном state; Development безусловно или по правилу) задаются конфигом стадии среды; мир передаёт в context контент, storage и action.

### 10.3 Завершённый мир (End World)

**End World** — мир, в сериализации которого присутствует **только контент** (второе поле), **без** исполняемой части. Такой мир нельзя перезапустить или изменить; к нему можно обращаться для чтения. Развивающиеся миры могут ссылаться на End World (гиперссылки, идентификатор); контекст End World обогащает стадии (например, автора, среды), при этом End World остаётся неизменным. Полная спецификация — в референсе [END_WORLDS.md](../../reference/WorldGenerator_2.0/END_WORLDS.md); уровень мира фиксирует, что сериализация различает два поля и что End World = только контент.

## 11. Граничные случаи и тонкости

### 11.1 Обучение на уровне мира

Отдельные стадии (и вложенные воркфлоу) могут быть помечены как trainable; при run(world, state, training=True) опция пробрасывается в run стадий. Сериализация чекпоинта мира включает чекпоинты всех стадий. Дедупликация по общим чекпоинтам (один ref — один экземпляр) по соглашению реализации.

### 11.2 Развёртывание и to(device)

Стадии остаются единицами развёртывания; мир оркестрирует вызовы. **to(device)**, **trainable_parameters()**, **set_trainable(stage_id, bool)** по соглашению делегируются стадиям. API-стадии (стадия, вызываемая по удалённому API) подключаются через тот же контракт run(stage, state, context) → state; мир передаёт state и получает state от API.

### 11.3 Обработка ошибок

Философия не предписывает единую политику. Сбой стадии может прерывать run мира или обрабатываться по политике вызывающего (ретраи, частичный state, логирование). Уровень мира фиксирует контракт run и передачу state; детали устойчивости — в реализации.

---

**Итог.** Документ задаёт **полную техническую спецификацию уровня мира**: мир как гиперграф стадий (узлы = стадии, по гиперрёбрам — state), **цикл** из трёх стадий (автор, среда, творец), **state** и state_schema, условия выполнения стадий (can_run), контракт **run(world, state, action?, context?)**, **первая итерация** без предзаданного state, **action** как опциональный контекст, **контент мира** и его обновление, сериализация в **два поля** (исполняемая часть и контент), **End World** (только контент), развёртывание и граничные случаи. Реализация и уровень вселенной опираются на эту спецификацию.
