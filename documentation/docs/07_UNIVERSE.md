# 07. Уровень вселенной

**Назначение:** полная **техническая** документация **последнего (пятого) исполняемого уровня** системы Иггдрасиль — вселенной. Документ задаёт вселенную как **гиперграф миров**: узлы вселенной — **целые миры** (развивающиеся или завершённые — End World), по гиперрёбрам передаётся **обмен** (payload_spec: снимки state, артефакты, контекст, события, сущности). Описаны **единая среда вселенной (эфир)**, **payload_spec** и маппинг между мирами, **переход сущностей** и **кросс-мировое влияние**, **End World** как узел только для чтения, **выполнение** (обход гиперграфа миров, run(world), передача по рёбрам), **сообщество** как целевое назначение уровня, **слияние миров**, построение и валидация, сериализация, **граница онтологии** (уровень метавселенной не вводится), граничные случаи. Канонический источник технических решений уровня вселенной.

**Предыдущий уровень:** [06_WORLD.md](06_WORLD.md) — мир (гиперграф стадий, state по циклу автор → среда → творец).

**Философский якорь:** [../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7 — уровень вселенной как воплощение идеи целого и неделимого; последний уровень исполнения и последний уровень с полной структурой гиперграфа; вселенная = гиперграф миров; единая среда вселенной; миры развивающиеся и завершённые; гиперрёбра и групповые связи; гиперссылки на завершённые миры; обмен данными и сущностями; переход сущностей из мира в мир и влияние на оба мира; слияние миров; единый поток трансляции. **Граница онтологии** (конец §7): метавселенная как уровень **не вводится**; мета = взаимодействие пользователей в реальном мире и публикация вселенных на единой площадке; вселенная — высший уровень.

**Язык:** русский.

---

## 1. Место уровня в иерархии

| Уровень | Сущность | Что течёт по связям | Описание |
|---------|----------|---------------------|----------|
| **Мир** | Гиперграф, узлы которого — стадии | **State** (контейнер слотов) | Один мир = один цикл стадий (автор → среда → творец). [06_WORLD.md](06_WORLD.md). |
| **Вселенная** | **Вселенная** — гиперграф, узлы которого — **миры** | **Обмен** (payload_spec) | **Пятый исполняемый уровень и высший уровень онтологии.** Узлы = миры (развивающиеся или End World); по гиперрёбрам — снимки state, артефакты, контекст, события, сущности. Единая среда вселенной (эфир). |

**Выше вселенной в онтологии системы уровня нет.** Вселенная — последний уровень, на котором система исполняется, и последний уровень, на котором структура гиперграфа представлена в полном виде ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7). То, что в массовой культуре именуют метавселенной, по замыслу проекта образуется из **взаимодействия пользователей в реальном мире** и **публикации вселенных на единой площадке** — без введения формального уровня «метавселенная» в онтологию.

## 2. Вселенная — определение и смысл

### 2.1 Определение

**Вселенная (Universe)** — это **гиперграф, узлами которого являются миры (World)**. Каждый узел — один мир (идентифицируется **world_id**): либо **развивающийся мир** (имеет исполняемую часть, цикл стадий, state обновляется при run), либо **завершённый мир (End World)** (фиксирован, только контент, только чтение). По **гиперрёбрам** между мирами передаётся **обмен** — содержимое которого задаётся **payload_spec** (снимок state, артефакты, контекст, события, сущности с состоянием). Вселенная обладает **единой средой вселенной (эфиром)** — общей средой, в которой миры существуют и взаимодействуют.

- **Узлы вселенной** — миры. Каждый мир — самостоятельная единица со своим циклом стадий (автор → среда → творец), своим state и контентом. Мир вызывается как run(world, state?, action?, context?) по контракту уровня мира ([06_WORLD.md](06_WORLD.md)).
- **Гиперрёбра вселенной** — связи между мирами. Каждое ребро (или гиперребро арности > 2) задаёт: **источник** (world_id или множество миров), **назначение** (world_id или множество), **payload_spec** — что передаётся (тип данных, направление, момент синхронизации). По рёбрам не течёт «сырой» state одного мира как целое без соглашения: payload_spec описывает, какие части state, артефактов или сущностей передаются и как маппятся на входы или контекст мира-назначения.
- **Выполнение вселенной** — обход гиперграфа миров (топологический порядок, фиксированный порядок или цикл по конфигу); для каждого мира вызов run(world, …); выходы мира A по payload_spec маппятся на входы или контекст мира B по входящим гиперрёбрам. При переходе **сущностей** (персонаж, агент) из мира A в мир B обновляются state мира-источника и мира-назначения по правилам кросс-мирового влияния.

Вселенная не подменяет мир, а **добавляет уровень над ним**: множество миров и связи между ними; пользователь может взаимодействовать с разными мирами, переходить от одного к другому, обогащать контекст одного мира контекстом другого ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7).

### 2.2 Граница с миром

**Мир** — атомарная единица уровня мира: один цикл стадий, state течёт по стадиям, контракт run(world, state, action?, context?) → state. **Вселенная** оркестрирует **множество миров**: задаёт, какие миры связаны, что передаётся по рёбрам (payload_spec), в каком порядке вызывать run(world, …), как маппить выходы одного мира на входы другого. Мир не знает о вселенной; вселенная передаёт мирам входы, собранные из исходящих данных входящих рёбер и из опций run вселенной. Контракт мира не меняется.

### 2.3 Целое и неделимое: последний уровень

Как и уровень гиперграфа задачи, уровень вселенной воплощает в коде идею **целого и неделимого** (первый пункт Части 1 философии). Вселенная — **последний уровень, на котором система исполняется**, и **последний уровень, на котором структура гиперграфа представлена в проекте в полном виде**. Подобно тому как гиперграф задачи может служить бесконечному числу задач, вселенная может вмещать **бесконечное число завершённых миров** и развивающихся миров ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7).

## 3. Узлы вселенной: миры (развивающиеся и End World)

### 3.1 Развивающийся мир (объектный мир)

- **Развивающийся мир** — мир, который **развивается**: имеет исполняемую часть (цикл стадий, конфиги стадий и вложенных воркфлоу), state обновляется при run(world, …), контент мира может обновляться при обновлении мира (World update в стадии среды). Сериализация развивающегося мира включает **два поля**: (1) структура/исполняемая часть, (2) контент мира ([06_WORLD.md](06_WORLD.md) §9.1).
- В гиперграфе вселенной развивающийся мир — узел, который может **получать** данные по входящим рёбрам (контекст, артефакты, сущности, снимки state) и **отдавать** данные по исходящим рёбрам (state snapshot, артефакты, сущности) согласно payload_spec.

### 3.2 Завершённый мир (End World)

- **End World** — мир, который **не изменяется** и **фиксирован**: в сериализации присутствует **только контент мира** (история, глоссарий, персонажи, сюжет — например, книга как законченный текст), **без** исполняемой части (без весов, без конфига запуска цикла). К End World можно **привязываться** по **гиперссылке** (world_id с типом end_world); развивающиеся миры и персонажи могут **читать** его контекст (обогащение диалогов, цитирование, отсылки), при этом **End World не меняется**.
- В гиперграфе вселенной End World — узел **без исходящих рёбер с записью** в другой мир: только входящие «ссылки» от других миров (чтение контента). Гиперрёбра вида «мир A → End World» семантически означают «мир A имеет доступ к контенту End World по чтению»; обратная запись в End World движком запрещена.

## 4. Гиперрёбра вселенной и payload_spec

### 4.1 Назначение гиперрёбер

Гиперрёбра задают **связи между мирами** и **что передаётся** по каждой связи. Формат ребра размера 2: (source_world_id, target_world_id, **payload_spec**). При **гиперрёбрах арности > 2** — множества источников и/или назначений и общий payload_spec (один источник → несколько миров, несколько миров → один мир или групповой обмен по соглашению реализации).

### 4.2 payload_spec

**payload_spec** — спецификация полезной нагрузки ребра: что именно передаётся от мира-источника к миру-назначению. Типичные элементы:

| Элемент | Описание |
|---------|----------|
| **state_snapshot** | Снимок state мира-источника (полный или подмножество слотов); мир-назначение может использовать его как контекст или для инициализации своего state. |
| **artifacts** | Артефакты (изображения, видео, текст, структурированные данные); передача по слоту артефактов state или по отдельному каналу. |
| **context** | Произвольный контекст (глоссарий, правила, описание сцены) для подстановки в мир-назначение. |
| **events** | События (например, «сущность X перешла в мир B»); триггеры для обновления state мира-назначения или мира-источника. |
| **entities** | **Сущности с состоянием** (персонажи, агенты): объект с идентификатором, состоянием и при необходимости **origin_world_id**. Передача сущности по ребру = «переход персонажа/агента из мира A в мир B»; в мире B сущность действует и влияет на историю B; обратная связь в мир A (отсутствие, возврат, отражение событий) обновляет историю в A ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7). |

Направление влияния может быть **A → B**, **B → A** или **двунаправленное**; в payload_spec или в конфиге ребра задаётся **момент синхронизации** (после run(world_A), после run(world_B), после обхода подграфа). Конкретный формат payload_spec задаётся реализацией и конфигом вселенной.

### 4.3 Маппинг выходов мира на входы другого мира

При выполнении run(universe, …) после run(world_A) выходы мира A (state, артефакты, список сущностей для передачи и т.д.) **маппятся** на входы или контекст мира B по правилу, заданному payload_spec входящего ребра (B ← A). Конкретика: какие поля state A подаются в мир B (как начальный state, как контекст для стадии среды, как слоты state) и как записываются сущности в мир B — задаётся конфигом вселенной и контрактом миров.

## 5. Единая среда вселенной (эфир)

**Единая среда вселенной** («эфир вселенной») — **общая среда**, в которой миры существуют и взаимодействуют ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7; в мифе Иггдрасиль — колодец Урд). В технической модели:

- **Общий контекст:** хранилище или каталог (глоссарий, правила, стиль), доступное всем мирам по гиперрёбрам или при инициализации. Миры могут читать из эфира при run (например, при развитии мира в стадии среды или при переходе персонажа).
- **Единый поток трансляции:** поток событий и взаимодействий, затрагивающих экосистему; при необходимости вселенная или платформа обеспечивает доведение значимых событий (переход сущности, обновление мира) до заинтересованных миров или пользователей.
- **Реализация:** минимально — общий контекст-словарь или каталог (ключ–значение); расширенно — хранилище с версионированием, подписками, правилами доступа. Контракт вселенной допускает наличие эфира как опционального компонента; миры получают доступ к эфиру при run (через context или по ссылке из конфига вселенной).

## 6. Состав вселенной: узлы, гиперрёбра, эфир, порядок обхода

### 6.1 Узлы = миры

- Каждый **узел вселенной** — целый **мир** ([06_WORLD.md](06_WORLD.md)). У каждого узла есть идентификатор **world_id**. Тип узла: развивающийся мир или End World (задаётся конфигом или маркером при добавлении).
- Мир может быть задан вложенным конфигом (по спецификации 06) или ссылкой (ref). При загрузке вселенной каждый мир-узел строится из своего конфига (06.from_config для развивающегося мира; для End World — только контент).

### 6.2 Гиперрёбра и порядок обхода

- **Гиперрёбра** — список или множество рёбер (source_world_id, target_world_id, payload_spec); при арности > 2 — add_hyperedge(sources, targets, payload_spec) по соглашению реализации.
- **Порядок обхода** задаётся конфигом вселенной: топологическая сортировка (если DAG), фиксированный список world_id, или цикл по подмножеству миров. Порядок определяет, в какой последовательности вызывается run(world, …) при выполнении вселенной.

### 6.3 Контракт вселенной

| Элемент | Описание |
|---------|----------|
| **worlds** | Множество узлов: world_id → мир (развивающийся или End World). |
| **edges** | Гиперрёбра (source, target, payload_spec); при арности > 2 — список концов с разметкой. |
| **ether** | Опционально: конфиг или ссылка на единую среду вселенной (общий контекст, поток трансляции). |
| **order / topology** | Порядок обхода миров при run(universe, …). |

## 7. Построение вселенной

### 7.1 Добавление миров

- **add_world(world_id, world)** — добавить мир (развивающийся или End World) как узел вселенной с заданным world_id. Мир уже собран. Дублирование world_id должно приводить к ошибке.
- **add_world(world_id, config, registry?)** — построить мир из конфига (06.from_config) и добавить как узел. Для End World конфиг задаёт только контент и тип «End World».
- **remove_world(world_id)** — удалить мир из вселенной; инвалидировать гиперрёбра, ссылающиеся на этот world_id.

### 7.2 Добавление гиперрёбер

- **add_edge(source_world_id, target_world_id, payload_spec)** — добавить ребро между мирами; payload_spec задаёт, что передаётся (state_snapshot, artifacts, context, events, entities) и при необходимости направление и момент синхронизации.
- **add_hyperedge(sources, targets, payload_spec)** — при поддержке гиперрёбер арности > 2: групповые связи (несколько источников и/или несколько назначений).
- **remove_edge(...)** / **remove_hyperedge(...)** — удалить ребро; инвалидирует кэш порядка обхода при его наличии.

### 7.3 Эфир и порядок обхода

- **set_ether(config)** или **set_ether(ether_ref)** — задать единую среду вселенной (общий контекст, путь к хранилищу, конфиг потока трансляции).
- **set_order(world_ids)** или **set_topology(...)** — задать порядок обхода миров (список world_id или правило топологии). При отсутствии — порядок выводится из структуры гиперграфа (топологическая сортировка для DAG) или задаётся реализацией.

### 7.4 Построение из конфига

- **Конфиг вселенной:** **worlds** — словарь world_id → конфиг мира (по 06 §9.2 или ref) с опциональным типом (object / end_world); **edges** — список (source_world_id, target_world_id, payload_spec) и при необходимости гиперрёбра; **ether** (опционально); **order** или **topology**; universe_id, schema_version, metadata.
- **from_config(config, registry?, validate?)** — построить миры из конфигов, добавить рёбра, восстановить эфир и порядок обхода. При validate — валидация вселенной (§8).

## 8. Валидация вселенной

Перед выполнением или сериализацией выполняется **валидация** структуры вселенной.

### 8.1 Обязательные проверки

| Проверка | Описание |
|----------|----------|
| **Наличие миров** | Все world_id в гиперрёбрах существуют в множестве узлов (worlds). |
| **End World** | Узлы с типом End World не имеют исходящих рёбер с записью в другой мир; допустимы только входящие связи «доступ на чтение». |
| **payload_spec** | Спецификации совместимы с контрактами миров (какие поля state/артефактов доступны, формат сущностей) в пределах проверяемого. |
| **Топология** | При заданном порядке обхода или DAG — достижимость и отсутствие неразрешимых циклов (или явная семантика цикла в конфиге). |

### 8.2 Дополнительные проверки

| Проверка | Описание |
|----------|----------|
| **Валидность миров** | Опционально вызывать валидацию каждого развивающегося мира по правилам 06 §6. |
| **Совместимость маппинга** | Выходы мира-источника (по payload_spec) совместимы с ожидаемыми входами или контекстом мира-назначения. |

### 8.3 Результат валидации

- Валидация возвращает список **ошибок** и **предупреждений**. В строгом режиме run не выполняется при наличии ошибок; при from_config(validate=True) невалидная вселенная не создаётся или выдаёт исключение.

## 9. Исполнитель вселенной и жизненный цикл run

Вселенная выполняет **обход миров** в порядке, заданном конфигом (топология, список world_id или цикл). Для каждого мира: собрать **входы** из исходящих данных входящих гиперрёбер (по payload_spec) и из опций run(universe, …); вызвать **run(world, state?, action?, context?)**; собрать **выходы** мира (state, артефакты, сущности для передачи) и записать их в буферы исходящих рёбер по payload_spec. При переходе сущностей — применить правила кросс-мирового влияния (обновление state мира-источника и мира-назначения).

### 9.1 Компоненты

| Компонент | Назначение |
|-----------|------------|
| **Хранилище структуры** | Множество узлов (world_id → мир), гиперрёбра (edges), эфир (ether), порядок обхода (order/topology), universe_id, metadata. |
| **Валидатор** | Проверка согласованности вселенной (§8). |
| **Планировщик** | Определение порядка обхода миров (топологическая сортировка, фиксированный список или цикл). |
| **Буферы** | Для каждого исходящего ребра — данные, подготовленные по payload_spec после run(world_A); при run(world_B) входы мира B собираются из буферов входящих рёбер. |
| **Исполнитель** | Обход миров по плану; для каждого мира — сбор входов из буферов и опций, вызов run(world, …), запись выходов в буферы по payload_spec; при необходимости — передача сущностей и обновление state по правилам кросс-мирового влияния. |

### 9.2 Жизненный цикл одного раунда (проход по мирам)

1. **Вход:** universe, world_inputs? (словарь world_id → начальный state или входы для мира), options (num_rounds, контекст эфира, активный мир пользователя и т.д.).
2. **План:** построить или взять из кэша порядок обхода миров (топология или order из конфига).
3. **Инициализация буферов:** для миров без входящих рёбер — входы из world_inputs или пустой state; для остальных — буферы входящих рёбер заполняются по мере выполнения миров-источников.
4. **Обход:** для каждого мира в порядке плана: собрать входы из буферов входящих рёбер (по payload_spec) и из world_inputs; сформировать context (в т.ч. эфир, ссылки на End World); вызвать **run(world, state?, action?, context?)** → state (или результат витка); из выходов мира по payload_spec заполнить буферы исходящих рёбер; при передаче сущностей — обновить state мира-источника и мира-назначения по правилам кросс-мирового влияния.
5. **Выход:** по соглашению реализации — словарь world_id → state (или результат витка каждого мира), агрегированный отчёт, итератор по раундам.

## 10. Контракт run вселенной

### 10.1 Сигнатура и семантика

- **run(universe, world_inputs?, **options) → results**
- **Вход (world_inputs):** опционально — словарь world_id → начальный state или входы для соответствующего мира. Используется для инициализации миров без входящих рёбер или для передачи пользовательского ввода в выбранный мир.
- **Выход (results):** зависит от реализации: например, словарь world_id → state после раунда; агрегированный отчёт; итератор по раундам. Канон задаёт, что вселенная **выполняет обход миров и передаёт данные по гиперрёбрам**; точный формат возврата — в реализации.
- **Опции:** num_rounds (число проходов по мирам), order_override, filter_worlds, ether_context, active_world_id (мир, в котором находится пользователь, для контекста), callbacks, dry_run и т.д.

### 10.2 Переход сущностей и кросс-мировое влияние

При передаче **сущностей** (entities) по payload_spec из мира A в мир B:

- Сущность помечается **origin_world_id** (мир A). В мире B она может участвовать в переходах (как персонаж или агент); её действия влияют на **историю мира B** и на state B.
- **Обратная связь в мир A:** при «отсутствии» сущности в A (она в B) state мира A может обновляться по правилам (например, слот «персонаж X в мире B»); при возврате сущности или при получении событий из B (отражение ключевых событий в биографию/state A) state мира A обновляется. Таким образом, **одна сущность, переместившаяся из A в B, влияет и на историю B, и на свою историю в A** ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7).
- Реализация правил обратной связи и момента синхронизации задаётся контрактом вселенной и конфигом payload_spec (например, «после run(world_B) отправить события в мир A и обновить state A»).

### 10.3 Режимы выполнения

- **Последовательный режим:** линейная цепочка миров; порядок задаётся списком world_id; рёбра (i → i+1); передача по payload_spec.
- **Параллельный запуск:** миры без рёбер между собой могут выполняться независимо (параллельно или в произвольном порядке); граф задаёт только связи там, где они есть.
- **Цикл по подмножеству:** опция num_rounds или итерация до условия (один проход по всем мирам или цикл по выбранным мирам) по соглашению реализации.

### 10.4 Детерминированность и воспроизводимость

При одних и тех же **конфиге вселенной**, **чекпоинтах всех развивающихся миров**, **world_inputs** и **опциях** результат run(universe, …) детерминирован в пределах контрактов миров и заданных правил payload_spec и кросс-мирового влияния. Конфиг вселенной + чекпоинты миров + world_inputs + опции задают воспроизводимое поведение.

## 11. Сериализация уровня вселенной

### 11.1 Конфиг вселенной (структура)

- **Конфиг вселенной** задаёт: **worlds** (world_id → конфиг мира по 06 §9.2 или ref; для End World — конфиг только контента и тип end_world); **edges** (список (source_world_id, target_world_id, payload_spec) и при необходимости гиперрёбра); **ether** (опционально: конфиг или путь к общей среде); **order** / **topology** (опционально); universe_id, schema_version, metadata.
- При загрузке по конфигу каждый мир строится из своего конфига (06.from_config для развивающегося мира; для End World — загрузка только контента); восстанавливаются гиперрёбра, эфир, порядок обхода.

### 11.2 Чекпоинт вселенной

- Вселенная **не имеет своего единого state_dict** — исполняемое состояние распределено по **развивающимся** мирам. **Чекпоинт вселенной** — совокупность чекпоинтов всех развивающихся миров (словарь world_id → чекпоинт мира по 06 §9.3 или директория по world_id). End World не имеют исполняемого чекпоинта; их контент хранится в сериализации контента (одно поле). Эфир вселенной при необходимости сохраняется отдельно (контекст, каталог).
- По соглашению реализация предоставляет **state_dict()** (агрегат state_dict развивающихся миров) и **load_state_dict(state)**. Полное сохранение: конфиг вселенной + чекпоинты всех развивающихся миров + опционально снапшот эфира. Загрузка: восстановление миров из конфигов, загрузка чекпоинтов в развивающиеся миры, восстановление эфира. Возможна загрузка только чекпоинтов в уже собранную вселенную.

## 12. End World в вселенной, сообщество, слияние миров

### 12.1 End World как узел вселенной

- **End World** в гиперграфе вселенной — узел с типом «только чтение». Гиперссылки на End World (world_id с типом end_world) могут храниться в state развивающегося мира или в метаданных вселенной. При выполнении стадий мира (автор, среда, диалоги) контекст End World подставляется по ссылке (чтение). Запись в End World движком запрещена.

### 12.2 Сообщество как целевое назначение уровня

Уровень вселенной предназначен **в первую очередь для сообщества пользователей** ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7):

| Цель | Описание |
|------|----------|
| **Сравнение миров** | Пользователи могут сравнивать свои миры друг с другом (контент, state, сюжеты). |
| **Развитие через взаимодействие** | Развитие миров на основе взаимодействия персонажей с мирами других пользователей (переход сущностей, кросс-мировое влияние). |
| **Единая система взаимодействий** | Построение сложных интеграций между мирами (направление влияния, синхронизация, payload_spec). |
| **Обмен и совместное творчество** | Делиться результатами, обмениваться идеями, расширять миры за счёт взаимодействия. |
| **Группы миров** | Формирование групп миров по общим идеям или сюжету. |
| **Влияние на экосистему** | Творческое и при необходимости деструктивное влияние: синтез новых миров из группы, аннигиляция миров по правилам сценария — по соглашению реализации. |

На уровне вселенной как объединения миров существует **единый поток трансляции** всех взаимодействий; важны **децентрализация** и **полная распределённость** при сохранении возможностей нижележащих уровней и удобстве для сообщества.

### 12.3 Слияние миров

Миры во вселенной могут быть **объединены в один мир** несколькими допустимыми способами: к мирам как узлам гиперграфа применяется слияние, объединяются содержание и состояние нескольких миров; выбор стратегии слияния остаётся в рамках контракта и канона ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §7; Philosophy-Part3 §3.8). Конкретные стратегии (объединение контента, слияние state, разрешение конфликтов) задаются реализацией и конфигом.

## 13. Граница онтологии: уровень метавселенной не вводится

**Уровень метавселенной в онтологии Иггдрасиль не вводится** ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md), конец §7). То, что в массовой культуре именуют метавселенной, по замыслу проекта должно складываться из **взаимодействия пользователей в реальном мире** и **публикации их вселенных на единой площадке**, где другие пользователи могут исследовать, сравнивать и взаимодействовать с этими вселенными. Реальное взаимодействие людей — вот где возникает подлинная «мета». Вселенная остаётся **высшим уровнем** онтологии. Мета — не следующий уровень гиперграфа, а **способ использования вселенной людьми**. Документация и реализация не добавляют уровень «метавселенная» над вселенной.

## 14. Граничные случаи и тонкости

### 14.1 Обучение на уровне вселенной

Отдельные развивающиеся миры (и вложенные стадии/воркфлоу) могут быть помечены как trainable; при run(universe, …, training=True) опция пробрасывается в run соответствующих миров. Сериализация чекпоинта вселенной включает чекпоинты всех развивающихся миров. Дедупликация по общим чекпоинтам (один ref — один экземпляр) по соглашению реализации.

### 14.2 Развёртывание и to(device)

Миры остаются единицами развёртывания; вселенная оркестрирует вызовы. **to(device)**, **trainable_parameters()**, **set_trainable(world_id, bool)** по соглашению делегируются развивающимся мирам. API-миры (мир, вызываемый по удалённому API) подключаются через тот же контракт run(world, state?, action?, context?) → state; вселенная передаёт state и получает state от API.

### 14.3 Обработка ошибок

Философия не предписывает единую политику. Сбой мира может прерывать run вселенной или обрабатываться по политике вызывающего (ретраи, пропуск мира, логирование). Уровень вселенной фиксирует контракт run и передачу данных по гиперрёбрам; детали устойчивости — в реализации.

---

**Итог.** Документ задаёт **полную техническую спецификацию уровня вселенной**: вселенная как гиперграф миров (узлы = миры развивающиеся и End World, по гиперрёбрам — обмен по payload_spec), **единая среда вселенной (эфир)**, **payload_spec** и маппинг между мирами, **переход сущностей** и **кросс-мировое влияние**, **End World** (только чтение, гиперссылки), контракт **run(universe, world_inputs?, **options)**, сериализация, **сообщество** как целевое назначение уровня, **слияние миров**, **граница онтологии** (метавселенная не вводится), граничные случаи. Вселенная — **последний исполняемый уровень** и **высший уровень онтологии** системы; реализация опирается на эту спецификацию.
