# 05. Уровень стадии

**Назначение:** полная **техническая** документация **третьего исполняемого уровня** системы Иггдрасиль — стадии. Документ задаёт стадию как **гиперграф воркфлоу**: узлы стадии — **целые воркфлоу**, по гиперрёбрам стадии между воркфлоу передаются **данные**; **внешний** вход и выход стадии в контексте мира — **state** (контейнер слотов). Описаны state и его связь со стадией, маппинг **state ↔ входы/выходы воркфлоу** (state_input_map, state_output_map), условие выполнения **can_run(stage, state)**, состав стадии (узлы-воркфлоу, гиперрёбра, контракт по state), построение и валидация, применение того же гиперграфового движка, контракт **run(stage, state) → state**, сериализация, **три стадии мира** (автор, среда, творец), вырожденный случай и граничные случаи. Канонический источник технических решений уровня стадии.

**Предыдущий уровень:** [04_WORKFLOW.md](04_WORKFLOW.md) — воркфлоу (гиперграф гиперграфов задач, данные между задачами).

**Философский якорь:** [../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §5 — уровень стадии как метаморфоза; три стадии (автор, среда, творец); state между стадиями; §6 — мир как гиперграф стадий, state по рёбрам.

**Язык:** русский.

## 1. Место уровня в иерархии

| Уровень | Сущность | Что течёт по связям | Описание |
|---------|----------|---------------------|----------|
| **Воркфлоу** | Гиперграф, узлы которого — гиперграфы задач | **Данные** между гиперграфами | Комбинированные сценарии (несколько задач в цепочке или ветвлении). [04_WORKFLOW.md](04_WORKFLOW.md). |
| **Стадия** | **Стадия** — гиперграф, узлы которого — **воркфлоу** | Внутри стадии: **данные** между воркфлоу; вход/выход стадии в мир: **state** | **Третий исполняемый уровень.** Одна стадия = **одна метаморфоза** state (качественное превращение: материя → идея, идея → материя). |
| **Мир** | Гиперграф стадий (узлы = стадии) | **State** (контейнер слотов) | Цикл мира: автор → среда → творец; по рёбрам передаётся state. [06_WORLD.md](06_WORLD.md). |

Ключевое отличие стадии от воркфлоу: на **воркфлоу** по гиперрёбрам передаются **данные** (тензоры, изображения, текст и т.д.); **стадия** в контексте мира получает и возвращает **state** — структурированный контейнер слотов (намерение, конфигурация, артефакты, описания, нарратив и т.д. по соглашению уровня мира). Одна стадия реализует **одну метаморфозу** в цикле мира ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §5, §6).

## 2. Стадия — определение и смысл

### 2.1 Определение

**Стадия** — это **гиперграф, узлами которого являются целые воркфлоу**. Стадия реализует **одну метаморфозу** state: качественное превращение (материя → идея, идея → материя, артефакты → описание, описание → конфигурация, конфигурация → артефакты). С точки зрения движка фреймворка стадия — **тот же гиперграф**: узлы = воркфлоу; по гиперрёбрам **внутри** стадии между воркфлоу передаются **данные** (как на уровне воркфлоу); **внешний** вход и выход стадии в контексте мира — **state**.

- **Узлы стадии** — воркфлоу (один или несколько). Каждый воркфлоу имеет контракт run(workflow, inputs) → outputs; входы и выходы воркфлоу **маппятся** на слоты state по конфигурации стадии (state_input_map, state_output_map).
- **Гиперрёбра стадии** (между воркфлоу): соединяют внешний выход одного воркфлоу с внешним входом другого; по ним текут данные. При одном воркфлоу рёбер нет — вырожденный случай.
- **Вход стадии (по state):** какие слоты state читаются и как подаются на воркфлоу-узлы. **Выход стадии (по state):** какие выходы воркфлоу записываются в какие слоты state.
- **Выполнение стадии:** проверить условие can_run(stage, state); если выполнено — собрать inputs для воркфлоу из state (и контекста), выполнить воркфлоу в порядке топологии (или по плану при циклах), записать результаты в state; вернуть обновлённый state.

Таким образом, **одна стадия** в цикле мира (автор, среда, творец) реализуется **одним** таким гиперграфом воркфлоу с явным контрактом «вход/выход по state» и опциональным условием выполнения.

### 2.2 Граница с воркфлоу

**Воркфлоу** оперирует **данными** (inputs/outputs — словари с тензорами, строками, сообщениями). **Стадия** в контексте мира оперирует **state**: при вызове из мира стадия получает state, маппит его слоты на входы воркфлоу-узлов, выполняет воркфлоу, маппит выходы воркфлоу обратно в слоты state и возвращает обновлённый state. Воркфлоу внутри стадии по-прежнему вызывается как run(workflow, inputs) → outputs; стадия лишь задаёт, откуда брать inputs (из state) и куда писать outputs (в state) ([04_WORKFLOW.md](04_WORKFLOW.md) §7).

### 2.3 Три стадии мира (философский контекст)

По философии проекта ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §5) вводятся **три стадии**: стадия автора, стадия среды, стадия творца.

| Стадия | Роль в цикле мира |
|--------|-------------------|
| **Стадия творца** | Воплощение **материального** начала: превращение идеальной составляющей в материальную. Идея того, что должно произойти → конфигурация для генерации → артефакты (видео, звук, изображения). |
| **Стадия автора** | Воплощение **идеального** начала: превращение материальных артефактов в идею. Что на них изображено, что происходит → описание в форме действий, произошедших в мире. |
| **Стадия среды** | **Переход** — живая связь материального и идеального: принимает описания от автора, помещает их в историю мира, порождает следующий шаг развития мира и передаёт его стадии творца. Может включать обновление мира по state и порождение нового state (развитие мира). |

Порядок исполнения в мире фиксирован; гиперграф **цикличен**. Детали цикла мира, условия выполнения стадий и семантика слотов state задаются на уровне мира; уровень стадии предоставляет контракт run(stage, state) → state и can_run(stage, state).

## 3. State — контейнер слотов и связь со стадией

### 3.1 Определение state

**State** (состояние мира) — **контейнер с именованными слотами (блоками)**. Он передаётся между стадиями в цикле мира и претерпевает метаморфозы: каждая стадия читает одни слоты и дополняет или перезаписывает другие. Семантика слотов (намерение, конфигурация, артефакты, описания, нарратив и т.д.) задаётся на **уровне мира** или соглашением; уровень стадии оперирует state как структурированным объектом с доступом по имени слота или индексу.

- **Представление state:** словарь (slot_name → value), объект с полями или индексируемая структура. Конкретный формат задаётся реализацией и уровнем мира.
- **Использование на стадии:** стадия получает state на входе, по **state_input_map** собирает из указанных слотов значения для входов воркфлоу-узлов; после выполнения воркфлоу по **state_output_map** записывает выходы воркфлоу в указанные слоты state; возвращает обновлённый state.

### 3.2 Маппинг state → inputs воркфлоу

Для каждого **внешнего входа воркфлоу-узла** задаётся правило, откуда брать значение при run стадии:

- **Из слота state:** например, вход `config` воркфлоу творца ← слот `config` (или индекс 2) state; вход `image` ← слот `artifacts` или вложенное поле.
- **Из контекста:** опционально часть inputs может приходить из **context** (мир, action пользователя, опции), а не только из state.

Конфиг стадии задаёт **state_input_map**: для каждого воркфлоу-узла (workflow_id) и порта — откуда брать значение (slot_name или path в state, или context). При выполнении стадии перед вызовом run(workflow, inputs) словарь inputs собирается по этому правилу из переданного state (и context).

### 3.3 Маппинг outputs воркфлоу → state

После run(workflow, inputs) → outputs значения из outputs записываются в state по правилу **state_output_map**: (workflow_id, port_name) или (port_name при одном воркфлоу) → slot_name или path в state. При нескольких воркфлоу в одной стадии выходы разных воркфлоу могут записываться в разные слоты (или в один по политике слияния). Итоговый state после стадии — результат последовательного применения state_output_map после каждого воркфлоу.

### 3.4 Условие выполнения стадии (can_run)

**Стадия выполняется** (run вызывается) **только если выполнено условие** для данной стадии. Типичное условие — **заполненность определённых слотов state**.

| Тип условия | Описание |
|-------------|----------|
| **required_slots** | Список имён или индексов слотов. run(stage, state) выполняется только если каждый из этих слотов state не пуст (заполнен). |
| **predicate** | Функция can_run(state, context) → bool. Произвольная логика (например, слот `config` имеет поле type = "generation"). |
| **always** | Условие отсутствует или всегда true — стадия выполняется при любом state (например, развитие мира в среде). |

Если условие **не выполнено**, стадия при вызове из мира **пропускается**: state возвращается без изменений (или копия state). Это обеспечивает корректный порядок цикла: стадия автора не запускается без нужных слотов, обновление мира — только при полном state и т.д., по правилам уровня мира.

## 4. Состав стадии: узлы, гиперрёбра, контракт по state

### 4.1 Узлы стадии = воркфлоу

- Каждый **узел стадии** — целый **воркфлоу** ([04_WORKFLOW.md](04_WORKFLOW.md)). У каждого узла есть идентификатор в рамках стадии (**workflow_id** или **stage_node_id**: например, `media_chain`, `narrator`).
- Воркфлоу может быть задан вложенным конфигом (по спецификации 04) или ссылкой (ref на конфиг воркфлоу). При загрузке стадии каждый воркфлоу-узел строится из своего конфига.
- Данные хранятся внутри воркфлоу (в гиперграфах задач и блоках); стадия задаёт только положение воркфлоу в графе стадии и **связь по state** (state_input_map, state_output_map).

### 4.2 Гиперрёбра стадии (между воркфлоу)

- Если узлов-воркфлоу **несколько**, гиперрёбра стадии соединяют внешний выход одного воркфлоу с внешним входом другого — по ним текут **данные**. Порядок выполнения воркфлоу задаётся топологией стадии; данные между ними могут передаваться по рёбрам стадии, не обязательно через state.
- **Внешний** вход стадии (в мир) и **внешний** выход стадии (в мир) — всегда **state**: что из state подаётся на первый(е) воркфлоу, что с последнего(их) воркфлоу пишется в state.

### 4.3 Контракт стадии: state_input_map, state_output_map, condition

| Элемент | Описание |
|---------|----------|
| **state_input_map** | Маппинг «слот state (или path) → (workflow_id, port_name)» для сбора inputs при run. Задаётся по узлам: для какого воркфлоу какие порты берутся из каких слотов state (и опционально из context). |
| **state_output_map** | Маппинг «(workflow_id, port_name) или port_name → слот state» для записи результатов воркфлоу в state после run. |
| **condition** | Предикат can_run(stage, state, context?): например, список обязательных слотов required_slots или функция (state, context) → bool. Если ложно — стадия пропускается (state без изменений). |

Для **вырожденной** стадии (один воркфлоу) state_input_map и state_output_map задают связь «state ↔ этот воркфлоу». Для стадии из нескольких воркфлоу — какие слоты идут на первый(е) воркфлоу, какие выходы какого воркфлоу пишутся в какие слоты (и при необходимости передача по гиперрёбрам стадии между воркфлоу).

## 5. Построение стадии

### 5.1 Добавление узлов (воркфлоу)

- **add_node(workflow_id, workflow)** — добавить воркфлоу как узел стадии с заданным workflow_id. Воркфлоу уже собран и имеет объявленные внешние входы/выходы.
- **add_node(workflow_id, config, registry?)** — построить воркфлоу из конфига (04.from_config) и добавить как узел. Конфиг — полный конфиг воркфлоу или путь к файлу. Дублирование workflow_id при добавлении должно приводить к ошибке.
- **remove_node(workflow_id)** — удалить узел-воркфлоу и связанные гиперрёбра стадии; инвалидирует кэш плана.

### 5.2 Добавление и удаление гиперрёбер стадии

- **add_edge(source_workflow_id, source_port, target_workflow_id, target_port)** — соединить внешний выход воркфлоу source с внешним входом воркфлоу target. Оба порта должны входить в контракт соответствующих воркфлоу.
- **remove_edge(...)** — удалить гиперребро стадии; инвалидирует кэш плана.
- При поддержке гиперрёбер арности > 2: **add_hyperedge(ends)**, **remove_hyperedge(ends)** по соглашению реализации.

### 5.3 Задание контракта по state

- **set_state_input_map(mapping)** — задать, какие слоты state (и при необходимости context) маппятся на какие входы воркфлоу-узлов. Формат: словарь или список правил (slot или path → workflow_id, port_name).
- **set_state_output_map(mapping)** — задать, какие выходы воркфлоу записываются в какие слоты state. Формат: (workflow_id, port_name) или port_name → slot или path.
- **set_condition(required_slots)** или **set_condition(predicate)** — задать условие выполнения стадии (какие слоты должны быть заполнены или произвольный предикат от state/context).

### 5.4 Построение из конфига

- **Конфиг стадии:** список **workflows** (workflow_id, конфиг воркфлоу — вложенный по 04 — или ref); список **edges** (source_workflow, source_port, target_workflow, target_port); **state_input_map**, **state_output_map**; **condition** (required_slots или predicate); stage_id, stage_kind (опционально: author, environment, creator), schema_version, metadata.
- **from_config(config, registry?, validate?)** — построить воркфлоу из конфигов, добавить как узлы; добавить гиперрёбра; установить state_input_map, state_output_map, condition. При validate — валидация стадии (§6).

### 5.5 Версионирование структуры

- При изменении структуры стадии (add_node, remove_node, add_edge, remove_edge, изменение маппингов или condition) инкрементировать **execution_version** для инвалидации кэша плана.

## 6. Валидация стадии

Перед выполнением или сериализацией выполняется **валидация** структуры стадии.

### 6.1 Обязательные проверки

| Проверка | Описание |
|----------|----------|
| **Существование узлов и портов** | Все workflow_id в гиперрёбрах стадии существуют в множестве узлов. Все port_name соответствуют внешним портам соответствующих воркфлоу (exposed_inputs, exposed_outputs). |
| **Совместимость типов на рёбрах** | Тип выхода источника совместим с типом входа приёмника. |
| **Топология** | При отсутствии циклов между воркфлоу — топологическая сортировка возможна. Циклы между воркфлоу внутри одной стадии по соглашению реализации могут поддерживаться (итеративная схема) или не допускаться (DAG). |
| **Маппинг state** | Все ключи в state_input_map и state_output_map ссылаются на существующие workflow_id и порты; слоты state (если задана схема state) существуют. |
| **Condition** | Если condition задан как required_slots, все указанные слоты входят в схему state (при её наличии). |

### 6.2 Дополнительные проверки

| Проверка | Описание |
|----------|----------|
| **Связность** | Узлы, получающие данные из state (граничные), достижимы; от узлов достижимы узлы, выходы которых пишутся в state. |
| **Валидность воркфлоу-узлов** | Опционально вызывать валидацию каждого воркфлоу по правилам 04 §5. |

### 6.3 Результат валидации

- Валидация возвращает список **ошибок** и **предупреждений**. В строгом режиме run не выполняется при наличии ошибок; при from_config(validate=True) невалидная стадия не создаётся или выдаёт исключение.

## 7. Гиперграфовый движок на уровне стадии

**Тот же гиперграфовый движок**, что на уровне гиперграфа задачи и воркфлоу ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §6, [04_WORKFLOW.md](04_WORKFLOW.md) §6), применяется к стадии: узлы интерпретируются как воркфлоу, вызов узла — run(workflow, inputs) → outputs. Отличие — **инициализация буферов** и **сбор результатов**: входы граничных воркфлоу собираются из **state** (по state_input_map), а не только из буферов рёбер; выходы записываются в **state** (по state_output_map), а не только в буферы. Внутри стадии между воркфлоу данные по-прежнему текут по гиперрёбрам через буферы.

### 7.1 Компоненты движка (на уровне стадии)

| Компонент | Назначение |
|-----------|------------|
| **Хранилище структуры** | Множество узлов (workflow_id → воркфлоу), гиперрёбра стадии, state_input_map, state_output_map, condition, execution_version. |
| **Валидатор** | Проверка согласованности стадии (§6). |
| **Планировщик** | План обхода узлов-воркфлоу (топологическая сортировка при DAG или итеративная схема при циклах). |
| **Буферы** | Для гиперрёбер стадии между воркфлоу — данные; для граничных узлов — подстановка из state и запись в state по маппингу. |
| **Исполнитель** | Проверка can_run(stage, state); сбор inputs из state (и context) по state_input_map; выполнение воркфлоу по плану; запись outputs в state по state_output_map; возврат state. |

### 7.2 Жизненный цикл одного run стадии

1. **Вход:** stage, state, context? (опционально: мир, action, опции).
2. **Проверка условия:** can_run(stage, state, context). Если ложно — вернуть state без изменений (или копию).
3. **План:** при необходимости построить/обновить план обхода воркфлоу (топология или итеративная схема).
4. **Инициализация буферов:** для граничных воркфлоу (не имеющих входящих рёбер от других воркфлоу) собрать inputs из state по state_input_map и из context; записать в буферы входящих портов.
5. **Обход по плану:** для каждого воркфлоу в порядке: разрешить входы (из буферов рёбер стадии и из state для граничных); вызвать **run(workflow, inputs)** → outputs; записать outputs в буферы исходящих рёбер стадии и по **state_output_map** — в соответствующие слоты state.
6. **Выход:** вернуть обновлённый state.

---

## 8. Контракт run стадии

### 8.1 Сигнатура и семантика

- **run(stage, state, context?, **options) → state**
- **Вход (state):** контейнер слотов (словарь, объект с полями или индексируемая структура). Слоты читаются по state_input_map для формирования inputs воркфлоу.
- **Вход (context):** опционально — мир, action пользователя, опции run; по соглашению часть inputs воркфлоу может браться из context.
- **Выход (state):** тот же контейнер с обновлёнными слотами согласно state_output_map (или state без изменений, если can_run ложно).
- **Опции:** training, device, seed, callbacks, dry_run и т.д. пробрасываются в run воркфлоу-узлов по соглашению реализации.

### 8.2 Обязательность слотов при run

Все слоты, указанные в **state_input_map** для граничных воркфлоу и необходимые для формирования inputs, должны присутствовать в state (или в context), иначе — ошибка или неопределённое поведение по соглашению реализации. При невыполненном **condition** run может сразу вернуть state без изменений (пропуск стадии).

### 8.3 Вызов из мира

Стадия вызывается из **мира** при обходе цикла: мир передаёт state в run(stage, state, context); стадия возвращает обновлённый state; мир передаёт его следующей стадии. Внутренняя топология стадии (какие воркфлоу, в каком порядке) для мира скрыта; стадия выступает как одна метаморфоза с контрактом по state.

### 8.4 Детерминированность и воспроизводимость

При одних и тех же **конфиге стадии**, **чекпоинтах всех воркфлоу-узлов**, **state** и **context** (и выполнении condition) результат run детерминирован в пределах контрактов воркфлоу. Конфиг стадии + чекпоинты воркфлоу + state + context задают воспроизводимое поведение.

## 9. Сериализация уровня стадии

### 9.1 Конфиг структуры стадии

- **Конфиг стадии** задаёт: **workflows** (workflow_id, конфиг воркфлоу по 04 §8.1 или ref); **edges** стадии; **state_input_map**, **state_output_map**; **condition** (required_slots или ссылка на предикат); **schema_version**, stage_id, опционально stage_kind (author, environment, creator), metadata.
- При загрузке по конфигу каждый воркфлоу строится из своего конфига (04.from_config); восстанавливаются гиперрёбра, маппинги, condition.

### 9.2 Чекпоинт стадии

- Стадия **не имеет своего state_dict** — состояние распределено по воркфлоу-узлам. **Чекпоинт стадии** — совокупность чекпоинтов всех входящих в неё воркфлоу (словарь workflow_id → чекпоинт воркфлоу или директория по workflow_id). По соглашению реализация предоставляет **state_dict()** (агрегат state_dict воркфлоу) и **load_state_dict(state)**.

### 9.3 Полное сохранение и загрузка

- **Сохранить:** конфиг стадии (config.json / config.yaml) + чекпоинты всех воркфлоу (save_dir/workflow_1/, …). **Загрузить:** прочитать конфиг, построить стадию (воркфлоу из конфигов), загрузить чекпоинты в воркфлоу. Возможна загрузка только чекпоинтов в уже собранную стадию (load_checkpoint(checkpoint_dir)).

## 10. Стадия и развёртывание

- **Воркфлоу** остаётся единицей развёртывания: воркфлоу-узлы стадии могут выполняться на разных эндпоинтах; стадия оркестрирует вызовы и передачу state. **API-воркфлоу:** если воркфлоу доступен по API (run(inputs) → outputs), он подключается как узел стадии через тот же контракт; при run стадии inputs собираются из state по state_input_map и передаются на API; ответ маппится в state.
- **to(device)**, **trainable_parameters()**, **set_trainable(workflow_id, bool)** — по соглашению делегируются воркфлоу-узлам, по аналогии с уровнем воркфлоу.

## 11. Граничные случаи и тонкости

### 11.1 Один узел (вырожденная стадия)

- Стадия из одного воркфлоу допустима и типична. План = один шаг; run(stage, state) делегирует сбор inputs из state, run(workflow, inputs), запись outputs в state.

### 11.2 Пропуск стадии (condition ложно)

- Если can_run(stage, state, context) ложно, run возвращает state без изменений. Мир при обходе цикла может проверять condition до вызова run и не вызывать run, передавая state дальше — или стадия сама возвращает state без выполнения воркфлоу.

### 11.3 Пустые или частичные state

- Теоретически стадия может объявлять опциональные слоты (часть inputs из state необязательна). При отсутствии слота в state поведение задаётся контрактом (значение по умолчанию или пропуск) по соглашению реализации.

### 11.4 Повторное использование одного воркфлоу

- Если один и тот же воркфлоу (по ref) используется в стадии несколько раз как разные узлы, реализация может инстанцировать копии или разделять один экземпляр по шагам плана; принцип «один чекпоинт — один экземпляр» применяется по аналогии с 04 §11.7.

### 11.5 Обучение на уровне стадии

- Отдельные воркфлоу-узлы могут быть помечены как trainable; при run(stage, state, training=True) опция пробрасывается в run воркфлоу. Сериализация чекпоинта стадии включает чекпоинты всех воркфлоу ([../../philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md)).

### 11.6 Обработка ошибок

- Философия не предписывает единую политику. Сбой воркфлоу-узла может прерывать run стадии или обрабатываться по политике вызывающего (мир, ретраи, частичный state). Уровень стадии фиксирует контракт run и маппинг state; детали устойчивости — в реализации.

---

**Итог.** Документ задаёт **полную техническую спецификацию уровня стадии**: стадия как гиперграф воркфлоу (узлы = воркфлоу, гиперрёбра = данные между воркфлоу внутри стадии), **внешний контракт по state** (state_input_map, state_output_map, condition can_run), контракт **run(stage, state, context) → state**, применение того же гиперграфового движка с инициализацией из state и записью в state, сериализация, **три стадии мира** (автор, среда, творец) в философском контексте, развёртывание и граничные случаи. Реализация и уровень мира опираются на эту спецификацию.
