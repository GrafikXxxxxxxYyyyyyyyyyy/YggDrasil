# Гиперграфовый движок

**Назначение:** детальное описание **гиперграфового движка** — центрального механизма выполнения системы Иггдрасиль. Документ отвечает на вопросы: для чего движок создан, какие задачи решает, что умеет делать, из каких компонентов состоит, как применяется на уровнях 0.3–0.7 и какие сценарии поддерживает (DAG, циклы, агентные циклы). Канонический источник для понимания и реализации движка.

**Связь с уровнями:** [01_FOUNDATION.md](01_FOUNDATION.md) §5–6 (структура гиперграфа, исполнитель), [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §6–7 (движок на уровне гиперграфа задачи), [04_WORKFLOW.md](04_WORKFLOW.md) §6 (движок на уровне воркфлоу), [05_STAGE.md](05_STAGE.md), [06_WORLD.md](06_WORLD.md) §7, [07_UNIVERSE.md](07_UNIVERSE.md) §9.

**Язык:** русский.

---

## 1. Для чего создан движок

**Гиперграфовый движок** — это **единый механизм выполнения** для всех исполняемых уровней системы (гиперграф задачи, воркфлоу, стадия, мир, вселенная). Он создан для того, чтобы:

1. **Унифицировать выполнение** — на любом уровне структура остаётся «гиперграф узлов»: движок обходит узлы, передаёт данные по гиперрёбрам и вызывает у каждого узла операцию **run**. Меняется только масштаб: узел = блок-задача, гиперграф, воркфлоу, стадия или мир.
2. **Отделить структуру от доменной логики** — движок **не знает**, что делают узлы внутри (диффузия, LLM, агент, генерация текста). Он только обеспечивает порядок обхода, передачу данных по рёбрам и вызов run; вся доменная логика живёт в узлах (блоках).
3. **Обеспечить расширяемость без изменения ядра** — новые типы узлов (новые блоки, новые задачи) добавляются через **реестр** и контракт (порты, run); движок не меняется. Один и тот же код планировщика и исполнителя обслуживает и граф из блоков-задач, и граф из целых миров.
4. **Поддержать сложную топологию** — не только линейные цепочки и DAG, но и **циклы** (итеративные фазы с фиксированным числом шагов), **гиперрёбра произвольной арности** (один выход → много входов, много выходов → один вход) и **агентные циклы** (вызов инструментов до завершения).

Таким образом, движок создан как **общий «рантайм» гиперграфа**: он решает задачу «как выполнить произвольный гиперграф с объявленными входами/выходами и контрактом run», не решая задачу «что именно вычислять в узлах».

## 2. Какие задачи решает движок

| Задача | Описание |
|--------|----------|
| **Определение порядка выполнения** | По структуре графа (узлы + гиперрёбра) построить **план**: в каком порядке вызывать узлы, чтобы к моменту вызова каждого узла все его входы были уже вычислены. Для DAG — топологическая сортировка; для графов с циклами — разбиение на фазы (начальная, циклическая с K итерациями, конечная). |
| **Передача данных между узлами** | Обеспечить, чтобы выход узла A по гиперребру попал на вход узла B в нужный момент. Движок поддерживает **буферы**: после run(A) выходы A записываются в буферы; при run(B) входы B собираются из буферов входящих рёбер и при необходимости из внешних входов графа. |
| **Вызов run узла** | Для каждого шага плана: собрать словарь входов узла из буферов и внешних входов; вызвать `node.run(inputs)` (или эквивалент для данного уровня); записать выходы в буферы исходящих рёбер и при необходимости во внешние выходы графа. |
| **Поддержка циклов** | Цикл (например, Backbone ↔ Inner Module в диффузии) не допускает одной топологической сортировки. Движок строит **итеративный план**: «повторить K раз блок [узел1, узел2, …]»; на каждой итерации буферы обновляются выходами узлов и на следующей итерации подаются снова на вход. Число K задаётся конфигом или опцией run. |
| **Поддержка агентного цикла (agent_loop)** | Если узел — агент (в выходе возможны tool_calls), движок не просто один раз вызывает run(agent), а входит в **подцикл**: вызвать агента → если есть tool_calls, выполнить соответствующие узлы-инструменты, собрать tool_results → снова вызвать агента с inputs + tool_results; повторять до отсутствия tool_calls или до max_steps. |
| **Валидация структуры** | Перед выполнением (или при загрузке) проверить, что граф **реализуем**: все узлы и порты в рёбрах существуют, типы портов совместимы, обязательные входы покрыты, при циклах задана семантика (число итераций, graph_kind). Движок опирается на **валидатор**; при ошибках выполнение не начинается или выдаётся явное исключение. |
| **Кэширование плана** | План выполнения строится по структуре графа и не зависит от конкретных значений входов. Движок **кэширует** план и привязывает его к версии структуры (execution_version); при изменении графа (add/remove node/edge) кэш инвалидируется и план перестраивается при следующем run. |

Итого: движок решает **оркестрационную** задачу — «когда и в каком порядке вызывать узлы и как передавать между ними данные», а не задачу «что вычислять внутри узла».

## 3. Что движок может и чего не может

### 3.1 Может

- **Обходить узлы** в порядке, выведенном из топологии (DAG) или из итеративного плана (циклы).
- **Инициализировать буферы** из внешних входов графа и **заполнять буферы** выходами узлов после каждого run.
- **Разрешать входы узла** — собирать по входящим гиперрёбрам значения из буферов (и из внешних входов для exposed_inputs); при нескольких рёбрах на один порт применять **политику агрегации** (concat, sum, first и т.д., по конфигу).
- **Вызывать run узла** с собранным словарём входов и **записывать выходы** в буферы приёмников и во внешние выходы графа.
- **Поддерживать гиперрёбра арности > 2** — один источник → несколько приёмников (раздача одного значения), несколько источников → один приёмник (агрегация по политике).
- **Строить и кэшировать план** для DAG, для графа с одним или несколькими циклами, для графа с узлом-агентом (agent_loop).
- **Инвалидировать кэш плана** при изменении структуры (execution_version).

### 3.2 Не может (и не должен)

- **Интерпретировать содержимое данных** — движок не знает, что за тензоры, строки или словари передаются по рёбрам; только передаёт их по портам. Семантика и типы — в контракте блоков и в валидаторе.
- **Содержать доменную логику** — никакой «специальной» обработки диффузии, LLM или агента в ядре движка; специфика — в узлах и в плане (итеративная фаза, agent_loop как метка в плане).
- **Гарантировать корректность результатов узлов** — если блок вернул некорректные данные или выбросил исключение, движок передаёт это вызывающему; единая политика ретраев/отката не задаётся движком, а может быть в обвязке run или в контракте блоков.
- **Хранить состояние между вызовами run** — буферы живут только в рамках одного run; между вызовами сохраняется только состояние внутри узлов (блоков).

## 4. Компоненты движка

| Компонент | Назначение |
|-----------|------------|
| **Хранилище структуры** | Множество узлов (идентификатор → узел-задача, т.е. объект Block+Node, или на верхних уровнях — вложенный гиперграф/воркфлоу/стадия/мир), множество гиперрёбер, списки внешних входов/выходов (exposed_inputs, exposed_outputs), опционально execution_version для инвалидации кэша плана. На уровнях стадии/мир/вселенная сюда же входят state_schema, цикл, маппинги state и т.д. |
| **Валидатор** | Проверка согласованности структуры перед выполнением: существование узлов и портов в рёбрах, совместимость типов, обязательные порты, при циклах — распознаваемость и заданность числа итераций. Возвращает список ошибок и предупреждений; в строгом режиме run не выполняется при ошибках. |
| **Планировщик (execution planner)** | Строит **план выполнения**: порядок вызовов узлов. Для DAG — топологическая сортировка. Для графа с циклами — разбиение на фазы: начальная (узлы до цикла), циклическая (повторить K раз заданный подграф), конечная (узлы после цикла). При наличии узла-агента вставляет в план метку agent_loop. План кэшируется и привязан к execution_version. |
| **Буферы (edge buffers)** | Во время одного run каждому приёмнику (target_node_id, target_port) сопоставляется буфер значения. Инициализация: внешние входы графа пишутся в буферы соответствующих портов. После run узла: выходы узла записываются в буферы всех приёмников по исходящим гиперрёбрам. При нескольких входящих рёбрах на один порт — агрегация по политике. Внешние выходы графа заполняются из буферов по exposed_outputs по окончании run. Буферы не сохраняются между вызовами run. |
| **Исполнитель (executor)** | Выполняет план шаг за шагом: для каждого шага (узел или agent_loop) разрешить входы узла из буферов и внешних входов; вызвать `node.run(inputs)`; записать выходы в буферы и во внешние выходы. Для agent_loop — внутренний подцикл вызова агента и инструментов до завершения. |

Эти пять компонентов присутствуют на всех уровнях, где применяется движок; на уровнях стадии, мира и вселенной «узлом» является целая стадия, мир и т.д., а по рёбрам течёт state или обмен (payload), но принцип тот же: план → буферы → обход → вызов run.

## 5. Применение движка по уровням

Один и тот же **принцип** движка (структура, валидация, план, буферы, обход, вызов run) применяется на каждом исполняемом уровне. Меняется только **что является узлом** и **что течёт по рёбрам**.

| Уровень | Узлы | Что течёт по рёбрам | Особенности |
|---------|------|---------------------|-------------|
| **Гиперграф задачи (0.3)** | Экземпляры узлов-задач (блоки) | **Данные** (тензоры, словари, сообщения) между портами | Классический случай: планировщик строит порядок или итеративный план (цикл Backbone ↔ Inner Module), agent_loop для узла-агента. Подробно: [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §6–7. |
| **Воркфлоу (0.4)** | Целые гиперграфы задач | **Данные** между внешними входами/выходами гиперграфов | Тот же движок: узлы = гиперграфы, у каждого свой run(inputs)→outputs. Поддержка циклов между гиперграфами — итеративная схема на уровне воркфлоу. [04_WORKFLOW.md](04_WORKFLOW.md) §6. |
| **Стадия (0.5)** | Воркфлоу | Внутри стадии: **данные** между воркфлоу; вход/выход стадии: **state** | План — порядок воркфлоу по топологии стадии; маппинг state → inputs воркфлоу и outputs → state задаётся state_input_map, state_output_map. Условие can_run перед выполнением стадии. [05_STAGE.md](05_STAGE.md). |
| **Мир (0.6)** | Стадии | **State** (контейнер слотов) по циклу автор → среда → творец | План — фиксированный цикл стадий; для каждой стадии проверка can_run, затем run(stage, state) → state. Буферы по сути — передача state от стадии к стадии. [06_WORLD.md](06_WORLD.md) §7. |
| **Вселенная (0.7)** | Миры | **Обмен** (payload_spec: state snapshot, артефакты, сущности и т.д.) | План — порядок обхода миров (топология или заданный список); буферы — данные по payload_spec после run(world) для входящих рёбер следующих миров. [07_UNIVERSE.md](07_UNIVERSE.md) §9. |

Во всех случаях движок **не привязан к типу узла**: узел рассматривается как чёрный ящик с входами, выходами и run. Это позволяет масштабировать одну и ту же модель от графа блоков до графа миров без замены ядра выполнения.

## 6. Сценарии, которые движок поддерживает

| Сценарий | Описание |
|----------|----------|
| **Линейная цепочка (DAG)** | Узлы A → B → C без циклов. План — топологический порядок [A, B, C]. Один проход, буферы заполняются по мере выполнения. |
| **Ветвление** | Один выход узла A раздаётся по нескольким гиперрёбрам в узлы B и C. Движок нативно поддерживает гиперрёбра с несколькими приёмниками; один вызов run(A), запись выхода в буферы (B, port) и (C, port). |
| **Слияние входов** | Несколько узлов A, B пишут в один вход узла C (несколько гиперрёбер на один порт). Движок собирает значения по политике агрегации (concat, sum, first и т.д.) и передаёт в C. |
| **Один итеративный цикл** | Классическая схема диффузии: начальная фаза (Conjector, Outer Module, encode) → циклическая фаза «K раз [Backbone, Inner Module]» → конечная фаза (Converter decode). Планировщик распознаёт цикл и строит такой план; исполнитель на каждой итерации обновляет буферы и снова вызывает Backbone и Inner Module. |
| **Несколько циклов** | В одном графе может быть несколько циклических подграфов (например, вложенные или параллельные). Планировщик строит план с несколькими циклическими фазами или с явным порядком итераций; детали задаются конфигом и анализом графа. |
| **Циклы на уровне воркфлоу** | Гиперграфы G1 ↔ G2 связаны рёбрами в обе стороны. План воркфлоу — итеративная схема (например, K раз [G1, G2]); опция num_loop_steps задаёт K. |
| **Агентный цикл (agent_loop)** | Узел-агент возвращает tool_calls; движок выполняет узлы-инструменты по таблице tool_id → node_id, собирает tool_results, снова вызывает агента; повторяет до отсутствия tool_calls или до max_steps. План содержит метку «agent_loop для node_id». |
| **Один узел** | Вырожденный случай: граф из одного узла. План = один шаг; внешние входы/выходы — порты этого узла. Run графа = один вызов run узла. |

## 7. Рекурсия и вложенность

Гиперграфовый движок поддерживает **рекурсию и произвольную вложенность** в двух смыслах: **структурная вложенность** (узел как целый граф) и **рекурсия выполнения** (повторный вызов одного и того же узла/графа до условия останова). Оба случая реализуются в рамках того же контракта run и той же семантики обхода; движок не вводит отдельного «режима рекурсии».

### 7.1 Структурная вложенность

**Масштаб в глубину:** блок может содержать подблоки; граф задачи может быть узлом воркфлоу; воркфлоу — узлом стадии; стадия — узлом мира; мир — узлом вселенной. На каждом уровне движок видит **узлы** и **гиперрёбра**; узел для движка — чёрный ящик с операцией `run(inputs) → outputs`. Если внутри узла — вложенный гиперграф (например, гиперграф задачи внутри узла воркфлоу), выполнение делегируется **тому же примитиву выполнения**: планировщик строит план верхнего уровня, на шаге «вызвать узел N» исполнитель вызывает `node.run(inputs)`, а реализация узла внутри себя при необходимости вызывает движок для вложенного графа. Тем самым **блок внутри блока, граф внутри графа** — поддерживаемые варианты без изменения ядра движка. Онтология допускает произвольную глубину композиции; ограничение — только ресурсы и соглашения реализации (например, максимальная глубина вложенности для отладки).

**Единообразие:** на уровне гиперграфа задачи узел = блок; на уровне воркфлоу узел = гиперграф задачи; на уровне стадии узел = воркфлоу и т.д. Движок на каждом уровне не знает, «простой» это узел или «составной» (внутри — ещё один граф); он только вызывает run и передаёт данные по рёбрам. Рекурсия по структуре сводится к цепочке вызовов run на разных уровнях.

### 7.2 Рекурсия выполнения

**Повторный вызов одного и того же узла/подграфа** с обновлением данных до выполнения условия — второй аспект рекурсии. Примеры:

- **Итеративная фаза цикла:** в графе задачи цикл Backbone ↔ Inner Module выполняется K раз; на каждой итерации одни и те же узлы вызываются снова с новыми данными из буферов. Планировщик строит план вида «повторить K раз [Backbone, Inner Module]»; исполнитель на каждой итерации обновляет буферы и снова вызывает те же узлы. Это **рекурсия выполнения** в рамках одного run гиперграфа.
- **Циклы на уровне воркфлоу:** два гиперграфа G1 и G2 связаны рёбрами в обе стороны; план — «K раз [G1, G2]». Каждый «шаг» — полный run G1 и run G2; данные между итерациями передаются по гиперрёбрам через буферы. Тот же принцип: повторный вызов тех же узлов (здесь узлы = гиперграфы) с обновлёнными входами.
- **Агентный цикл (agent_loop):** один и тот же узел-агент вызывается многократно до тех пор, пока в выходе не исчезнут tool_calls или не будет достигнут max_steps. Между вызовами выполняются узлы-инструменты, их выходы подаются агенту как tool_results. Это тоже рекурсия выполнения: один узел вызывается рекурсивно по данным.

**Условие останова** задаётся конфигом или опциями run: фиксированное число шагов (num_loop_steps, max_steps), сходимость (если блок возвращает флаг), отсутствие tool_calls в ответе агента. Движок не интерпретирует семантику «сходимости» — он лишь повторяет фазу или вызов узла согласно плану; критерий останова реализуется либо в плане (фиксированное K), либо в логике узла/обвязки (агент возвращает «готово»).

### 7.3 Итог по рекурсии и вложенности

- **Структурная вложенность:** граф внутри узла выполняется тем же движком при вызове run узла; глубина не ограничена онтологией.
- **Рекурсия выполнения:** один и тот же узел или подграф вызывается многократно в рамках одного run; план (итеративная фаза, agent_loop) задаёт число и порядок повторов; буферы переносят данные между итерациями.
- Единый контракт run и единая семантика обхода; никакой отдельной онтологии «рекурсии» — только композиция вызовов и план с циклическими фазами.

*Философский якорь: [../../philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md) §3.2 (масштаб в глубину, рекурсия выполнения).*

## 8. Жизненный цикл одного run (сводка)

1. **Вход:** граф (гиперграф задачи, воркфлоу, стадия, мир или вселенная), входные данные (inputs / state / world_inputs), опции (num_loop_steps, max_steps, seed и т.д.).
2. **Валидация (опционально):** при первом run или по опции — вызов валидатора; при наличии ошибок run не выполняется.
3. **План:** если кэш плана устарел (execution_version изменился) — планировщик строит новый план и кэширует его.
4. **Инициализация буферов:** внешние входы графа записываются в буферы соответствующих портов (и при необходимости в слоты state на уровнях стадии/мир).
5. **Обход по плану:** для каждого шага плана разрешить входы узла из буферов (и внешних входов); вызвать run узла; записать выходы в буферы (и во внешние выходы). Для agent_loop — внутренний подцикл до завершения.
6. **Сбор выходов:** по списку внешних выходов графа собрать значения из буферов в результирующий словарь (outputs / state / results).
7. **Выход:** вернуть результат вызывающему.

Доменная логика (что именно делают блоки, воркфлоу, стадии, миры) не входит в этот цикл — только структура, план, буферы и вызов run.

## 9. Связь с философией и каноном

По замыслу проекта ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md)) **переход** между материальным (блок, содержание) и идеальным (узел, форма) реализован в виде правила «блок → узел» и **гиперграфа**. Гиперграфовый движок — это механизм, который **приводит гиперграф в исполнение**: он не добавляет новой онтологии, а реализует уже заданную структуру (узлы, рёбра, порты) в виде порядка вызовов и передачи данных. Единая модель на всех уровнях (0.3–0.7) отражает принцип **единства онтологии**: один и тот же способ представления и выполнения — гиперграф и его движок — от задачи до вселенной.

---

**Итог.** Документ задаёт **назначение, задачи, возможности и границы гиперграфового движка**: для чего он создан (унификация выполнения, отделение структуры от доменной логики, расширяемость, сложная топология); какие задачи решает (порядок выполнения, передача данных, вызов run, циклы, agent_loop, валидация, кэш плана); что может и чего не может; компоненты (хранилище, валидатор, планировщик, буферы, исполнитель); применение по уровням 0.3–0.7; поддерживаемые сценарии; **рекурсия и вложенность** (структурная вложенность и рекурсия выполнения); жизненный цикл run. Движок — общий рантайм гиперграфа без доменной логики в ядре.
