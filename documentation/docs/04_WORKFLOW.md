# 04. Уровень воркфлоу

**Назначение:** полная **техническая** документация **второго исполняемого уровня** системы Иггдрасиль — воркфлоу. Документ задаёт воркфлоу как **гиперграф гиперграфов**: узлы воркфлоу — **целые гиперграфы задач**, по гиперрёбрам воркфлоу передаются **данные** между внешними выходами одного гиперграфа и внешними входами другого. Описаны состав воркфлоу (узлы-гиперграфы, гиперрёбра, внешние входы/выходы воркфлоу), гиперграф как узел и его контракт, построение и валидация, **применение того же гиперграфового движка** к уровню воркфлоу, контракт run, сериализация, типичные сценарии, вырожденный случай (воркфлоу из одного гиперграфа), граничные случаи и связь со стадией. Канонический источник технических решений уровня воркфлоу.

**Предыдущий уровень:** [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) — гиперграф задачи (узлы = экземпляры узлов-задач, внешние входы/выходы, движок, run).

**Философский якорь:** [../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §4 — уровень воркфлоу как гиперграф гиперграфов; комбинированные сценарии; данные между задачами; второй исполняемый уровень.

**Язык:** русский.

## 1. Место уровня в иерархии

| Уровень | Сущность | Описание |
|---------|----------|----------|
| **Гиперграф задачи** | Гиперграф, узлы которого — экземпляры узлов-задач | Один гиперграф = **одна цельная задача**; по гиперрёбрам — данные между портами. [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md). |
| **Воркфлоу** | **Воркфлоу** — гиперграф, узлы которого — **гиперграфы задач** | **Второй исполняемый уровень:** комбинированные сценарии — цепочка или ветвление из нескольких задач; по гиперрёбрам воркфлоу передаются **данные между задачами**. |
| **Стадия** | Гиперграф воркфлоу (узлы = воркфлоу) | Одна стадия = одна метаморфоза; по рёбрам передаётся **state**. [05_STAGE.md](05_STAGE.md). |

Воркфлоу **не разбивает** одну задачу на части: текст, диффузия, декодирование остаются внутри одного гиперграфа задачи. Воркфлоу **соединяет готовые задачи** (каждая = один гиперграф) в сложный сценарий: текст → картинка → апскейл → видео и т.д. ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §4).

## 2. Воркфлоу — определение и смысл

### 2.1 Определение

**Воркфлоу** — это **гиперграф, узлами которого являются целые гиперграфы задач**. С точки зрения движка фреймворка воркфлоу — **тот же гиперграф** со всеми теми же возможностями, что и гиперграф задачи: та же структура (узлы, гиперрёбра, внешние входы/выходы), тот же принцип планирования (топологический порядок для DAG **или итеративная схема при циклах**), та же семантика run (собрать входы по рёбрам, вызвать run узла, раздать выходы по рёбрам). **Циклы между гиперграфами на уровне воркфлоу допустимы и поддерживаются** — как и циклы между узлами на уровне гиперграфа задачи. Меняется только **содержимое узла**: узел воркфлоу — не блок, а **гиперграф**, который выполняется вызовом run(hypergraph, inputs) → outputs. Гиперграфовый движок ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §6) применяется на уровне воркфлоу **без смены модели**: узлы = гиперграфы; по рёбрам = данные между внешними выходами одного гиперграфа и внешними входами другого.

- **Узлы воркфлоу** — целые гиперграфы (каждый гиперграф = одна задача: text-to-image, upscale, image-to-video, агент с инструментами и т.д.). Узлов может быть **один или несколько**.
- **Гиперрёбра воркфлоу** — связи между **внешним выходом** одного гиперграфа и **внешним входом** другого (или с внешним входом/выходом самого воркфлоу). Формат: (source_graph_id, source_port_name) → (target_graph_id, target_port_name), где порты — внешние порты соответствующих гиперграфов (exposed_outputs источника, exposed_inputs приёмника).
- **Выполнение воркфлоу** — в порядке, заданном планом (топологический порядок при отсутствии циклов; при наличии циклов между гиперграфами — итеративная схема с повторением циклической фазы K раз), выполнять **целые гиперграфы**; на каждом шаге вызывать run(hypergraph, inputs) для гиперграфа-узла, где inputs собираются из буферов входящих рёбер воркфлоу (и из внешних входов воркфлоу); выходы гиперграфа записываются в буферы исходящих рёбер воркфлоу (и во внешние выходы воркфлоу).

### 2.2 Граница с гиперграфом задачи

Один **гиперграф задачи** реализует **одну законченную цель** (одну задачу). **Воркфлоу** реализует **комбинированный сценарий** из нескольких таких целей. Разбивать одну задачу на несколько гиперграфов только ради «удобства» не требуется — внутри одного гиперграфа умещаются и цикл деноизинга, и кондиционирование, и декодирование. Воркфлоу нужен, когда пользователь или разработчик хочет **собрать цепочку или ветвление** из уже готовых задач ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §2.3).

### 2.3 Роль воркфлоу для пользователей и разработчиков

По философии проекта ([../../philosophy/Philosophy-Part2.md](../../philosophy/Philosophy-Part2.md) §4) уровень воркфлоу — **важнейший уровень для пользователей и разработчиков**: на нём раскрывается разнообразие форм и смыслов. Воркфлоу сочетает идеи **масштабирования и рекурсии** и даёт **максимальную гибкость** — возможность, как из конструктора, собирать сложные сценарии из готовых задач или строить собственные задачи и собирать из них сценарии. В известном смысле воркфлоу — **песочница** для любых комбинированных решений и уровень **бесконечного разнообразия форм и смыслов**. Техническая спецификация ниже обеспечивает единый контракт и предсказуемость при этой гибкости.

### 2.4 Примеры сценариев, реализуемых воркфлоу

| Сценарий | Узлы воркфлоу (гиперграфы) | Поток данных |
|----------|----------------------------|---------------|
| **Текст → видео** | text-to-image → upscale → image-to-video | text → image → image (upscaled) → video |
| **Аутпейнтинг → апскейл → стилизация → видео** | Четыре гиперграфа | Цепочка по портам image / condition |
| **RAG + генерация + постобработка** | Поиск (Helper-гиперграф) → LLM-генерация → форматтер | context → response → formatted |
| **Одна задача под единым интерфейсом** | Один гиперграф (вырожденный воркфлоу) | Входы/выходы воркфлоу = входы/выходы этого гиперграфа |

## 3. Состав воркфлоу: узлы, гиперрёбра, внешние входы/выходы

### 3.1 Узлы воркфлоу = гиперграфы задач

- Каждый **узел воркфлоу** — целый **гиперграф** одной задачи. У каждого такого узла есть идентификатор в рамках воркфлоу (**graph_id** или **node_id**: например, `text2img`, `upscale`, `img2video`), чтобы задавать рёбра между выходами и входами гиперграфов.
- Гиперграф может быть задан **в конфиге воркфлоу** (вложенный конфиг: nodes, edges, exposed_inputs, exposed_outputs по спецификации [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md)) или **ссылкой** (путь к конфигу графа, идентификатор в реестре графов). При загрузке воркфлоу каждый гиперграф-узел строится из своего конфига (через реестр блоков и контракт уровня гиперграфа).
- Данные хранятся **внутри гиперграфа** (в блоках узлов-задач); воркфлоу задаёт только **положение** гиперграфа в графе и **связи** между внешними портами гиперграфов. Правило «блок → узел» на уровне задачи сохраняется; на уровне воркфлоу **гиперграф целиком** выступает как «блок» с контрактом run(inputs) → outputs, реализованным через внутренний движок гиперграфа.

### 3.2 Гиперграф как узел: контракт и чёрный ящик

Каждый гиперграф задачи объявляет **внешние входы и выходы** (exposed_inputs, exposed_outputs) — [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §3.3, §4.3. Это и есть **контракт задачи** для воркфлоу: имена и типы входов/выходов, по которым гиперграф вызывается извне.

- **Входы гиперграфа** — то, что нужно задать для задачи (текст для text-to-image, изображение для upscale или image-to-video, prompt для агента и т.д.). При вызове run(hypergraph, inputs) значения из словаря inputs подаются на эти порты.
- **Выходы гиперграфа** — результат задачи (изображение, видео, латенты, response агента и т.д.). run возвращает словарь outputs с этих портов.

Воркфлоу **соединяет** выход одного гиперграфа с входом другого по этим портам; внутреннее устройство гиперграфа (узлы-задачи, рёбра, агентный цикл внутри) для воркфлоу **не важно** — только контракт внешних портов.

### 3.3 Гиперрёбра воркфлоу

- **Гиперребро воркфлоу** соединяет **внешний выход одного гиперграфа** с **внешним входом другого** (или с внешним выходом воркфлоу). Формат ребра размера 2: (source_graph_id, source_port_name) → (target_graph_id, target_port_name). Типы и семантика портов должны совпадать: то, что один гиперграф выдаёт по данному порту, подходит как вход другому по указанному порту.
- Движок может поддерживать гиперрёбра воркфлоу **арности > 2** по той же схеме, что на уровне задачи: один источник → несколько приёмников (раздача выхода одного гиперграфа на несколько входов других) или **несколько источников → один приёмник** (агрегация). При нескольких входящих гиперрёбрах воркфлоу на **один внешний вход** гиперграфа-приёмника требуется **политика агрегации** (concat, sum, first и т.д.), задаваемая конфигом ребра, порта или гиперграфа-приёмника — по аналогии с уровнем гиперграфа задачи ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md), [01_FOUNDATION.md](01_FOUNDATION.md) §5). Конкретное кодирование задаётся реализацией.
- По гиперрёбрам воркфлоу передаются **данные между задачами** (тензоры, словари, сообщения и т.д.) — те же типы, что и на внешних портах гиперграфов.

### 3.4 Внешние входы и выходы воркфлоу

- **Внешние входы воркфлоу (exposed inputs):** порты гиперграфов (или одного гиперграфа), не имеющие входящих рёбер от других гиперграфов — они получают данные извне при вызове run(workflow, inputs). Задаются списком (graph_id, port_name, name?) по аналогии с [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §4.3.
- **Внешние выходы воркфлоу (exposed outputs):** порты гиперграфов, с которых снимается итоговый результат воркфлоу. Задаются списком (graph_id, port_name, name?). При run(workflow, inputs) **ключи словаря inputs** — name (если задан при expose_input) или канонический идентификатор **(graph_id, port_name)** или строка вида `graph_id:port_name` по соглашению реализации; **ключи словаря outputs** — name или канонический идентификатор порта. Контракт узла-гиперграфа для движка воркфлоу — перечень внешних входов и выходов (exposed_inputs/exposed_outputs гиперграфа или get_input_spec/get_output_spec по соглашению реализации).

Воркфлоу вызывается как **чёрный ящик**: снаружи передаются данные на внешние входы воркфлоу и снимаются данные с внешних выходов; внутренняя топология (какие гиперграфы, в каком порядке) скрыта за контрактом воркфлоу. По соглашению реализации воркфлоу может предоставлять **get_input_spec()** / **get_output_spec()** (список внешних портов в формате, совместимом с вызовом run) — для единообразия с гиперграфом задачи и для использования на уровне стадии: стадия тогда работает с воркфлоу так же, как с гиперграфом (те же ключи inputs/outputs).

## 4. Построение воркфлоу

### 4.1 Добавление узлов (гиперграфов)

- **add_node(node_id, hypergraph)** — добавить гиперграф как узел воркфлоу с заданным node_id (или graph_id). Гиперграф уже собран и имеет объявленные exposed_inputs и exposed_outputs.
- **add_node(node_id, config, registry?)** — построить гиперграф из конфига (по спецификации 03: from_config для гиперграфа) и добавить как узел. Конфиг — полный конфиг гиперграфа (nodes, edges, exposed_inputs, exposed_outputs) или **путь к файлу** конфига (YAML/JSON) по соглашению реализации. Дублирование node_id при добавлении узла должно приводить к ошибке (ValueError или аналог).
- **remove_node(node_id)** — удалить узел-гиперграф и связанные с ним гиперрёбра воркфлоу; инвалидирует кэш плана (execution_version). При загрузке из конфига удаление обычно не используется.
- Узлов может быть **один** (вырожденный воркфлоу) или несколько.

### 4.2 Добавление и удаление гиперрёбер воркфлоу

- **add_edge(source_graph_id, source_port, target_graph_id, target_port)** — соединить внешний выход гиперграфа source_graph_id (порт source_port) с внешним входом гиперграфа target_graph_id (порт target_port). Оба гиперграфа должны быть узлами воркфлоу; оба порта должны входить в exposed_outputs источника и exposed_inputs приёмника соответственно. Дублирование ребра (один и тот же источник и приёмник) по соглашению реализации может игнорироваться (no-op) или приводить к ошибке валидации.
- **add_hyperedge(ends)** — при поддержке гиперрёбер арности > 2: добавить гиперребро воркфлоу; `ends` — список пар (graph_id, port_name) с разметкой источника/приёмника по соглашению реализации. Типы портов на концах должны быть согласованы.
- **remove_edge(...)** / **remove_hyperedge(ends)** — удалить ребро размера 2 или гиперребро по списку концов; инвалидирует кэш плана.
- Для связи с **внешним миром** воркфлоу: внешние входы воркфлоу — порты гиперграфов, на которые не ведут рёбра от других гиперграфов (они заполняются из словаря inputs при run). Внешние выходы воркфлоу — порты, объявленные как выходы воркфлоу (значения с них попадают в словарь outputs при run).

### 4.3 Объявление внешних входов/выходов воркфлоу

- **expose_input(graph_id, port_name, name?)** — пометить порт гиперграфа graph_id как внешний вход воркфлоу (данные приходят из inputs при вызове run(workflow, inputs)).
- **expose_output(graph_id, port_name, name?)** — пометить порт гиперграфа graph_id как внешний выход воркфлоу (значение попадает в outputs).
- Обычно внешние входы воркфлоу — входы «первого» по топологии гиперграфа(ов); внешние выходы — выходы «последнего» гиперграфа(ов). При ветвлении может быть несколько входов/выходов.

### 4.4 Построение из конфига

- **Конфиг воркфлоу:** список **graphs** (каждый элемент — конфиг гиперграфа или ссылка на конфиг: node_id/graph_id, config или ref), список **edges** воркфлоу ({source_graph, source_port, target_graph, target_port}; при арности > 2 — список концов с разметкой), **exposed_inputs**, **exposed_outputs** воркфлоу (списки {graph_id, port_name, name?}). Опционально: workflow_id, **workflow_kind**, metadata, schema_version. **workflow_kind** — тег конфига (например, `"chain"`, `"dag"`, `"cyclic"`) для подсказки планировщику и валидатору; не заменяет анализ структуры (топология, циклы), но может ускорить построение плана и проверки — по аналогии с graph_kind на уровне гиперграфа ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §4.4).
- **from_config(config, registry?, validate?)** — для каждого элемента graphs построить гиперграф (из вложенного config или по ref), добавить как узел; добавить гиперрёбра воркфлоу; восстановить exposed_inputs/exposed_outputs воркфлоу. При validate=True выполнить валидацию воркфлоу (§5).
- **infer_exposed_ports()** (опционально) — вывести внешние входы/выходы воркфлоу по правилу: внешний вход = порт гиперграфа, не имеющий входящего гиперребра воркфлоу от другого гиперграфа; внешний выход = порт гиперграфа, не имеющий исходящего гиперребра к другому гиперграфу. Используется, если пользователь не задал expose_input/expose_output явно; при явном объявлении не перезаписывает их.

### 4.5 Версионирование структуры

- При любом изменении структуры воркфлоу (add_node, remove_node, add_edge, remove_edge, изменение exposed_inputs/exposed_outputs) рекомендуется инкрементировать **execution_version** воркфлоу. Это используется движком для инвалидации кэша плана выполнения: при следующем run планировщик перестраивает план, если версия изменилась (§6).

## 5. Валидация воркфлоу

Перед выполнением или сериализацией выполняется **валидация** структуры воркфлоу. Валидатор проверяет, что воркфлоу **реализуем**.

### 5.1 Обязательные проверки

| Проверка | Описание |
|----------|----------|
| **Существование узлов и портов** | Все graph_id в гиперрёбрах воркфлоу и в exposed_inputs/exposed_outputs воркфлоу существуют в множестве узлов воркфлоу. Все port_name соответствуют **внешним** портам соответствующих гиперграфов (exposed_inputs, exposed_outputs уровня гиперграфа). |
| **Совместимость типов на рёбрах** | Тип данных внешнего выхода источника (по контракту гиперграфа) совместим с типом внешнего входа приёмника. |
| **Топология и циклы** | При отсутствии циклов между гиперграфами — топологическая сортировка. **Циклы между гиперграфами допустимы**: планировщик строит итеративную схему (начальная фаза → циклическая фаза K раз → конечная фаза), число K задаётся конфигом или опцией run (num_loop_steps). Валидатор может предупреждать о циклах и проверять, что цикл помечен или распознаваем (workflow_kind, num_loop_steps). Циклы **внутри** гиперграфа-узла обрабатываются движком гиперграфа при вызове run(hypergraph, …). |
| **Внешние входы/выходы** | Каждый объявленный внешний вход воркфлоу — порт какого-то гиперграфа; каждый внешний выход — порт какого-то гиперграфа. Обязательные входы гиперграфа-приёмника должны иметь входящее гиперребро воркфлоу или быть во внешних входах воркфлоу; **опциональные** входы гиперграфа (по контракту уровня 03) могут оставаться без ребра — при run гиперграф использует значение по умолчанию. |

### 5.2 Дополнительные проверки

| Проверка | Описание |
|----------|----------|
| **Связность (опционально)** | Узлы, участвующие во внешних входах воркфлоу, достижимы; от узлов достижимы узлы внешних выходов (если воркфлоу не допускает «висячих» гиперграфов). |
| **Валидность гиперграфов-узлов** | Каждый гиперграф-узел может быть провалидирован по правилам уровня гиперграфа задачи ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §5); опционально воркфлоу-валидатор вызывает валидацию вложенных гиперграфов. |

### 5.3 Результат валидации

- Валидация возвращает список **ошибок** (блокирующие выполнение) и **предупреждений**. В строгом режиме run не выполняется при наличии ошибок; при загрузке из конфига с validate=True невалидный воркфлоу не создаётся или выдаёт явное исключение.

## 6. Гиперграфовый движок на уровне воркфлоу

**Тот же гиперграфовый движок**, что и на уровне гиперграфа задачи ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §6), применяется к воркфлоу **без изменений ядра**: узлы интерпретируются как гиперграфы, вызов узла — это вызов run(hypergraph, inputs) → outputs. Движок не содержит доменной логики; только структура, порядок обхода, буферы и вызов run.

### 6.1 Компоненты движка (на уровне воркфлоу)

| Компонент | Назначение |
|-----------|------------|
| **Хранилище структуры** | Множество узлов (graph_id → гиперграф), множество гиперрёбер воркфлоу, списки exposed_inputs и exposed_outputs воркфлоу. Версия структуры (execution_version) для инвалидации кэша плана. |
| **Валидатор** | Проверка согласованности структуры воркфлоу (§5). Вызывается до первого run или при from_config(validate=True). |
| **Планировщик** | Строит **план выполнения**: порядок обхода узлов-гиперграфов. **Без циклов между гиперграфами** — топологическая сортировка по направлению гиперрёбер воркфлоу. **При наличии циклов** между гиперграфами — итеративная схема (начальная фаза → повторять K раз циклическую фазу [гиперграфы в цикле] → конечная фаза), аналогично [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §6.2, §7.2. План кэшируется и привязан к execution_version. Циклы **между** гиперграфами обрабатываются итеративной схемой плана (повтор циклической фазы K раз); циклы **внутри** гиперграфа — движком гиперграфа при вызове run(hypergraph, …). |
| **Буферы (edge buffers)** | Во время run(workflow, inputs) каждому приёмнику (target_graph_id, target_port) сопоставляется **буфер значения**: результат с (source_graph_id, source_port). Внешние входы воркфлоу инициализируются из словаря inputs; по мере выполнения гиперграфов их выходы записываются в буферы исходящих рёбер воркфлоу. |
| **Исполнитель** | По плану выполнения для каждого узла-гиперграфа: (1) разрешить входы гиперграфа (собрать из буферов входящих гиперрёбер воркфлоу и внешних входов воркфлоу), (2) вызвать **run(hypergraph, inputs)** → outputs (внутри гиперграфа выполняется свой движок — циклы, agent_loop и т.д.), (3) записать выходы в буферы исходящих гиперрёбер воркфлоу и при необходимости во внешние выходы воркфлоу. |

### 6.2 Планирование выполнения

- **Цель плана:** определить последовательность вызовов узлов-гиперграфов так, чтобы к моменту вызова гиперграфа все его входящие гиперрёбра воркфлоу были заполнены (результатами предыдущих гиперграфов или внешними входами воркфлоу).
- **Воркфлоу без циклов (DAG):** при отсутствии циклов между гиперграфами достаточно **топологической сортировки** (Kahn или DFS) по направлению гиперрёбер воркфлоу. План = этот порядок; выполнение — один проход по узлам-гиперграфам. Узлы одной «волны» (между которыми нет рёбер) могут быть упорядочены неединственным образом; для **воспроизводимости** реализация должна задавать стабильный порядок (например, лексикографически по graph_id), чтобы при одних и тех же конфиге и входе порядок вызовов не менялся.
- **Воркфлоу с циклом между гиперграфами:** при наличии цикла (например, гиперграф A → гиперграф B → гиперграф A) план задаётся как **итеративная схема**: (1) **начальная фаза** — узлы-гиперграфы, не входящие в цикл; (2) **циклическая фаза** — повторять K раз обход гиперграфов, входящих в цикл (порядок внутри итерации задаётся рёбрами); (3) **конечная фаза** — узлы после цикла. Число K задаётся конфигом (num_workflow_steps, num_loop_steps) или опцией run. Планировщик распознаёт циклы (анализ сильно связных компонент или явная разметка workflow_kind) и формирует такой план — по той же логике, что на уровне гиперграфа задачи ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §6.2, §7.2).
- Кэширование и инвалидация плана — по той же схеме, что на уровне гиперграфа задачи: execution_version, при изменении структуры план перестраивается.

### 6.3 Буферы и разрешение входов узла-гиперграфа

- **Инициализация:** для каждого внешнего входа воркфлоу (graph_id, port_name) записать в буфер значение из inputs.
- **Разрешение входов гиперграфа:** для узла-гиперграфа G и его внешнего входного порта P собрать все гиперрёбра воркфлоу, у которых (G, P) — приёмник; взять значения из буферов (и если (G, P) в exposed_inputs воркфлоу — добавить из inputs). При **нескольких входящих рёбрах** на один порт применить **политику агрегации** (concat, sum, first и т.д.), заданную конфигом. Сформировать словарь inputs для run(hypergraph, inputs).
- **Запись выходов:** после run(hypergraph, inputs) → outputs для каждого внешнего выходного порта P гиперграфа и каждого исходящего гиперребра воркфлоу записать outputs[P] в буферы приёмников. Если (G, P) в exposed_outputs воркфлоу — записать в слот внешнего выхода воркфлоу.
- Буферы воркфлоу создаются в начале run(workflow, …) и уничтожаются в конце; между вызовами run не сохраняются.

### 6.4 Жизненный цикл одного run воркфлоу

1. **Вход:** workflow, inputs, options (training, device, seed и т.д.; опции при необходимости пробрасываются в run каждого гиперграфа-узла).
2. **Валидация (опционально):** при первом run или по опции выполнить валидацию воркфлоу (§5).
3. **Проверка плана:** если закэшированный план устарел (execution_version изменился), вызвать планировщик, обновить кэш плана (топологический порядок узлов-гиперграфов).
4. **Инициализация буферов:** для каждого внешнего входа воркфлоу записать в буфер значение из inputs.
5. **Обход по плану:** для каждого шага плана (гиперграф или фаза цикла): разрешить входы гиперграфа из буферов и внешних входов воркфлоу; вызвать **run(hypergraph, inputs)** (внутри — полный жизненный цикл run гиперграфа, включая циклы и agent_loop); записать выходы в буферы и при необходимости во внешние выходы воркфлоу. При итеративной схеме (цикл между гиперграфами) циклическая фаза выполняется K раз с обновлением буферов между итерациями.
6. **Сбор выходов:** по списку exposed_outputs воркфлоу собрать из буферов значения в словарь outputs.
7. **Выход:** вернуть outputs.

Доменная логика задач (диффузия, LLM, агент) выполняется **внутри** гиперграфов-узлов; движок воркфлоу только передаёт данные между гиперграфами и вызывает их run.

## 7. Контракт run воркфлоу

### 7.1 Сигнатура и семантика

- **run(workflow, inputs, **options) → outputs**
- **Вход (inputs):** словарь — значения для внешних входов воркфлоу. Ключи — name (если задан при expose_input) или канонический идентификатор (graph_id, port_name). Значения — данные в формате, ожидаемом соответствующими гиперграфами (тензоры, словари, строки и т.д.). **Все объявленные внешние входы воркфлоу** (кроме опциональных по контракту) должны быть представлены в inputs; отсутствие обязательного ключа ведёт к ошибке или неопределённому поведению по соглашению реализации.
- **Выход (outputs):** словарь — значения с внешних выходов воркфлоу после выполнения всех гиперграфов в порядке топологии. Ключи — name или канонический идентификатор порта.

### 7.2 Опции run

| Опция | Назначение |
|-------|------------|
| **training** | bool; пробрасывается в run гиперграфов-узлов; режим train/eval для обучаемых блоков внутри. |
| **device** | Устройство (CPU/GPU); по соглашению делегируется всем гиперграфам-узлам (to(device)). |
| **num_loop_steps** | Число итераций циклической фазы **на уровне воркфлоу** (при цикле между гиперграфами); при необходимости пробрасывается в гиперграфы (например, num_inference_steps для диффузии). |
| **num_workflow_steps** | То же, что num_loop_steps для цикла воркфлоу (алиас по соглашению). |
| **max_steps** | Пробрасывается в гиперграфы (например, лимит шагов agent_loop). |
| **seed** | Начальное значение для стохастичности; обеспечивает воспроизводимость при пробросе в гиперграфы. |
| **callbacks** | Опциональные колбэки: на уровне воркфлоу — например, hook(graph_id, phase, **kwargs) с phase `"before"` / `"after"` при вызове run(hypergraph, …) для каждого узла-гиперграфа; пробрасываются в run гиперграфов по соглашению. |
| **dry_run** | Проверить план без фактического вызова run гиперграфов-узлов. |

Конкретный набор опций задаётся реализацией; контракт воркфлоу не меняется от опций — опции лишь параметризуют выполнение.

### 7.3 Вызов извне

Воркфлоу вызывается из **стадии**, **этапа** или теста: передаются данные на внешние входы воркфлоу, снимаются данные с внешних выходов. Внутренняя топология (список гиперграфов, порядок) для вызывающего может быть скрыта; воркфлоу выступает как единая комбинированная задача с контрактом run(inputs) → outputs.

### 7.4 Детерминированность и воспроизводимость

При одних и тех же **конфиге воркфлоу**, **чекпоинтах всех гиперграфов-узлов** и **входах** порядок исполнения задан топологией воркфлоу и планом; результат run детерминирован в пределах контрактов гиперграфов. **Конфиг воркфлоу + чекпоинты гиперграфов + входы** задают воспроизводимый эксперимент (философия проекта, [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §8.4).

## 8. Сериализация уровня воркфлоу

### 8.1 Конфиг структуры воркфлоу

- **Конфиг воркфлоу** задаёт: **graphs** (список узлов: для каждого — graph_id/node_id, конфиг гиперграфа — вложенный объект с nodes, edges, exposed_inputs, exposed_outputs по [03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §9.1 — или ссылка ref на отдельный файл конфига); **edges** воркфлоу (source_graph, source_port, target_graph, target_port); **exposed_inputs**, **exposed_outputs** воркфлоу (graph_id, port_name, name?); **schema_version**, workflow_id, опционально workflow_kind, **metadata**. Поле **schema_version** — версия схемы конфига для обратной совместимости при эволюции формата ([01_FOUNDATION.md](01_FOUNDATION.md)). Поле **metadata** — произвольные метаданные (описание, версия, теги) для каталогизации и отображения; семантика на уровне воркфлоу не задаётся.
- При загрузке по конфигу каждый гиперграф строится из своего конфига (03.from_config); затем собираются гиперрёбра воркфлоу и внешние входы/выходы воркфлоу.
- **Пример минимальной структуры конфига** (псевдо): `graphs: [{graph_id: "text2img", config: {...}}, {graph_id: "upscale", ref: "path/to/upscale.yaml"}]; edges: [{source_graph: "text2img", source_port: "image", target_graph: "upscale", target_port: "image"}]; exposed_inputs: [{graph_id: "text2img", port_name: "text", name: "prompt"}]; exposed_outputs: [{graph_id: "upscale", port_name: "image"}]`. Реальный формат (JSON/YAML, имена полей) задаётся реализацией.

### 8.2 Чекпоинт воркфлоу

- Воркфлоу **не имеет своего отдельного state_dict** — состояние распределено по гиперграфам-узлам. **Чекпоинт воркфлоу** — это совокупность чекпоинтов всех входящих в него гиперграфов (словарь graph_id → чекпоинт гиперграфа, или директория с поддиректориями по graph_id, в каждой — конфиг и чекпоинт гиперграфа по правилам 03 §9.2). По соглашению реализации воркфлоу предоставляет **state_dict()** (агрегат state_dict всех гиперграфов-узлов по graph_id) и **load_state_dict(state)** (загрузка в соответствующие гиперграфы) — для единообразия с уровнем гиперграфа и фундаментом ([01_FOUNDATION.md](01_FOUNDATION.md) §9).
- Сохранение: для каждого узла-гиперграфа сохранить его чекпоинт (по правилам 03). Загрузка: после построения воркфлоу из конфига загрузить чекпоинт каждого гиперграфа в соответствующий узел. Если несколько узлов воркфлоу ссылаются на один и тот же гиперграф по ref/block_id, применяется принцип одного экземпляра в памяти ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §12.6; на уровне воркфлоу — один экземпляр гиперграфа при общем ref).

### 8.3 Полное сохранение и загрузка

- **Сохранить:** записать конфиг воркфлоу (config.json или config.yaml) и чекпоинты всех гиперграфов (например, save_dir/graph_1/, save_dir/graph_2/, … или один файл с ключами graph_id). **Загрузить:** прочитать конфиг, построить воркфлоу (гиперграфы из конфигов), загрузить чекпоинты в гиперграфы. Конфиг воркфлоу + чекпоинты гиперграфов + входы задают воспроизводимое поведение. По соглашению реализации возможна **загрузка только чекпоинтов** в уже собранный воркфлоу (без перечтения конфига): например, load_checkpoint(checkpoint_dir) или load_from_checkpoint(..., checkpoint_dir=...) — веса подставляются в существующие гиперграфы-узлы по graph_id; структура воркфлоу не меняется.

## 9. Типичные сценарии и примеры

### 9.1 Текст → видео (цепочка из трёх гиперграфов)

| Элемент | Реализация |
|---------|------------|
| **Узлы воркфлоу** | 1) Гиперграф text-to-image (вход: text, выход: image). 2) Гиперграф upscale (вход: image, выход: image). 3) Гиперграф image-to-video (вход: image, выход: video). |
| **Гиперрёбра** | out(1, image) → in(2, image); out(2, image) → in(3, image). |
| **Внешние входы воркфлоу** | text (порт гиперграфа 1). |
| **Внешние выходы воркфлоу** | video (порт гиперграфа 3). |
| **Выполнение** | run(workflow, {"text": "..."}) → топологический порядок [1, 2, 3]; каждый гиперграф вызывается run(hypergraph, inputs); выход предыдущего подаётся на вход следующего. |

### 9.2 Ветвление: один выход — несколько гиперграфов

- Один гиперграф (например, text-to-image) имеет выход image. Два следующих гиперграфа: upscale и image-to-image (стилизация). Оба получают image на вход. Гиперрёбра воркфлоу: (text2img, image) → (upscale, image); (text2img, image) → (style, image). Внешние выходы воркфлоу могут быть два: upscaled image и styled image. Политика агрегации на уровне воркфлоу не требуется, если порты разные; оба гиперграфа выполняются после text2img (порядок между upscale и style может быть произвольным или задан явно).

### 9.3 Вырожденный воркфлоу: один гиперграф

- Воркфлоу может состоять из **одного гиперграфа**. Тогда у воркфлоу один узел; внутренних рёбер между гиперграфами нет. Внешние входы воркфлоу = внешние входы этого гиперграфа; внешние выходы воркфлоу = внешние выходы этого гиперграфа. run(workflow, inputs) по сути делегирует run(hypergraph, inputs) и возвращает те же outputs.
- **Зачем:** единый интерфейс «вход/выход воркфлоу» для одной задачи удобен на уровне **стадии**: стадия мира может быть реализована как гиперграф воркфлоу, где каждый узел — воркфлоу (в том числе вырожденный — одна задача). Тогда между стадиями передаётся state; внутри стадии воркфлоу вызывается с частью state и возвращает результат в state. Вырожденный воркфлоу даёт единообразие: и «одна задача», и «цепочка задач» представлены как воркфлоу.

### 9.4 Минимальный воспроизводимый пример

- Воркфлоу из двух гиперграфов: A (вход in_a, выход out_a) и B (вход in_b, выход out_b). Ребро: (A, out_a) → (B, in_b). Внешние входы воркфлоу: (A, in_a). Внешние выходы: (B, out_b). run(workflow, {in_a: x}) → выполнить A, затем B; outputs = {out_b: result}. Тест фиксирует конфиг воркфлоу, конфиги и чекпоинты A и B, вход x и проверяет совпадение выхода (или структуры) для воспроизводимости.

## 10. Воркфлоу и развёртывание

- **Гиперграф задачи** остаётся **единицей развёртывания** ([CANON.md](../CANON.md)): отдельные гиперграфы-узлы воркфлоу могут выполняться на разных эндпоинтах; гиперрёбра воркфлоу передают данные между эндпоинтами по согласованному протоколу. Воркфлоу как целое может выполняться **распределённо**: каждый run(hypergraph, …) — на том узле, где развёрнут соответствующий гиперграф; оркестратор воркфлоу передаёт inputs/outputs по сети.
- **API-гиперграфы:** если гиперграф задачи доступен по API (удалённый сервис с контрактом run(inputs) → outputs), он подключается как узел воркфлоу через тот же контракт; для воркфлоу он остаётся «гиперграфом» с вызовом run. Воркфлоу с такими узлами может выполняться гибридно: часть узлов локально, часть — через API.
- **Перенос на устройство:** по соглашению реализации воркфлоу предоставляет **to(device)** — делегирование переноса всем гиперграфам-узлам (как на уровне гиперграфа задачи — перенос блоков на устройство). Нужен для единообразной работы со стадией и при распределённом выполнении.
- **Trainable-узлы:** реализация может предоставлять **trainable_parameters()** (обход параметров только тех гиперграфов-узлов, которые помечены как trainable) и **set_trainable(graph_id, bool)** для пометки узла как обучаемого/необучаемого — по аналогии с уровнем гиперграфа и для совместимости со стадией/обучением (§11.8).

## 11. Граничные случаи и тонкости

### 11.1 Один узел (вырожденный воркфлоу)

- Воркфлоу из **одного гиперграфа** допустим и целесообразен (§9.3). План = один шаг; run(workflow, inputs) делегирует run(hypergraph, inputs). Валидация и планировщик применимы (план = один шаг).

### 11.2 Пустые внешние входы/выходы

- Теоретически воркфлоу может не иметь объявленных внешних входов (все входы гиперграфов заполняются с исходящих рёбер других гиперграфов) или не иметь внешних выходов (результаты только внутри). Для использования как чёрного ящика извне (стадия, этап) нужны хотя бы один внешний вход и один внешний выход; для вложенного или тестового вызова — не обязаны.

### 11.3 Опциональные входы гиперграфов-узлов

- Гиперграф задачи может объявлять **опциональные** внешние входы (по контракту уровня 03 — порт с значением по умолчанию или необязательный для run). В воркфлоу на такой порт не обязано вести гиперребро и не обязано подаваться значение в inputs при run(workflow, …): движок передаёт в run(hypergraph, inputs) только те ключи, для которых есть данные из буферов или из внешних входов воркфлоу; недостающие опциональные входы обрабатываются внутри гиперграфа (default или пропуск по его контракту).

### 11.4 Циклы между гиперграфами

- **Циклы между гиперграфами на уровне воркфлоу поддерживаются.** Воркфлоу — такой же гиперграф, как и гиперграф задачи, со всеми теми же возможностями: при наличии цикла (например, граф A → граф B → граф A) планировщик строит итеративную схему (начальная фаза → циклическая фаза K раз → конечная фаза); число K задаётся конфигом или опцией run (num_loop_steps, num_workflow_steps). Буферы между гиперграфами обновляются на каждой итерации. Циклы **внутри** гиперграфа (диффузия, agent_loop) по-прежнему обрабатываются движком гиперграфа при вызове run(hypergraph, …).

### 11.5 Проброс опций и контекста

- Опции run(workflow, inputs, options) (device, training, num_loop_steps, seed и т.д.) по соглашению реализации **пробрасываются** в run каждого гиперграфа-узла. Часть опций может относиться только к отдельным гиперграфам (например, num_loop_steps только для графа диффузии); конфиг воркфлоу или реализация могут задавать маппинг опций по graph_id.

### 11.6 Обработка ошибок

- Философия проекта не предписывает единую политику обработки ошибок. Сбой одного гиперграфа-узла (исключение, таймаут) может прерывать run воркфлоу или обрабатываться по политике вызывающего (ретраи, fallback, частичный результат). Уровень воркфлоу фиксирует лишь то, что при run выполняются шаги плана и передача данных по буферам; детали устойчивости — в рамках контракта и реализации ([03_TASK_HYPERGRAPH.md](03_TASK_HYPERGRAPH.md) §12.7).

### 11.7 Повторное использование одного гиперграфа (один экземпляр)

- Если один и тот же гиперграф (по ref или config) используется в воркфлоу **несколько раз** как логически разные узлы (например, один и тот же граф «upscale» вызывается дважды с разными входами от двух разных графов), реализация может либо создавать два экземпляра гиперграфа (два узла с одинаковой конфигурацией), либо разделять один экземпляр с разными входами по шагам плана. Канон рекомендует для **разных узлов воркфлоу** явно различать graph_id и при необходимости инстанцировать копии; принцип «один чекпоинт — один экземпляр в памяти» ([philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md)) применяется к блокам внутри гиперграфа и к случаю, когда один и тот же ref на гиперграф явно переиспользуется как один узел (один graph_id, один экземпляр).

### 11.8 Обучение на уровне воркфлоу

- По философии проекта ([../../philosophy/Philosophy-Part3.md](../../philosophy/Philosophy-Part3.md)) гиперграф, воркфлоу, этап, мир — с **выбором trainable-сущностей** из состава. Отдельные гиперграфы-узлы могут быть помечены как trainable; при run(workflow, inputs, training=True) опция **training** пробрасывается в run соответствующих гиперграфов. Градиенты и обновление весов обрабатываются внутри гиперграфов согласно их контракту; детали (оптимизатор, обратное распространение по воркфлоу при необходимости) задаются реализацией. Уровень воркфлоу фиксирует лишь единый контракт run и проброс опций; сериализация чекпоинта воркфлоу включает чекпоинты всех гиперграфов (§8.2).

---

**Итог.** Документ задаёт **полную техническую спецификацию уровня воркфлоу**: воркфлоу как гиперграф гиперграфов (узлы = гиперграфы задач, гиперрёбра = данные между задачами) **со всеми возможностями гиперграфа** — в том числе **поддержка циклов между гиперграфами** (итеративная схема, num_loop_steps), гиперрёбра арности > 2 и **политика агрегации** при нескольких входящих рёбрах на один порт. Контракт гиперграфа как узла (внешние входы/выходы, ключи inputs/outputs), построение (add_node, add_edge/add_hyperedge, remove_edge/remove_hyperedge, expose_input/expose_output, from_config, опционально infer_exposed_ports, workflow_kind), валидация (порты, типы, циклы при необходимости), применение того же гиперграфового движка (план: топология или итеративная схема, буферы с агрегацией, вызов run(hypergraph, …)), контракт run(workflow, inputs) → outputs, сериализация (конфиг и чекпоинт как совокупность чекпоинтов гиперграфов), типичные сценарии, вырожденный случай (один гиперграф), развёртывание и граничные случаи (в том числе обучение на уровне воркфлоу). Реализация и уровень стадии опираются на эту спецификацию.
